# v3.6.0 Deep Research Tailoring Engine — Opening Brief

**Opened:** 2026-04-13
**Status:** Milestone discussion brief

## Why this milestone now

MarkOS already has three strong foundations in place:

1. a literacy base with retrieval contracts and provenance
2. MIR and MSP structures that can hold rich strategy context
3. governance and verification flows that keep closeout honest

What is still missing is a truly advanced research layer that can decide which context matters most for a specific company, industry, audience, and execution scenario, then use that context to tailor output instead of producing near-generic drafts.

## Proposed milestone goal

Build the most advanced Deep Research function in MarkOS so Literacy, MIR, and MSP artifacts can be:

- tailored by industry, company, audience, and additional filters
- refreshed when context changes
- grounded in evidence rather than prompt-only intuition
- updated selectively instead of requiring whole-repo regeneration

## Recommended filter stack

### Core filters
- industry
- company profile
- audience / ICP / persona
- product or offer family
- business model
- company size and growth stage

### Strategic filters
- funnel stage
- campaign objective
- channel / discipline
- competitive pressure
- market maturity
- pricing tier

### Risk and context filters
- geography / region
- regulation / compliance environment
- brand constraints
- language / localization
- operator intent

## Recommended product behaviors

### 1. Research synthesis packs
A single run should create a ranked, evidence-backed context pack that answers:
- what matters for this company now
- what matters for this audience now
- what should change in MIR
- what should change in MSP
- which literacy artifacts are most relevant

### 2. Artifact delta generation
Instead of rewriting everything, the engine should output:
- sections to update
- rationale for each change
- evidence source links
- confidence and freshness indicators

### 3. Tailored literacy retrieval
The literacy layer should no longer be filtered only by discipline. It should also support:
- audience_tags
- industry_tags
- region_tags
- compliance_tags
- stage_tags
- intent_tags
- product_tags

### 4. Operator-visible reasoning
For every generated recommendation, show:
- active filter stack
- top retrieved evidence
- why this recommendation won over alternatives
- what changed from the prior artifact state

## Recommended external research stack

### Primary orchestration layer
- OpenAI Responses API with web search and deep-research-capable reasoning models for long-running, cited investigations when live internet evidence is needed.
- Use background or longer-running reasoning modes for deep market synthesis, not for every quick retrieval call.

### Internal knowledge access layer
- Add a read-only MarkOS remote MCP server exposing search and fetch over company knowledge, literacy evidence, and approved MIR/MSP state.
- Keep the first version data-only and read-only so it is safe for deep research clients and operator review.

### Web research and extraction layer
- Extend the existing Tavily integration for fast search, extraction, and competitor discovery.
- Use Firecrawl when page scraping, structured extraction, branding capture, or site-wide crawl depth is needed.
- Prefer official or vendor-hosted MCP endpoints when available; otherwise use direct HTTP APIs with the same evidence rules.

### Safety posture
- Official providers first
- Read-only tools by default
- domain allow-lists for sensitive research lanes
- explicit citations and freshness metadata on every external claim
- graceful fallback to vault-only retrieval when external providers are disabled or unavailable

## Relevant implementation sources

- modelcontextprotocol/servers — reference patterns for compliant MCP servers
- jlowin/fastmcp — lightweight MCP server framework referenced by current OpenAI MCP guidance
- firecrawl/firecrawl — crawl, scrape, and structured extraction reference implementation
- Existing MarkOS code already includes Tavily search hooks and OpenAI SDK support, so this milestone should extend those first rather than introducing unnecessary provider churn.

## Recommended initial phase sequence

### Phase 91 — Filter taxonomy and provider contract
Define the canonical filter schema, weighting rules, provider interface, fallback order, allowed-domain policy, and context-pack contract.

### Phase 92 — MarkOS company-knowledge MCP surface
Expose literacy, MIR, MSP, and approved evidence through a read-only MCP-compatible search/fetch layer for deep research clients.

### Phase 93 — Multi-source deep research orchestration
Merge local retrieval, Tavily search/extraction, OpenAI web search or deep research, and Firecrawl structured crawl results into one ranked research pipeline.

### Phase 94 — MIR and MSP delta patch engine
Generate precise artifact updates and section-level refreshes rather than broad rewrites, with operator-visible evidence diffs.

### Phase 95 — Evaluation and governance
Add provider comparison, relevance scoring, grounding checks, artifact-diff review, and operator acceptance surfaces.

## Success bar

The milestone is successful when MarkOS can take the same base literacy repository and produce meaningfully different, evidence-backed outputs for:
- different industries
- different company types
- different audience segments
- different channels and campaign goals

without losing provenance, auditability, or output consistency.

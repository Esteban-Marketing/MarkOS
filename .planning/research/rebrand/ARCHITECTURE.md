### Data Flow Through Renamed Components

```
npx markos install
  → bin/install.cjs reads VERSION from .agent/markos/VERSION
  → Copies .agent/markos/ → target/.agent/markos/
  → Creates .markos-install-manifest.json
  → Creates .markos-local/ scaffold
  → Writes .markos-project.json

npx markos update
  → Reads .markos-install-manifest.json (falls back to .markos-install-manifest.json)
  → Checks .markos-local/ for overrides (falls back to .markos-local/)
  → Updates .agent/markos/ (detects old .agent/markos/ and migrates)

node onboarding/backend/server.cjs
  → Reads .markos-project.json (falls back to .markos-project.json)
  → Supabase + Upstash Vector collections: markos-{slug}-{section}
  → Writes to .markos-local/MIR/
  → Telemetry: $lib = "markos-backend-telemetry"
```

## Patterns to Follow

### Pattern 1: Dual-Path Detection (Backward Compatibility)
**What:** Every file operation that reads/writes MARKOS-prefixed paths should check both old and new names  
**When:** During the transition period (at least one major version)  
**Example:**
```javascript
// In install.cjs / update.cjs / handlers.cjs:
const NEW_PATH = '.markos-local';
const LEGACY_PATH = '.markos-local';
const localPath = fs.existsSync(NEW_PATH) ? NEW_PATH : 
                  fs.existsSync(LEGACY_PATH) ? LEGACY_PATH : NEW_PATH;
```

### Pattern 2: Auto-Migrate on First Run
**What:** When update.cjs detects old paths, rename them before proceeding  
**When:** First run after upgrade to markos-named version  
**Example:**
```javascript
// Detect and migrate old directory names
if (fs.existsSync('.agent/markos') && !fs.existsSync('.agent/markos')) {
  fs.renameSync('.agent/markos', '.agent/markos');
  console.log('âœ“ Migrated .agent/markos → .agent/markos');
}
```

### Pattern 3: Bridge Deprecation Package
**What:** Publish final version of `markos` that only prints a migration message  
**When:** Same day as `markos` first publish  

## Anti-Patterns to Avoid

### Anti-Pattern 1: Hard Cut Without Migration
**What:** Changing all paths without backward compatibility checks  
**Why bad:** Existing installs break silently — update.cjs can't find manifest, install.cjs can't detect existing install  
**Instead:** Dual-path detection for at least one version

### Anti-Pattern 2: Renaming Supabase + Upstash Vector Collections In-Place
**What:** Trying to rename existing Supabase + Upstash Vector collections  
**Why bad:** Supabase + Upstash Vector has no native "rename collection" operation; requires delete+recreate  
**Instead:** New installs use `markos-` prefix; migration creates new collections and copies data if old ones exist

### Anti-Pattern 3: Mass Find-Replace in Agent Prompts
**What:** Blindly replacing every "MARKOS" string in 317 agent files  
**Why bad:** Agent prompt content uses "MARKOS" as a token prefix (MARKOS-AGT-STR-01); changing these affects the entire token ID system  
**Instead:** Rename file names and directory, but leave internal token IDs for a separate token-system migration phase

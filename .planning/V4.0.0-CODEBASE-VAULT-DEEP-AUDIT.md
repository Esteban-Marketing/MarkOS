# v4.0.0 Codebase vs Vault Deep Audit

**Date:** 2026-04-23  
**Purpose:** Perform a full planning-grade scan of the current repository and compare the implemented routes, folders, files, and business logic against the active Obsidian vault doctrine and incoming documents from this week.  
**Use:** Canonical artifact for later GSD discuss/research/plan work. This is not an execution report; it is the prep document that should shape future phase planning and tech-debt routing.

## Executive Result

The codebase already has strong foundation work in tenancy, auth, MCP, webhooks, CRM, billing ledger/reconciliation, governance, docs, contracts, and tests. The vault, however, now describes a larger Marketing Operating System than the current implementation actually runs.

The good news: most of the missing work is already routed to the right future phases.  
The important caveat: those phases now need codebase-specific deltas, because the repo is broader and more mature than the earlier planning artifacts assumed.

**Decision:** no new top-level phase is required from this audit alone.  
**Required outcome:** use this audit to strengthen Phases 204-220 so they explicitly absorb existing code surfaces, static residues, migration paths, and verification burdens.

## Concrete Tech Debt Register

### TD-01 Static pricing residue

**Evidence**

- `lib/markos/billing/entitlements.ts` uses hard-coded allowances
- `api/billing/tenant-summary.js` emits hard-coded pricing usage rows
- `app/(markos)/settings/billing/page-shell.tsx` renders fixed included usage and invoice values

**Impact**

Pricing Engine doctrine cannot become true until these surfaces either consume approved engine records or intentionally render `{{MARKOS_PRICING_ENGINE_PENDING}}`.

**Route**

Strengthen Phase 205.

### TD-02 Fixture-backed operations UI

**Evidence**

- `app/(markos)/operations/tasks/page.tsx` explicitly says `TaskStoreProvider seeded with fixtures`

**Impact**

Current UI is useful for interaction design, but it is not yet a Morning Brief, Task Board, or Approval Inbox backed by canonical state.

**Route**

Strengthen Phase 208.

### TD-03 Fragmented AgentRun substrate

**Evidence**

- `onboarding/backend/agents/run-engine.cjs` provides in-memory run/event/effect helpers
- `supabase/migrations/53_agent_run_lifecycle.sql` provides durable schema
- CRM copilot/playbook flows use run objects independently
- MCP has its own pipeline/event model

**Impact**

The vault expects one cross-surface orchestration substrate. The repo currently has adjacent foundations, not one unified system.

**Route**

Strengthen Phase 207.

### TD-04 Evidence is present but not normalized

**Evidence**

- `audit_claim`/`audit_claim_strict` tools exist
- governance evidence-pack surfaces exist
- CRM and billing use evidence language
- no `EvidenceMap`, no TTL/freshness object, no source-quality object in implementation search

**Impact**

The code can talk about evidence without yet enforcing the full vault doctrine of grounded claims and freshness.

**Route**

Strengthen Phase 209.

### TD-05 Connector doctrine has substrate but no operating loop

**Evidence**

- webhooks and MCP are mature
- onboarding has extractors/research helpers
- no `ConnectorInstall` object or wow/recovery operating layer detected

**Impact**

Connector strategy in the vault is about fast value unlock and recovery workflows, not just technical plumbing.

**Route**

Strengthen Phase 210.

### TD-06 Content/revenue loop is split across subsystems

**Evidence**

- marketing MCP tools exist
- CRM execution/outbound/reporting exists
- no single canonical implementation of full v2 loop from onboarding to learning

**Impact**

The repo can do parts of the loop, but not yet prove the loop as one governed product behavior.

**Route**

Strengthen Phase 211.

### TD-07 Learning and literacy promotion loop is absent

**Evidence**

- retrieval and literacy tooling exist
- no `ArtifactPerformanceLog`, `TenantOverlay`, or `LiteracyUpdateCandidate` implementation detected

**Impact**

The vault now expects learning to update local and central behavior in a governed way.

**Route**

Strengthen Phase 212.

### TD-08 Tenant 0 is still doctrine-only

**Evidence**

- no Tenant 0 runtime object, route family, or proof policy implementation detected in code scan

**Impact**

MarkOS cannot credibly claim the doctrine until it runs itself on itself.

**Route**

Strengthen Phase 213.

### TD-09 SaaS Suite is not runtime truth yet

**Evidence**

- SaaS pack and onboarding taxonomy exist
- no `SaaSSuiteActivation`, no `/v1/saas/*`, no `markos-saas`, no SaaS-specific tables or UI families detected

**Impact**

The vault canon is ahead of the implementation. This is okay, but phases must treat it as future foundation work.

**Route**

Phases 214-217.

### TD-10 SaaS growth strategy is doctrine-only

**Evidence**

- no growth-mode profile objects, PLG/PQL runtime objects, ABM objects, referral/community/partnership growth workflows, or growth agent surfaces detected

**Impact**

Growth-strategy canon should remain post-foundation planning truth only.

**Route**

Phases 218-220.

### TD-11 Codebase map had drifted behind the repo

**Evidence**

- prior `.planning/codebase/*` documents mostly described the smaller onboarding-era repo and did not reflect `app/`, `lib/`, `contracts/`, `sdk/`, `supabase/`, `tools/`, or the full API/UI estate

**Impact**

Planning based on stale topology leads to weak phase prep.

**Route**

Resolved in this pass by refreshing `.planning/codebase/*`.

### TD-12 Integrated acceptance still lags doctrine

**Evidence**

- deep domain tests exist, but there are no implemented acceptance suites yet for Pricing Engine, SaaS Suite, growth-mode routing, or full v2 loop closure

**Impact**

Without acceptance suites, future phases may ship local correctness without proving product-level doctrine.

**Route**

Future test planning inside each phase; no new top-level phase required yet.

## Phase Strengthening Instructions

### Phase 204

- absorb the real current CLI surface from `bin/cli-runtime.cjs`
- explicitly reconcile package binary mapping (`markos -> bin/install.cjs`) with future GA operator commands
- make `status` and `doctor` aware of v2 doctrine inputs and new audit artifacts

### Phase 205

- treat billing UI/API/lib cleanup as part of Pricing Engine readiness, not side cleanup
- sweep hard-coded allowances/prices from settings/admin/API language
- define placeholder policy for all unresolved public pricing statements

### Phase 206

- govern AI, pricing, connector, research, learning, and Tenant 0 risks explicitly
- reuse existing governance/audit foundations instead of inventing parallel compliance surfaces

### Phase 207

- unify onboarding run-engine, durable run tables, CRM playbook runs, and MCP pipeline concepts
- define the canonical shared substrate instead of letting each domain keep its own run language

### Phase 208

- treat current `operations/tasks` UI as the migration baseline
- replace fixture state with live run/task/approval/evidence-backed state
- absorb Morning Brief, Approval Inbox, Task Board, and Weekly Narrative into the real workspace

### Phase 209

- convert existing evidence-adjacent tools and pages into one explicit evidence model
- make freshness, source quality, TTL, and known-gap handling first-class

### Phase 210

- build ConnectorInstall and wow/recovery flows on top of already strong webhook/MCP foundations
- avoid treating connectors as passive integrations

### Phase 211

- bridge marketing MCP tools with CRM execution/outbound/reporting and approval workflows
- prove one end-to-end operating loop, not a loose collection of subsystems

### Phase 212

- connect literacy retrieval to actual performance feedback and admin-reviewed promotion

### Phase 213

- define the real internal dogfood tenant and public-proof posture

### Phases 214-217

- preserve existing billing/CRM/MCP/tenancy patterns
- do not fork separate SaaS-only infrastructure when shared MarkOS domains already exist

### Phases 218-220

- keep these as governed future work
- require contracts, approvals, costs, APIs, MCP tools, UI, and tests before any growth-agent claims become implementation truth

## Planning Decision

**No new top-level phase was added in this audit.**  
The current phase family `204 -> 220` is still the right shape.

What changed is the expected depth:

- each phase now has explicit codebase migration targets
- tech debt is routed as part of the planned phase work, not as unnamed cleanup
- future discuss/research/planning should use this artifact alongside the vault canon and requirements matrix

# Phase 214 — UI Design Contract

> Visual and interaction contract for the **SaaS Suite Activation operator surface**. Phase 214 is a HYBRID phase: five backend plans (214-01..05) ship migrations + library + API handlers + tests with **zero UI surface**, and a single UI plan (214-06) ships **one operator settings page** plus **two extracted reusable components** that subsequent SaaS Suite phases (P215 billing, P216 health/churn, P217 revenue agents) consume verbatim.
>
> P214 is the **first SaaS-Suite phase** to render any pixel. The contract is small in surface count (one page, two components) but heavy in inheritance: every approval gate binds to 206 mutation-class doctrine, every subscription lifecycle row binds to 207 RunApiEnvelope, every pricing field binds to 205 PricingRecommendation or the verbatim sentinel, and the operator settings shell binds to 213.3 billing-page pattern. The 213.4 wave-4 D-08..D-15 carry-forward decisions remain in force.
>
> Phase 214 introduces a **surface gating contract**: `business_type != 'saas'` tenants render a single `.c-notice c-notice--info` page and never reach the activation panel. This is the canonical reference for every downstream SaaS Suite surface.
>
> Phase 214 stores **growth_extension as metadata only**: a `.c-chip` badge labels the extension hint, but no growth surface is runnable until P218 ships. The `<PlaceholderBanner variant="future_phase_217">` is the canonical render for the growth-mode wizard placeholder.
>
> Authority chain: DESIGN.md v1.1.0 → 213.4-VALIDATION.md carry-forward (D-08..D-15) → 205-UI-SPEC (pricing sentinel origin) → 206-UI-SPEC (mutation-class origin) → 207-UI-SPEC (orchestration origin) → 208-UI-SPEC (parent shell + Approval Inbox) → 213-UI-SPEC (T0 gate consumer) → this document. Generated by gsd-ui-researcher. Status: draft (checker upgrades to approved once 6-pillar audit passes).

**Hybrid scope rationale.** The five backend plans ship the durable substrate (migrations + libs + handlers + tests) without rendering any pixel. They earn no UI-SPEC entry beyond their no-UI declaration. The single UI plan (214-06) ships the operator-facing settings surface that exposes the substrate, with the two reusable components extracted ahead of consumption (D-15 reuse threshold satisfied: `<SaaSActivationPanel />` is consumed by P217 SaaS Suite revenue UI nav and P218 growth wizard; `<SaaSSubscriptionsTable />` is consumed by P215 billing reconciliation viewer and P216 churn cohort table). Extracting at P214 prevents a P215 mid-phase refactor.

**Mobile priority.** Settings → SaaS is **`secondary`** per 208-01 mobile-priority literals (`critical | secondary | desktop_only`). It is usable on mobile but does not block the daily operator flow. `desktop_only` is forbidden as a mobile_priority value (208-01 architecture-lock string), so the surface meets WCAG 2.1 AA touch targets via the global `(pointer: coarse) { .c-button { min-height: 44px } }` rule already shipping per 213.2 carry-forward.

All acceptance criteria below apply to plan 214-06 deliverables only. Plans 214-01..05 are backend-only and do not produce UI artifacts.

## Spacing Scale

Authoring rule: every `padding`, `margin`, `gap`, `inset` in scope MUST cite a `--space-*` token. No arbitrary px. Off-grid values are auto-FAIL.

| Token | Value | DESIGN.md citation | Usage in this phase |
|-------|-------|--------------------|---------------------|
| `--space-none` | 0 | `spacing.none` | Reset margins on `<h1>`, `<h2>`, `<p>`, `<table>`, `<ul>` |
| `--space-xxs` | 2px | `spacing.xxs` | Badge inner padding, status-dot offset, activation-state-glyph adjacency, plan-catalog price-currency adjacency |
| `--space-xs` | 8px | `spacing.xs` | Activation panel section gap, plan-row chip group gap, subscription-row metadata gap, lifecycle-event timeline marker gap |
| `--space-sm` | 16px | `spacing.sm` | Card vertical rhythm, notice padding-block, table `th`/`td` padding-block, mobile horizontal page padding (max-width ≤ 640px), wizard-step header gap |
| `--space-md` | 24px | `spacing.md` | Card padding (via `.c-card`), gap between settings-page sections, modal padding, page horizontal padding at sm–md viewport, wizard-step body padding |
| `--space-lg` | 32px | `spacing.lg` | Page horizontal padding ≥ md, inter-section gap within settings page, mutation-queue group gap |
| `--space-xl` | 48px | `spacing.xl` | Page vertical padding ≥ lg, settings-page top padding |
| `--space-xxl` | 96px | `spacing.xxl` | Reserved — not used in this phase |

**Allowed exceptions (DESIGN.md documented):**
1. `1px` for hairline borders (`var(--color-border)`) — preferred over shadows on dark surfaces (DESIGN.md "Borders over shadows").
2. `2px` for focus ring width and offset (`var(--focus-ring-width)`, `var(--focus-ring-offset)`); never suppressed.
3. `4px` for `.c-notice` `border-inline-start` accent — composed via `.c-notice` primitive; modules never declare this directly.
4. `max-width: 1280px` for cockpit container (`--w-container`). Settings page uses `--w-container` with internal `--w-prose` (720px) constraint on text columns inside the activation hero card.
5. `max-width: 560px` for confirmation modals (`--w-modal`).
6. **Mobile touch target `44px`** via `--h-control-touch` on `.c-button` for `(pointer: coarse)` viewports — already declared globally in `styles/components.css` per 213.2 cross-cutting fix; this phase consumes it without re-declaration.

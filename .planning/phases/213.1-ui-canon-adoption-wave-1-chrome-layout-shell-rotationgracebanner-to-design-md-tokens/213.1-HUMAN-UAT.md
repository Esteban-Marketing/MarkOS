### 2. Windows High Contrast / `forced-colors` mode verification

expected: On a Windows machine with High Contrast enabled (Settings → Accessibility → Contrast themes → Aquatic / Desert / Dusk / Night sky):
- Load any `(markos)` chrome surface (e.g. `/markos`, `/markos/operations`, denied state)
- Verify focus rings remain visible using `Highlight` system color (not collapsed)
- Verify borders use `CanvasText` (not `--color-border` which gets overridden in forced-colors mode per `app/tokens.css`)
- Verify text contrast does not collapse on `surface-raised` cards
- Verify the bracketed-glyph prefix (`[warn]`, `[err]`) remains visible (DESIGN.md "Color blindness" requires text/icon as a state signal, never color alone — confirm the bracketed text passes through forced-colors mode)
- Banner severity color may be overridden to system colors — confirm `[warn]`/`[err]` text prefix still communicates state

result: passed (2026-04-28; resolved via code fix — `app/tokens.css` `@media (forced-colors: active)` block remaps `--color-border` + `--color-border-strong` → `CanvasText`, `--focus-ring-color` → `Highlight`, and added `--color-primary-text` → `LinkText` so active nav / protocol chip / tertiary button / mint chip read as semantic links under High Contrast. Bracketed-glyph state prefixes `[warn]`/`[err]` survive forced-colors automatically as text content — verified by `test/ui-a11y/213-1-light-and-forced-colors.test.js` Gate 2 suite (5 tests / 5 pass). `.c-input` outline-suppression paired with `box-shadow` ring replacement so focus indicator survives forced-colors.)

### 3. `[data-theme="light"]` light-mode flip verification

expected: In a chrome surface (e.g. Storybook preview or running app):
- Set `<html data-theme="light">` via DevTools
- Verify chrome flips to Lino Light `#F4F4F0` surface
- Verify Protocol Mint accent unchanged across modes (per DESIGN.md "Light-mode complements" — primary, accent, and state colors are unchanged)
- Verify text contrast remains AA / AAA (no contrast collapse on light raised surfaces)
- Verify focus rings still 2px solid `#00D9A3` with 2px offset
- Verify banner alpha-tint backgrounds (`rgb(255 184 0 / 0.12)`, `rgb(248 81 73 / 0.12)`) remain readable on Lino Light surface
- Verify NO mint as text on light (DESIGN.md forbidden — mint-on-light contrast = 1.6:1, fails)
- If the toolbar wiring for theme switching is missing in Storybook, file as a follow-up phase (213.1.1 or 213.4)

result: passed (2026-04-28; resolved via code fix — introduced `--color-primary-text` semantic token in `app/tokens.css`. Default (dark) aliases `var(--color-primary)` (10.5:1 on Kernel Black, AAA). Light theme override sets `#00805F` (4.95:1 on White, 4.49:1 on Lino Light — both AA normal text). `[data-theme="light"]` block also overrides every theme-sensitive surface token (surface, surface-raised, surface-overlay, on-surface, on-surface-muted, on-surface-subtle, border, border-strong, border-subtle). `styles/components.css` patched to consume `var(--color-primary-text)` (not raw `var(--color-primary)`) for every TEXT-color rendering of mint: `.c-button--tertiary`, `.c-nav-link[aria-current="page"]`, `.c-chip--mint`, `.c-chip-protocol`. Chrome modules (`layout-shell.module.css`, `RotationGraceBanner.module.css`) verified to NOT redeclare mint as text color — they inherit from globals + compose `.c-*` primitives. Banner alpha-tint backgrounds remain readable on Lino Light because the tint is computed against the active surface — confirmed via `test/ui-a11y/213-1-light-and-forced-colors.test.js` Gate 3 suite (7 tests / 7 pass). Storybook theme-toolbar wiring deferred — not blocking; will land in a future phase if needed.)

## Summary

This is a tactical CSS-canon-adoption redesign. UI-SPEC.md (28k tokens) locks every visual decision. CONTEXT.md locks every implementation-strategy decision. The remaining unknowns are purely *execution-path* mechanics:

1. **How does class-rename work in this repo?** Via template-literal `${styles.x} ${styles.y}` composition (no `clsx` package — verified absent from `package.json`). Every rename in `.module.css` is a literal find-and-replace in the consuming `.tsx`.

2. **What Phase 203 tests does the redesign actually break?** Two files exist: `test/webhooks/layout-shell-banner.test.js` (wiring contract — import/mount/fetch shape — does NOT break) and `test/webhooks/ui-s4-a11y.test.js` (CSS-token + copy assertions — **DOES** break on 5 hex assertions + 1 focus-outline assertion + needs `[warn]`/`[err]` prefix on `<strong>` content). CONTEXT.md and the objective both pointed at the wrong file by name. **The plan must update `ui-s4-a11y.test.js`, not `layout-shell-banner.test.js`.**

3. **Is Storybook 8 + Chromatic wired?** Yes — `.storybook/main.ts` exists, `@storybook/react-vite ^8.6.18`, addon-a11y is loaded, Chromatic addon is loaded, 18 existing `*.stories.tsx` files. `.storybook/preview.tsx` registers a custom `ThemeProvider` decorator with `defaultBrandPack` from `lib/markos/theme/brand-pack` — but **this is the legacy theme system, not DESIGN.md tokens**. The decorator inlines `--color-bg-canvas`, `--color-action-primary`, etc. — NONE of those names exist in `app/tokens.css` (which uses `--color-surface`, `--color-primary`, etc.). New stories will render in the legacy decorator's color space unless we either (a) bypass the decorator or (b) align `preview.tsx` with `app/tokens.css`. **This is a Plan-03 risk to surface to the planner.** Storybook does not currently import `app/globals.css`. To get DESIGN.md tokens into stories, `preview.tsx` must `import "../app/globals.css"` (which transitively imports `tokens.css` + `components.css`).

4. **Is there an axe-core test harness?** Sort of. `test/ui-a11y/accessibility.test.js` is a single grep-shape test that asserts story files exist and that `.storybook/preview.tsx` enables the a11y addon. It does NOT actually run axe-core against rendered components; it asserts the *configuration* exists. The `axe-core` package is in `devDependencies`, but the only place axe runs is via the Storybook a11y addon at story-render time (which happens during Chromatic builds and `npm run test:ui-all`). For Plan-03 axe tests, the canonical pattern is the existing grep-shape test (config assertion) — NOT a runtime axe assertion. If runtime axe is needed, that requires `axe-playwright` (already in deps) + a Playwright runner that does not currently exist for `test/ui-a11y/`.

5. **Tailwind v4 + CSS Modules interaction.** `app/globals.css` line 12 uses `@config "../tailwind.config.ts"` and explicitly skips `@import "tailwindcss/base"` (only loads `theme.css` + `utilities.css`). Tailwind preflight is intentionally OMITTED. CSS Modules consume `var(--*)` from `:root` (defined in `app/tokens.css`) which is imported via `app/globals.css`. **No preflight conflict.** Next.js 15's CSS pipeline handles `*.module.css` natively (PostCSS via `postcss.config.mjs` with `@tailwindcss/postcss` plugin only — no separate CSS-Modules plugin needed).

6. **`(pointer: coarse)` ≥44px placement.** Best location: `styles/components.css` immediately after the `.c-nav-link` block (line 207) and before `.c-topbar`. This places the rule where future `.c-nav-link` consumers will inherit it. Token to use: `padding-block: 11px` (which gives 11+11+16×1.6=44.6px — passes 44px). Avoid putting it in `app/(markos)/layout-shell.module.css` (would only fix the chrome nav, not future surfaces).

7. **Renaming `.pulseDot` → `.warningDot`.** UI-SPEC.md prescribes this rename. The class is consumed in exactly one place: `RotationGraceBanner.tsx:97` (`<span className={styles.pulseDot} aria-hidden="true" />`). One-line update. Phase 203 wiring test does NOT reference `pulseDot` literally (verified: `grep pulseDot test/` returns 0).

**Primary recommendation:** Plan as 3 plans. Plan-01 (layout-shell module + tsx, no Phase 203 test impact). Plan-02 (RotationGraceBanner module + tsx + bracketed-glyph prefix + **`test/webhooks/ui-s4-a11y.test.js` rewrite — NOT layout-shell-banner.test.js**). Plan-03 (Storybook stories colocated next to components + a11y test pair + `(pointer: coarse)` rule in `styles/components.css` + Storybook `preview.tsx` import of `app/globals.css`). Atomic commit per plan; verify `npm run build` + `npm test` between plans.

## Architecture Patterns

### CSS Modules + token vars (NEW — adopted by this phase)

**Pattern:** Every `*.module.css` class consumes only `var(--*)` tokens from `app/tokens.css` + `.c-*` primitives from `styles/components.css`. No inline hex (except whitelisted alpha-tint formulas), no inline px (except 1px borders, 2px focus offsets, 4px banner border-inline-start).

**Example (from UI-SPEC.md Component primitive reuse table — verified extracted):**

```css
/* layout-shell.module.css after redesign */
.page {
  min-height: 100dvh;
  background: var(--color-surface);
  padding: var(--space-md);
}
@media (max-width: 1023.98px) {
  .page { padding: var(--space-sm); }
}

.shell {
  max-width: var(--w-container);
  margin: 0 auto;
  display: grid;
  grid-template-columns: var(--w-sidebar) minmax(0, 1fr);
  gap: var(--space-md);
}
```

`[VERIFIED: app/tokens.css + UI-SPEC.md component primitive reuse table]`

### JSX class composition (existing convention — preserve)

**Pattern:** Template-literal composition. `clsx` is not a dependency.

```tsx
// Composing local module class with global .c-* primitive
<div className={`${styles.tenantPill} c-chip c-chip--mint`}>...</div>

// Multiple local classes
<aside className={`${styles.sidebar} c-sidebar`}>...</aside>
```

`[VERIFIED: 17 occurrences in app/(markos)/ via Grep — including operations/page.tsx lines 43/47/75, evidence-panel.tsx lines 58/130/179/192, etc.]`

### `globals.css` cascade (existing — inherit, do NOT redeclare)

**Pattern:** Modules INHERIT typography (`<h1>`, `<h2>`, `<p>`), focus rings (`:focus-visible`), reduced motion, scrollbars, selection — DO NOT redeclare them locally.

**Example:**
- The legacy `.brandTitle { font-family: 'Sora'; font-size: 2rem; ... }` is **deleted** entirely. Just use `<h1>MarkOS</h1>` and let `globals.css` lines 70–80 supply `var(--font-mono)` + `var(--fs-h1)` + `var(--fw-bold)` + `var(--lh-h1)` + `var(--ls-h1)` + `color: var(--color-on-surface)`.
- The legacy `.graceLink:focus-visible { outline: 2px solid #0d9488; ... }` is **deleted**. The global `:focus-visible` rule (`globals.css` lines 121–125) supplies `outline: var(--focus-ring-width) solid var(--focus-ring-color); outline-offset: var(--focus-ring-offset);` automatically.

## Common Pitfalls

### Pitfall 1: Renaming a class without updating every consumer
**What goes wrong:** `styles.heroCard` reference in `layout.stories.tsx` becomes `undefined` after `.heroCard` is deleted from `.module.css`. Build doesn't fail (CSS Modules don't have a TS error path for missing keys — they return `undefined`), but rendered output is unstyled.
**Why it happens:** `*.module.css` runtime keys are dynamic; TypeScript doesn't type-check them.
**How to avoid:** Run a verification grep BEFORE deleting any class:
```bash
grep -rn "styles\.<className>" app/ --include="*.tsx"
```
If matches exist outside the file being edited, plan the rename to update those call sites in the same commit.
**Warning signs:** Stories render with browser-default styling. Visual snapshot diff is dramatic.

### Pitfall 2: Forgetting to convert `LayoutShell` to client component for `usePathname()`
**What goes wrong:** UI-SPEC AC#8 requires `aria-current="page"` wired via pathname. `usePathname()` is a client-only hook. `MarkOSLayoutShell` is currently a server component (no `'use client'` directive). Adding `usePathname()` directly errors at build time.
**Why it happens:** Next.js 15 enforces server/client boundaries strictly.
**How to avoid:** Two options —
1. Add `'use client'` directive at top of `layout-shell.tsx`. Consequence: shell becomes client-rendered, increases bundle size of layout root, breaks server-only data fetching if any.
2. Extract the nav into a small client subcomponent (`NavList.tsx`) that uses `usePathname()`, leave `MarkOSLayoutShell` as a server component. Recommended.
**Warning signs:** Build error: *"You're importing a component that needs `usePathname`. It only works in a Client Component."*

### Pitfall 3: Storybook stories rendering in legacy theme decorator
**What goes wrong:** New stories for redesigned banner render with white background + lavender primary because `.storybook/preview.tsx` ThemeProvider decorator overrides `:root` CSS custom properties with legacy `tokens["color.bg.canvas"]` etc.
**Why it happens:** The decorator was authored for the legacy theme system; it ships with the workspace and applies to every story by default.
**How to avoid:** Plan-03 must (a) `import "../app/globals.css"` in `preview.tsx` AND (b) confirm the ThemeProvider decorator's inline style overrides don't clobber the imported tokens. ### Nyquist 4 Concerns (Inputs / Behaviors / States / Boundaries)

#### Inputs (what comes IN to the chrome)

1. **Authorized**: tenantId present → render full shell. Sub-states by pathname for active nav.

3. **Banner — empty**: no rotations → `RotationGraceBanner` returns `null`.

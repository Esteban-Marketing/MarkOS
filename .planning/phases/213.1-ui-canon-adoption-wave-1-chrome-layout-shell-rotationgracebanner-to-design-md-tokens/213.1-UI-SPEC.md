# Phase 213.1 — UI Design Contract

> Visual and interaction contract for redesigning the global `(markos)` chrome (sidebar shell, hero/content cards, access-denied state, and rotation-grace banner) from the legacy light-mode "Sora + teal + 28px radii + gradients + drop shadows" implementation into the canonical MarkOS visual contract defined in `/DESIGN.md` (dark Kernel Black surface, Protocol Mint signal, JetBrains Mono + Inter, strict 8px grid, borders preferred over shadows, flat). Generated by gsd-ui-researcher; verified by gsd-ui-checker.

**This is a redesign, not a port.** Every value in the in-scope module.css files is replaced with a token citation from `/DESIGN.md`. Inline hex, inline px (excepting documented 1px borders / 2px focus offsets), and gradients are removed wholesale.

## Spacing Scale

Authoring rule: every `padding`, `margin`, `gap`, `inset` in scope MUST cite a `--space-*` token. **No arbitrary px.** Off-grid values are auto-FAIL.

| Token | Value | DESIGN.md citation | Usage in this phase |
|-------|-------|--------------------|---------------------|
| `--space-none` | 0 | `spacing.none` | Reset margins on `<h1>`, `<p>` |
| `--space-xxs` | 2px | `spacing.xxs` | Banner border-left thickness setpoint, badge inner padding |
| `--space-xs` | 8px | `spacing.xs` | Icon gaps, inline tag gaps, sidebar inter-link spacing, status-dot adjacency |
| `--space-sm` | 16px | `spacing.sm` | Banner inline padding (replaces legacy `12px 20px`), button padding alignment, mobile horizontal padding |
| `--space-md` | 24px | `spacing.md` | Card internal padding, grid gutters, sidebar/content gap, paragraph spacing |
| `--space-lg` | 32px | `spacing.lg` | Sidebar internal padding, hero-card padding, inter-section grouping |
| `--space-xl` | 48px | `spacing.xl` | Page horizontal padding ≥ md, major section breaks, denied-card padding |
| `--space-xxl` | 96px | `spacing.xxl` | Reserved for landing — not used in chrome |

**Exceptions (allowed by DESIGN.md):**
1. `1px` for hairline borders (`var(--color-border)`) — explicitly preferred over shadows on dark surfaces.
2. `2px` for focus ring width and offset (`var(--focus-ring-width)`, `var(--focus-ring-offset)`).
3. Banner border-left accent at `4px` is captured via `var(--space-xxs)` × 2 — but `4px` is on-grid (a multiple of 2 within the canonical 0/2/8/16/24/32/48/96 scale; `2px` is `xxs`, `4px` is allowed as a `xxs × 2` accent since it's still on the spec's hairline ladder). Banner spec uses **`border-inline-start: 4px solid …`** — see Surface 4 below.

Banned exceptions (auto-FAIL): `10px`, `12px`, `14px`, `18px`, `20px`, `22px`, `28px`, `36px`, `40px` (as padding/margin), `45px`. These all appear in the existing chrome and MUST be eliminated.

**Accent reserved-for list (10% slice):**
1. `RotationGraceBanner` T-7 + T-1 + multi stages — warning border-left, warning text, optional warning pulseDot.
2. NOT used elsewhere in chrome. Sidebar, hero, denied state, nav, tenant pill MUST NOT use accent amber.

**Single mint signal (< 5% rule):**
1. **Active nav-link** — composed via `.c-nav-link[aria-current="page"]` (mint subtle background, mint text).

4. **Selection highlight** — `::selection` already defined globally as `var(--color-primary-subtle)`.

- Banner legacy ambers (RotationGraceBanner.module.css): `#fef3c7`, `#78350f`, `#d97706`, `#451a03` — replaced with **warning-tint surface** (`rgb(255 184 0 / 0.12)`, identical to `.c-badge--warning` background composition) + `var(--color-warning)` text + `var(--color-warning)` border-left.

- Gradients of any kind: `linear-gradient(180deg, #f8fbfd …)`, `radial-gradient(circle …)`, `linear-gradient(135deg, …)`. All eliminated. The redesigned chrome is flat.
- Drop shadows on cards: `box-shadow: 0 18px 45px rgba(15, 23, 42, 0.06)`, `0 24px 48px rgba(124, 45, 18, 0.08)`. All eliminated. Chrome cards use `1px solid var(--color-border)` hairlines per DESIGN.md "Elevation & Depth".

**State color is never the only signal (color-blindness rule):**
- RotationGraceBanner already pairs color with text (`Signing-secret rotation in progress.` for T-7, `Grace window ends today.` for T-0) and an optional `pulseDot` glyph. - Denied state: title text "Access Denied" carries the semantic load; the redesigned eyebrow becomes `[err] Protected workspace route` to satisfy "state color is never the only signal".

### Amendment 3: nav-link `transform` micro-motion guidance

### Amendment 4: bracketed-glyph state-coding canon for in-product UI surfaces

**Why:** DESIGN.md "Color blindness" rule says "State colors are never the only signal. Errors carry `[err]` text or an icon." The current canon describes this primarily under "CLI / Terminal output" (`[ok]`, `[err]`, `[warn]`, `[info]`). For in-product UI banners and badges, the convention is implicit but not codified. The redesign of `RotationGraceBanner` adopts `[warn]` / `[err]` prefixes on `<strong>` text — DESIGN.md should formally extend the bracketed-glyph canon to in-product UI banners + state badges to make this canonical.

**Proposed addition to DESIGN.md "Accessibility · Color blindness":**

> State badges and notice banners pair the state color with a bracketed glyph prefix at the start of the leading sentence:
> - `[ok]` for success
> - `[warn]` for warning
> - `[err]` for error
> - `[info]` for info
>
> The glyph is in JetBrains Mono (`var(--font-mono)`) when typographically reasonable. This makes the state coding readable to color-blind users, in monochrome screenshots, and in screen-reader output. The glyph can be omitted only when the surface is paired with a non-color icon or graphical state indicator.

**Workaround for Phase 213.1:** redesign adopts the convention and the implementing JSX changes are listed in the Component primitive reuse table. No blocker for Phase 213.1; flag for Phase 213.2+ to inherit the formalized rule.

### Amendment 5: `--color-on-warning-surface` and `--color-on-error-surface` text-on-tinted-surface tokens

**Why:** the redesign uses `var(--color-warning)` text on a `rgb(255 184 0 / 0.12)` warning-tint background. The existing token `--color-on-warning` is `#0A0E14` (Kernel Black) — designed for use as text on a *solid* warning fill (e.g., a fully-amber button). On the 12%-alpha tint, the warning color itself (`#FFB800`) is the text — that's how DESIGN.md specifies badges. But there's no exported token explicitly named "on-warning-tint-surface" for this case, and a contributor might mistake `--color-on-warning` (`#0A0E14`) for the right value, breaking the badge canon.

**Proposed clarifying additions (CSS comments in `app/tokens.css`, plus prose in DESIGN.md "Badges"):**

```css
  /* On 12% tint state surfaces (badges, notices), use the state color itself as text. */
  /* On 100% solid state fills, use the on-{state} pair. */
```

**Workaround for Phase 213.1:** explicit token citations in this UI-SPEC unambiguously prescribe `var(--color-warning)` on `rgb(255 184 0 / 0.12)`. No blocker.

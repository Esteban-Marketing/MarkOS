# Plan 24-01: Runtime Split Audit

## Wave 1 — Map Where Behavior Diverges

<task id="1" name="Inventory local-only and hosted-only branches">
  <read_first>
    - onboarding/backend/handlers.cjs
    - onboarding/backend/server.cjs
    - api/approve.js
    - api/submit.js
    - api/status.js
  </read_first>
  <action>
    Audit the onboarding runtime and list every place where behavior diverges by environment, including:
    - `process.env.VERCEL` branches
    - direct filesystem writes
    - `.mgsd-project.json` reads/writes
    - config loading assumptions
    - request parsing or wrapper-specific behavior
  </action>
  <acceptance_criteria>
    - Every known environment-specific branch in shared handlers and entrypoints is cataloged
    - The audit distinguishes intended divergence from accidental drift
  </acceptance_criteria>
</task>

<task id="2" name="Classify divergence by runtime responsibility">
  <read_first>
    - .planning/phases/23-identity-normalization/23-CONTEXT.md
    - .planning/phases/23-identity-normalization/23-03-PLAN.md
    - onboarding/backend/path-constants.cjs
  </read_first>
  <action>
    Group the audited differences into responsibility buckets:
    - entrypoint-only concerns
    - shared runtime concerns
    - local persistence concerns
    - hosted constraint concerns
    Identify which branches should be centralized, which should remain isolated, and which should become documented limitations.
  </action>
  <acceptance_criteria>
    - Divergence is categorized by responsibility rather than by file alone
    - Later refactors have a concrete target list instead of open-ended cleanup work
  </acceptance_criteria>
</task>

<task id="3" name="Turn the audit into Phase 24 refactor inputs">
  <read_first>
    - .planning/ROADMAP.md
    - .planning/research/ARCHITECTURE.md
    - .planning/research/PITFALLS.md
  </read_first>
  <action>
    Convert the audit into an explicit Phase 24 working contract: which runtime helpers need extraction, which behaviors need tests, and which hosted-local mismatches must be documented instead of hidden.
  </action>
  <acceptance_criteria>
    - Phase 24 implementation work can proceed from a finite, prioritized list
    - Hidden runtime assumptions become visible inputs to the next plans
  </acceptance_criteria>
</task>

## Verification

Manual review should be able to answer, for every environment-sensitive branch, whether it is entrypoint glue, shared business logic that needs centralization, or a deliberate deployment constraint.

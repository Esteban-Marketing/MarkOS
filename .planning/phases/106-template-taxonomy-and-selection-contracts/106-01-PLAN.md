<threat_model>
**Threat 1 — Malformed pack manifest JSON loaded from disk**
Risk: An operator drops a broken .pack.json into lib/markos/packs/. If parsed unchecked it could crash the loader or inject unexpected field values.
Mitigation: Wrap each `JSON.parse(fs.readFileSync(...))` call in a try/catch. Validate every parsed manifest through the compiled Ajv schema. On parse error or validation failure: `console.warn('[pack-loader] skipping invalid manifest [filename]: [reason]')` and continue — never throw at module load time. The registry simply excludes the bad pack.
Implementation: In `getFamilyRegistry()` loop, catch JSON.parse exceptions per-file; run Ajv validator and check `result.valid`; skip on failure with warn.

**Threat 2 — Path traversal via pack slug or alias values**
Risk: A slug like `../../etc/passwd` or alias used as a filesystem key could cause path traversal if slugs are ever interpolated into file paths by callers.
Mitigation: `pack-schema.json` enforces `"pattern": "^[a-z0-9][a-z0-9-]*[a-z0-9]$|^[a-z0-9]$"` on the `slug` field — only lowercase alphanumeric and internal hyphens are valid. Ajv validation runs at load time, so any manifest with a traversal-style slug is rejected before it enters the registry. Aliases are not used as file paths anywhere in this phase.
Implementation: Slug pattern enforced in JSON Schema; Ajv rejects non-conforming manifests.

**Threat 3 — Stale singleton cache after test manipulation**
Risk: Tests that modify the filesystem or swap pack content between test cases will see cached stale data if the module-level singleton is not reset.
Mitigation: Export `_resetCacheForTests()` from pack-loader.cjs. Each test group that needs a clean registry calls this function in a beforeEach/setup hook. The function sets `_registryCache = null` so the next `getFamilyRegistry()` call re-scans disk.
Implementation: `function _resetCacheForTests() { _registryCache = null; }` — always exported (production callers do not call it; no production code path invokes it).

**Threat 4 — Industry alias map lookup with user-controlled industry string**
Risk: The `seed.company.industry` value comes from user-provided onboarding input. If the resolver does eval-style dispatch or dynamic require on this value, it could be exploited.
Mitigation: `resolveIndustryOverlay(industry)` uses `canonicalizeValue()` (strips all non-alphanumeric except spaces and hyphens) then performs a plain object property lookup against the static `INDUSTRY_ALIAS_MAP` constant. No eval, no dynamic require, no filesystem path derived from the input. If the canonicalized value is not a key in the map, it returns null.
Implementation: `const slug = INDUSTRY_ALIAS_MAP[canonicalizeValue(industry)] || null;` — static object lookup only.
</threat_model>

# Phase 213.2: UI Canon Adoption Wave 2 — auth surfaces — Research

**Researched:** 2026-04-28
**Domain:** CSS Modules → DESIGN.md token migration · 4 auth surfaces · Storybook 8 CSF3 · grep-shape a11y harness
**Confidence:** HIGH (all source files read directly; UI-SPEC.md + CONTEXT.md locked; existing test contracts inventoried)

## Summary

This is **tactical execution research** for Phase 213.2. UI-SPEC.md (29 ACs across 4 surfaces) is canonical for every visual decision; CONTEXT.md (D-01..D-22) is canonical for implementation strategy. This research investigates the exact mechanics of how to migrate the 4 auth surfaces — login (12 LOC CSS / 61 LOC TSX), signup (186 / 141 + 6), invite/[token] (73 / 75), oauth/consent (253 / 265) — from legacy teal-Sora-light-bg to DESIGN.md tokens, while preserving every wiring contract caught by tests under `test/auth/`, `test/tenancy/invites.test.js`, `test/mcp/consent-ui-a11y.test.js`, and `test/onboarding-server.test.js`.

The redesign is pure className-rewrite + CSS-module-rewrite + copy-revision. DOM tree, prop interfaces, server/client boundaries, server actions, and fetch-wiring are all preserved verbatim per CONTEXT D-05. The single load-bearing JSX **change** is removal of the inline `style={{ '--accent': primary } as React.CSSProperties}}` BYOD override on login (UI-SPEC L-4) — tenants no longer brand the CTA color in this canon.

The largest test-impact discovery: **`test/mcp/consent-ui-a11y.test.js` (15 assertions) and `test/auth/signup.test.js` (last 2 of 12 assertions) both grep for the legacy class names, hex codes, fonts, and shadows we are about to delete.** These suites WILL fail post-redesign as-written and MUST be updated in lockstep. CONTEXT D-19 says "wiring contracts stay green" — but the assertion bodies in these two suites are visual-token assertions, not wiring assertions. Plan must explicitly update them; they cannot pass otherwise.

**Primary recommendation:** Plan as **5 plans, single wave, sequential** (mirrors CONTEXT D-10 with explicit cross-cutting plan): 213.2-01 login → 213.2-02 signup → 213.2-03 invite → 213.2-04 oauth/consent → 213.2-05 stories + a11y suite + test-assertion updates. Plan 213.2-05 holds the cross-cutting concerns and the breaking-test-update load that would otherwise fragment plans 01-04.

## Architecture Patterns

### Pattern 1: Module rewrite + tsx className update in lockstep

**What:** Every renamed `.module.css` class produces a corresponding edit in the `.tsx` consumer in the same plan / same commit. Acceptance: 0 broken `styles.X` references.
**When to use:** Every plan in this phase.
**Example (Phase 213.1 layout-shell precedent):**
```css
/* layout-shell.module.css — 106 LOC, token-only */
.page { min-height: 100dvh; background: var(--color-surface); padding: var(--space-md); }
.shell { max-width: var(--w-container); margin: 0 auto; display: grid; grid-template-columns: var(--w-sidebar) minmax(0, 1fr); gap: var(--space-md); }
```
[VERIFIED: Read `app/(markos)/layout-shell.module.css` — 106 LOC, zero inline hex, every value tokenized]

### Pattern 2: Form-primitive composition (D-11)

**What:** Wrap label+input+help/error trio in `.c-field`. Compose `.c-field__label`, `.c-input`, `.c-field__help`, `.c-field__error` (auto-prepends `[err] ` via `::before`).
**Example:**
```tsx
// Login form pattern
<div className="c-field">
  <label htmlFor="email" className="c-field__label">Email</label>
  <input id="email" name="email" type="email" required className="c-input" />
  <p className="c-field__help">We send a magic link instead of asking for a password.</p>
</div>
<button type="submit" className="c-button c-button--primary">Send magic link</button>
```
[VERIFIED: `styles/components.css` lines 105-159 declare these exactly]

### Pattern 3: Storybook 8 CSF3 5-state pattern (Phase 213.1 precedent)

**What:** `meta` default-export + N named state exports.
**Example:** [VERIFIED: Read `app/(markos)/_components/RotationGraceBanner.stories.tsx`]

### Anti-Patterns to Avoid

- **Re-implementing form-input styling locally:** `.c-input` exists. Never author `.input { padding: 10px 14px; ... }` or any rule that re-styles inputs.
- **Tenant-color CTA override:** UI-SPEC L-4 explicitly removes `style={{ '--accent': primary }}`. Tenants no longer brand the CTA fill.
- **Inline JSX color overrides for state:** Use `.c-field__error` primitive (auto `[err]` glyph), not `<span style={{ color: '#dc2626' }}>` or local `.errorText` classes.
- **Touch-target media query inside scope module.css:** Global `(pointer: coarse)` rule lives in `styles/components.css` line 215. Auth modules MUST NOT redeclare it (Phase 213.1 AC#15 precedent).
- **Local `@keyframes spin`:** Primitive provides spin via `.c-button.is-loading::after { animation: spin var(--duration-pulse) linear infinite; }`. Delete signup's local `@keyframes spin` block.
- **Sora/Space Grotesk fallback declarations:** Forbidden everywhere in scope. Body inherits Inter from globals; headings inherit JetBrains Mono.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Form input styling | Local `.input { ... }` rule | `.c-input` primitive | Already shipped with focus-visible mint border + ring + `aria-invalid` red border + disabled subtle bg |
| Form label uppercase + mono | Local `.label { font-weight: 600; ... }` | `.c-field__label` primitive | DESIGN.md `typography.label-caps` already encoded |
| Submit-button styling | Local `.primaryCta { background: ...; }` | `.c-button.c-button--primary` | Hover + active + disabled + focus-visible all encoded |
| Loading-state spinner | Local `<span class="spinner" />` + `@keyframes spin` | `.c-button.is-loading` (primitive provides `::after` spinner) | One source of truth; reduced-motion auto-collapsed |
| Error-text rendering with red color | Local `.errorText { color: #9a3412; }` | `.c-field__error` (auto `[err] ` prefix via `::before`) | Color-blindness rule satisfied automatically |
| Inline-code redirect URI display | Local `.redirectUri { font-family: ui-monospace, ...; }` | `.c-code-inline` primitive | Mono font + token bg + `--radius-sm` already encoded |
| OAuth scope chip rendering | Local pill chip with rounded `999px` border-radius | `.c-chip-protocol` (auto `[`/`]` brackets) | Squared signature MarkOS pattern; consumes `--color-primary-text` |
| Touch-target media query | Local `@media (pointer: coarse) { .button { min-height: 44px; } }` | Inherits from `styles/components.css` line 215 (or extension to `.c-button`) | Single source; Phase 213.1 AC#15 contract |
| Focus ring outline | Local `:focus { outline: 2px solid #0d9488; }` | Global `:focus-visible` from `globals.css` lines 121-125 | Token-cited; forced-colors aware (`Highlight`); never suppressed |
| `prefers-reduced-motion` motion-collapse | Local `@media (prefers-reduced-motion: reduce) { .x { transition: none; } }` | Global rule in `globals.css` lines 175-184 | Collapses to `0.01ms !important` automatically |
| Page-shell composition (`.c-auth-page`) | Repeat 5-line CSS recipe across 4 modules | Workaround: local `.page` class with `min-height: 100dvh; display: grid; place-items: center; ...` per UI-SPEC Amendment B-5 | Amendment B-5 not landed; documented duplication acceptable per CONTEXT D-22 |

**Key insight:** `styles/components.css` already encodes every form-authoring concern in this phase. Local module classes carry only **page composition** (centered grid, padding, viewport min-height) and **state-conditional accents** (success-tint border, error-tint border on a notice). Anything else is hand-rolling.

## Common Pitfalls

### Pitfall 1: BYOD `--accent` inline-style override removal breaks tenant branding visually but not functionally
**What goes wrong:** Removing `style={{ '--accent': primary }}` from login `<main>` (UI-SPEC L-4) means tenant primary color no longer overrides the CTA fill. Tenants who relied on this visual customization will see mint instead of their brand color.
**Why it happens:** UI-SPEC says "Single mint signal — Mint is never used as a fill on surfaces larger than a single button" + tenant-color override violates the <5% rule.
**How to avoid:** Verify with product/PM that BYOD branding via logo + name + Powered-by attribution is sufficient. If not, file a separate "tenant-color exception primitive" amendment.
**Warning signs:** A `test/auth/branding.test.js` (if exists) asserting `style="--accent:` in rendered HTML — verify before plan.

### Pitfall 2: Spinner element deletion breaks any test that asserts on `.spinner` class or `<span class="spinner">` JSX
**What goes wrong:** Signup spinner `<span className={styles.spinner} aria-hidden="true" />` (line 124 of `signup/page.tsx`) is replaced by `.c-button.is-loading` primitive's `::after`. Any test asserting on the span breaks.
**Why it happens:** Primitive composition replaces local element.
**How to avoid:** grep `test/` for `\.spinner` and `class.*spinner` — verify no current assertions exist (preliminary check: `test/auth/signup.test.js` does NOT assert on spinner; safe).
**Warning signs:** None found in current test base [VERIFIED: Read test/auth/signup.test.js].

### Pitfall 3: `consent-ui-a11y.test.js` line 35 / line 38 copy assertions are tightly coupled to legacy phrasing
**What goes wrong:** "This authorization expires in 24 hours" → "Authorization expires in 24 hours." (drops "This"). Test line 35 will FAIL.
**Why it happens:** Test was written against locked Phase 202 copy; this phase revises that copy per Brand Stance.
**How to avoid:** Plan 213.2-04 explicitly includes the test patch (see Test-update summary).
**Warning signs:** `npm test -- test/mcp/consent-ui-a11y.test.js` will fail until patched.

### Pitfall 4: `'use client'` directive removal breaks SSR/CSR boundary
**What goes wrong:** Accidental removal of `'use client'` on signup, invite, or oauth/consent breaks `useState`/`useEffect`/`fetch`/event handlers.
**Why it happens:** Aggressive cleanup or refactoring.
**How to avoid:** CONTEXT D-05 explicit: "do NOT add or remove `'use client'` directives unless UI-SPEC requires." UI-SPEC requires NO changes here. Login is server-rendered and stays server. Other 3 are client and stay client.
**Warning signs:** Build error: "useState can only be used in Client Components."

### Pitfall 5: WebAuthn / passkey reference in login page (none currently — but if product adds one mid-phase, must preserve wiring)
**What goes wrong:** A future passkey UI conditional render in login surface might be touched.
**Why it happens:** Login.tsx currently has NO passkey reference [VERIFIED: Read full file]. PasskeyPrompt lives in `components/markos/auth/PasskeyPrompt.tsx`.
**How to avoid:** Keep scope strict to the 4 surfaces in UI-SPEC. If login.tsx grows passkey conditional during phase, escalate.
**Warning signs:** `git diff` shows changes outside the 8 in-scope files.

### Pitfall 6: `aria-invalid` migration on signup `<input>` requires JSX restructure
**What goes wrong:** Current code: `className={\`${styles.input}${ui.status === 'error' ? ' ' + styles.inputError : ''}\`}`. UI-SPEC says: replace `.inputError` with primitive's `aria-invalid="true"` attribute.
**Why it happens:** Primitive uses `aria-invalid`; CSS Modules class concat is what we leave behind.
**How to avoid:** Plan 213.2-02 task explicitly: replace template-literal class concat with `aria-invalid={ui.status === 'error' ? "true" : undefined}` + `className="c-input"`.
**Warning signs:** A test asserting `class.*inputError` will fail (none currently exist).

### Pitfall 7: Inline `style={{ '--accent': primary }}` removal must be matched by removing the unused `primary` variable
**What goes wrong:** Login page currently computes `const primary = branding?.primary_color || '#0d9488'` (line 28). Removing inline style without removing `primary` leaves dead code.
**Why it happens:** Incomplete cleanup.
**How to avoid:** Plan 213.2-01 task: remove both the variable + the inline style + any test mock that references `primary_color`.
**Warning signs:** TypeScript "unused variable" warning; lint.

### Pitfall 8: `(pointer: coarse)` rule extension to `.c-button` not yet shipped
**What goes wrong:** UI-SPEC Spec line 619 says "button height bumps to 44px on coarse pointer." The current `styles/components.css` line 215 only scopes `.c-nav-link`, not `.c-button`. Auth surfaces depend on this.
**Why it happens:** Extension to `.c-button` was discussed but not yet shipped.
**How to avoid:** Verify before Plan 213.2-01: does `styles/components.css` have `.c-button { min-height: var(--h-control-touch); }` under `(pointer: coarse)`? **PRELIMINARY CHECK [VERIFIED: lines 215-217 of styles/components.css show only `.c-nav-link { padding-block: 11px; }` — `.c-button` extension NOT present.** Plan must include ext or rely on `var(--h-control)` desktop 40px (acceptable per WCAG 2.1 AA target-size minimum on coarse pointer). ## Risk Register

### R-1 (HIGH severity, MEDIUM probability) — `consent-ui-a11y.test.js` breaking 9-15 assertions in single commit

**Description:** Plan 213.2-04 (oauth/consent) inevitably touches the largest existing test surface: `test/mcp/consent-ui-a11y.test.js` has 15 assertions, of which 9 visual-token + 1 Sora-font + 1 prefers-reduced-motion + ~3 copy = ~13 BREAK under redesign.

**Mitigation:**
- Plan 213.2-04 atomic commit MUST include the test patch in the same commit (`feat(213.2-04): redesign oauth/consent module + tsx + update visual-token assertions`).
- Ensure `npm run test` passes after each plan commit (CONTEXT D-04 atomic-commit + D-21 build-green requirement).
- Researcher pre-builds the `.replaces` patch text in the executor task brief.

### R-2 (MEDIUM severity, MEDIUM probability) — Server-component login + Storybook impedance mismatch

**Description:** Login is a server component (`async function LoginPage()`). Storybook 8 with `@storybook/react-vite` cannot directly render server components without a wrapper. Story authoring requires either:
1. Extracting a presentational `LoginCard` client subcomponent (similar to NavList extraction in Phase 213.1)
2. Mocking `next/headers` and `getTenantBranding` (heavy)
3. Rendering a static visual mock without the data flow

**Mitigation:**
- Plan 213.2-05 documents the trade-off; recommended approach: extract `LoginCard` client subcomponent with `BYODBranded` prop; preserve the `async LoginPage` server component as the route entry that fetches data and passes props to `LoginCard`.
- Maps to Phase 213.1 NavList extraction precedent ("'use client' added when truly needed").

### R-3 (MEDIUM severity, LOW probability) — `.c-button` `(pointer: coarse)` rule absence

**Description:** UI-SPEC promises 44px touch targets on coarse pointer. Current `styles/components.css` only scopes `.c-nav-link`. Auth surfaces use `.c-button` exclusively (no `.c-nav-link`).

**Mitigation:**
- VERIFIED via Read: `styles/components.css` lines 215-217 only declare `.c-nav-link { padding-block: 11px; }` under `(pointer: coarse)`.
- Plan 213.2-05 (or earliest plan) includes a `styles/components.css` extension: `@media (pointer: coarse) { .c-button { min-height: var(--h-control-touch); } }`.
- Alternatively: rely on `var(--h-control)` 40px (which does meet WCAG 2.1 AA target size with hit-target padding). Document explicitly.

### R-4 (MEDIUM severity, MEDIUM probability) — OAuth/consent `useState` story authoring complexity

**Description:** OAuth/consent has 6+ internal state variables (`params`, `invalidReason`, `tenants`, `clientName`, `selectedTenantId`, `submitState`, `csrfToken`). Storybook stories for state variants need either decorator setup or component extraction.

**Mitigation:**
- Recommend extracting `ConsentCard` presentational component (Plan 213.2-04 within scope) taking all 7+ values as props. Production page.tsx wraps it with `useEffect`/`fetch` glue.
- This is a structural change beyond pure className rename — flag explicitly in Plan 213.2-04 task brief.
- Acceptance: `consent-ui-a11y.test.js` line 53 (`/oauth/authorize/approve`) still grep-passes (the URL string is in `ConsentCard`'s `onApprove` callback or in the parent — either way, the grep finds it).

### R-5 (LOW severity, MEDIUM probability) — Login BYOD `--accent` removal causes test diff in unseen branding test

**Description:** Login currently passes `style={{ '--accent': primary }}` inline. There may be a server-side branding test (not in `test/auth/`) that asserts on rendered HTML containing `--accent` style.

**Mitigation:**
- PRELIMINARY CHECK: `test/auth/`, `test/tenant-auth/`, `test/onboarding-server.test.js` — none of the inventoried tests assert on inline-style HTML.
- Run `grep -r "'--accent'" test/` before plan-phase to confirm.
- If a test exists, Plan 213.2-01 patches it.

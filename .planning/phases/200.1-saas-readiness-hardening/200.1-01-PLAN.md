Deny-list (frozen):
- IPv4 CIDRs: 127.0.0.0/8 (loopback), 10.0.0.0/8 (RFC1918), 172.16.0.0/12 (RFC1918), 192.168.0.0/16 (RFC1918), 169.254.0.0/16 (link-local incl. 169.254.169.254 cloud metadata), 0.0.0.0/8 (unspecified)
- IPv6 CIDRs: ::1/128 (loopback), fe80::/10 (link-local), fc00::/7 (unique-local), ::/128 (unspecified), ::ffff:0:0/96 (IPv4-mapped — re-validate the embedded IPv4 against IPv4 list)
- Protocols denied: file:, gopher:, data:, ftp:, dict:
- Protocols allowed: https: always; http: only when opts.allowLocalhostHttp===true AND host in {'localhost','127.0.0.1'}

DNS-pin algorithm:
1. parse URL, reject denied protocol
2. dns.lookup(hostname, { all: true, family: 0 })  → array of {address, family}
3. for each resolved address, run cidrContains against deny-list — if ANY match, reject with reason+detail
4. pick first non-blocked result (prefer family:4 for IPv4 to keep behaviour consistent with current ssrf-guard); record (ip, family)
5. return { ok: true, resolvedIp, resolvedFamily, hostname }

resolvePinnedAgent algorithm (WARNING #10 SNI fix):
- new https.Agent({ servername: hostname, lookup: (h, opts, cb) => cb(null, resolvedIp, family === 'AAAA' ? 6 : 4) })
- The `servername` Agent option is the SNI hint Node uses for the TLS ClientHello — without it, Node defaults to the connection-target IP, which fails cert hostname validation against any real cert. Setting `servername: hostname` explicitly ensures SNI uses the original hostname for cert validation while the TCP connection still targets the pre-validated IP.
- Caller ALSO sets `Host: <original-hostname>` header on the request AND passes `servername: hostname` through https.request options as belt-and-braces (some https.request paths re-derive servername from the request options instead of the Agent options).

Redirect handling (delivery.processDelivery):
- maxRedirects = 3
- on 3xx response, follow Location header; pass new URL through validateWebhookUrl AGAIN; if rejected, surface as a non-retryable failure.

       }

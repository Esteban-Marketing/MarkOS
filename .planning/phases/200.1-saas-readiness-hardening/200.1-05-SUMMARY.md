# Phase 200.1 Plan 05 Summary

Plan `200.1-05` added the missing replay-protection substrate for webhook signing without breaking the legacy `signPayload` and `verifySignature` surface that later Phase 203 work still depends on. The new structured signing path emits `t=<unix-seconds>,n=<16-byte-hex>,sha256=<digest>`, verification rejects timestamps outside the five-minute freshness window, and duplicate nonces are blocked by a dedicated nonce-store table plus periodic purge cron.

## What changed

- Added `supabase/migrations/70.1_markos_webhook_delivery_nonces.sql`.
- Added `supabase/migrations/rollback/70.1_markos_webhook_delivery_nonces.down.sql`.
- Added `lib/markos/webhooks/replay-protection.cjs` and `lib/markos/webhooks/replay-protection.ts`.
- Extended `lib/markos/webhooks/signing.cjs` and `lib/markos/webhooks/signing.ts` with additive `sign`, `parseSignatureHeader`, and async `verify` exports while preserving the legacy `signPayload`, `signPayloadDualSign`, and `verifySignature` helpers.
- Added `api/cron/webhooks-nonce-purge.js`.
- Registered the nonce purge sweep in `vercel.ts` on `*/5 * * * *`.
- Added `test/webhooks/replay-protection.test.js` with replay, stale timestamp, malformed header, malformed timestamp, signature mismatch, and nonce-store failure coverage.

## Notes

- The migration uses `subscription_id text`, not `uuid`, because the current `markos_webhook_subscriptions.id` column is still text-backed in migration `70`.
- The structured replay-protection path is additive on purpose: outbound delivery and BYOD verification keep the older header contract for now, while the new verifier path is available for the Phase 200 hardening closure and future switchover work.

## Outcome

Review concern `H2` is now closed in the codebase: there is a five-minute freshness window, a nonce-backed replay rejection path, and an automated cleanup loop for short-lived nonce rows. The repo also keeps backward compatibility with the existing Phase 203 webhook tests while exposing the new hardened signature format for the follow-on vault and endpoint work.

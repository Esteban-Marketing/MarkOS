<objective>
Ship per-tenant API-key bearer auth + sliding-window rate-limit + per-tool cost meter + billing-hold kill-switch for `api/mcp/*`. Closes review concern **H4**: "MCP session has no documented auth. DISCUSS.md says 'All new endpoints JWT-protected via requireHostedSupabaseAuth' but Claude Desktop won't carry a Supabase JWT. Without it: open-relay LLM cost vector, tenant data leakage. No per-tenant rate-limit or cost-meter mentioned either → gate-10 violation."

Purpose: Phase 200-06 shipped MCP server + 10 tools with `requireHostedSupabaseAuth` middleware, but Claude Desktop never sends a Supabase JWT — it sends `Authorization: Bearer <client-supplied>`. The gap means today's `api/mcp/*` endpoints either reject every Claude Desktop request or accept anonymous traffic — both are wrong. D-204 closes this with the 4-layer pattern from QUALITY-BASELINE gate-10 and the MCP 2025-03 spec auth model:

1. **Bearer-token issuance:** new `markos_mcp_api_keys` table; UI/CLI mints `mks_<32b-base32>` keys per tenant; only sha256 hash stored; constant-time compare via `crypto.timingSafeEqual`.
2. **Rate-limit:** sliding window 100 req/min per tenant_id, atomic SQL fn (no read-then-write race).
3. **Cost-meter:** every tool invocation writes a row to `markos_mcp_cost_events` inline (Phase 201.1 D-101 fail-closed pattern). Aggregated hourly into `markos_tenant_billing_holds` by an existing aggregator (Phase 205 will replace; for now, document the upstream consumer).
4. **Kill-switch:** middleware rejects with HTTP 402 when `markos_tenant_billing_holds.exceeded_at IS NOT NULL`.

**Plus inline tool.invoked audit row (WARNING #9 fix):** Plan 200.1-04's STRIDE register T-200.1-mcp-R-01 promises an audit row per tool invocation as the Repudiation mitigation. The original Plan 07 only wrote a cost-event row, leaving the STRIDE/code mismatch the review surfaced. This revision adds the audit emit inline alongside the cost-event emit so the tool dispatch path produces BOTH.

**Plus Task 2 split (WARNING #5):** Original Plan 07 Task 2 modified 14 files in one task (above the executor context budget). This revision splits it into Task 2a (migration ref + 3 lib modules: auth-bearer, api-keys, kill-switch + tests + TS twins) and Task 2b (rate-limit-bearer + cost-events lib modules + tests + TS twins). Wave-2 placement unchanged.

Rate-limit SQL fn `markos_mcp_increment_rate_window` shape:
```sql
create or replace function markos_mcp_increment_rate_window(p_tenant_id uuid, p_window_start timestamptz)
returns int
language plpgsql
as $$
declare
  v_count int;
begin
  insert into markos_mcp_rate_windows (tenant_id, window_start, count)
  values (p_tenant_id, p_window_start, 1)
  on conflict (tenant_id, window_start) do update set count = markos_mcp_rate_windows.count + 1
  returning count into v_count;
  return v_count;
end$$;
```

Plan note: this also requires a small `markos_mcp_rate_windows` rolling table (drop rows older than 5 min via cron).

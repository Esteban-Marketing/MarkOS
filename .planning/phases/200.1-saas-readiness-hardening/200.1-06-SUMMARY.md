# Phase 200.1 Plan 06 Summary

Plan `200.1-06` moved the primary webhook signing secret out of the plaintext subscription row and into a Vault-backed `secret_vault_ref` flow, then added a show-once rotation endpoint for re-issuing secrets without ever re-reading them from the API surface. Because this repository already contains later Phase 203 rotation work, the implementation also added a compatibility bridge so the downstream rotation-finalize migration promotes `secret_v2` back into Vault instead of trying to write to a dropped `secret` column.

## What changed

- Added `supabase/migrations/70.2_markos_webhook_secret_vault.sql`.
- Added `supabase/migrations/rollback/70.2_markos_webhook_secret_vault.down.sql`.
- Added `lib/markos/webhooks/secret-vault.cjs` and `lib/markos/webhooks/secret-vault.ts`.
- Updated `lib/markos/webhooks/engine.cjs` and `lib/markos/webhooks/engine.ts` so subscribe now stores the signing secret in Vault, persists `secret_vault_ref`, and returns `plaintext_secret_show_once` alongside a sanitized subscription object.
- Updated `lib/markos/webhooks/delivery.cjs`, `lib/markos/webhooks/delivery.ts`, `lib/markos/webhooks/store-supabase.cjs`, `lib/markos/webhooks/store-supabase.ts`, and `lib/markos/webhooks/store.ts` so dispatch resolves the primary signing secret via Vault while in-memory and Supabase stores both expose the client needed for read-time secret lookup.
- Updated `api/webhooks/subscribe.js`, `api/webhooks/list.js`, and `api/webhooks/unsubscribe.js` so public webhook responses no longer leak secret-bearing fields.

- Updated `supabase/migrations/72_markos_webhook_dlq_and_rotation.sql` so rotation finalization writes the promoted secret back into Vault instead of targeting the removed plaintext `secret` column.

## Notes

- The `secret_vault_ref` naming convention is deterministic: `markos:webhook:secret:<subscription_id>`. That made it possible to store the secret before insert rollback-cleanup and to keep the later Phase 203 finalize path compatible with the new primary-secret storage model.
- The runtime still carries a legacy fallback for raw fixture rows seeded with `subscription['secret']`. That keeps older tests and any pre-migration in-memory fixtures dispatchable while the persisted row model moves to Vault.
- The existing Phase 203 dual-sign grace-window field `secret_v2` remains unchanged in this plan. The compatibility bridge only ensures that once grace closes, the promoted primary secret lands back in Vault rather than reintroducing a plaintext steady-state column.

## Outcome

Review concern `H3` is closed for the primary webhook subscription secret path in this repository: subscribe and rotate now mint or re-issue secrets through Vault, the steady-state plaintext `secret` column is removed from the migration chain, and delivery resolves the active primary secret at dispatch time instead of reading it from the subscription row. The later Phase 203 rotation substrate also remains coherent on fresh installs because rotation finalization now promotes the grace-window secret back into Vault instead of targeting a column that no longer exists.

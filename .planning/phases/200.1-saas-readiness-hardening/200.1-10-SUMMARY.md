# Phase 200.1 Plan 10 Summary

Plan `200.1-10` closes review concern `H5` by replacing the original open demo proxy with a four-layer sandbox: BotID-gated session issue, a 15-minute signed `demo_session_token`, a server-enforced tool subset of `draft_message` + `audit_claim`, and a per-token total cost cap of `$0.50`.

## What changed

- Added `lib/markos/marketing/demo-sandbox.cjs` plus `lib/markos/marketing/demo-sandbox.ts`.
  - Exports `issueDemoSessionToken`, `verifyDemoSessionToken`, `assertToolAllowed`, and `recordDemoCost`.
  - Tokens are HS256 JWTs signed with `MARKOS_DEMO_TOKEN_SECRET`.
  - Claims are pinned to `aud='demo-sandbox'`, `iss='markos.dev'`, `allowed_tools=['draft_message','audit_claim']`, `cost_cap_cents=50`, and the synthetic sandbox tenant.
  - Signature verification uses `crypto.timingSafeEqual`.
- Added `app/(marketing)/integrations/claude/demo/api/issue-token/route.ts`.
  - Wraps the route in `withSpan('demo.issue_token', ...)`.
  - Issues demo tokens after BotID verification succeeds in the shared sandbox lib.
- Added `app/(marketing)/integrations/claude/demo/api/invoke/route.ts`.
  - Wraps the route in `withSpan('demo.invoke', ...)`.
  - Verifies the token, checks the allowed tool subset, meters cost before dispatch, and returns `402` on `cost_cap_exceeded`.
  - Uses explicit hand-dispatch for the two allowed tools instead of modifying the Phase 200 MCP registry surface.
- Replaced the legacy open proxy at `app/(marketing)/integrations/claude/demo/api/route.ts`.
  - It now returns a deprecation response instead of leaving the original unauthenticated invoke path active.
- Updated `app/(marketing)/integrations/claude/demo/page.tsx`.
  - Added the persistent `Demo mode` banner.
  - Added the BotID-gated `Try the demo` flow.
  - Restricted the visible tool buttons to `draft_message` and `audit_claim`.
  - Added client-side expiry reset and cost-cap messaging while keeping server-side enforcement authoritative.
- Added `test/marketing/demo-sandbox-abuse.test.js`.
  - Covers BotID failure, valid token issue, signature tampering, token expiry, tool subset enforcement, cap enforcement, and DB failure handling.
- Added `scripts/verify/verify-200-1-10-task-2.cjs`.
  - Verifies the two new routes, direct hand-dispatch, `recordDemoCost`, and the `402` cost-cap branch.

## Repo-reality adaptations

- `draft_message` was still an inline descriptor inside `lib/markos/mcp/tools/index.cjs`, not a standalone tool file. To keep the Phase 200 registry unchanged, this closeout added `lib/markos/mcp/tools/marketing/draft-message.cjs` as a narrow wrapper for the demo route only.
- `marketing` was not yet a valid audit `source_domain`, so `lib/markos/audit/writer.cjs` was extended to allow demo-session audit rows.
- The DB-backed cost counter path has real foreign keys through `markos_orgs`, `markos_tenants`, `markos_mcp_sessions`, and `markos_mcp_cost_events`. The sandbox therefore bootstraps a synthetic org/tenant/session chain before writing cost rows when a service-role Supabase client is available.
- For local environments that do not expose Supabase service-role env vars, `recordDemoCost` falls back to an in-memory per-token counter so the sandbox still runs locally. Production keeps the DB-backed path.

## JWT shape and sandbox model

- JWT payload:
  - `iss: 'markos.dev'`
  - `aud: 'demo-sandbox'`
  - `sub: 'demo:<uuid>'`
  - `iat`
  - `exp = iat + 900`
  - `allowed_tools = ['draft_message', 'audit_claim']`
  - `cost_cap_cents = 50`
  - `synthetic_tenant_id = '00000000-0000-0000-0000-000000000001'`
- The invoke route does not hit `/api/mcp/*`.
  - It bypasses the marketplace bearer middleware and dispatches the two approved tools directly under the synthetic tenant context.
  - `lib/markos/mcp/tools/index.cjs` remains unchanged.

## Notes

- The prior marketing a11y contract still asserted the old single-endpoint demo proxy. This closeout updates the stale `DS-8` assertions in `test/ui-a11y/213-5-marketing-a11y.test.js` to the hardened `issue-token` + `invoke` flow.
- The per-token cost cap still accepts the small race window documented in the plan: concurrent invokes can observe the pre-insert total before the next write lands. That is acceptable for this marketing-only sandbox.

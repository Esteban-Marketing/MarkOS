# Phase 200.1 Plan 07 Summary

Plan `200.1-07` hardened the MCP HTTP surface with tenant-minted `mks_...` bearer keys, a SQL-backed per-tenant minute throttle, inline per-tool cost-event recording, and a billing-hold kill-switch across both `POST /api/mcp/session` and the legacy `POST /api/mcp/tools/{toolName}` dispatch path. The implementation preserves the older opaque MCP session-token flow already present in Phase 202, so existing clients keep working while the new bearer-key path closes review concern `H4`.

## Notes

- The plan’s original migration sketch used `uuid` ids, but the real repository schema is still text-backed for `markos_tenants`, `markos_orgs`, and `markos_mcp_sessions`. The new MCP auth/cost tables follow that existing text-id convention to stay FK-compatible.
- The plan text also referenced `markos_tenant_billing_holds.exceeded_at`, but the real billing hold substrate in this repo is `tenant_billing_holds` with `hold_state` and `released_at`. The kill-switch consumes the actual table shape already present on disk rather than introducing a second hold source.
- Because this repo already had a mature Phase 202 MCP pipeline, the safest change was to extend that pipeline with an `auth_context` path instead of creating a second parallel invocation stack. That keeps the stricter input validation, approval-token, cost-admission, and output-validation behavior in one place.
- I attempted the standalone `test/mcp/pipeline.test.js` suite during implementation, but the local environment does not currently have the declared `ajv` package installed. The final verification therefore used the new focused bearer/cost/kill-switch suites, the updated `server.test.js` suite, and the standalone wiring verifier.

## Outcome

Review concern `H4` is closed in the codebase: MCP HTTP calls can now be tenant-authenticated with hashed `mks_...` bearer keys, rate-limited at 100 requests per minute per tenant, blocked behind a billing hold, and measured with inline per-tool cost telemetry. The legacy direct tool route is no longer a security bypass, and the promised `tool.invoked` audit trail from the Phase 200.1 STRIDE register is now emitted inline before the response is returned.

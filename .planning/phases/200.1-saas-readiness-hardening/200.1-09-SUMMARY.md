# Phase 200.1 Plan 09 Summary

Plan `200.1-09` closes review concern `M5` by shipping a shared OTEL wrapper for the SaaS control-plane handlers, wiring spans into the webhook and MCP API surfaces, and adding a CI gate that fails if future `api/webhooks/**` or `api/mcp/**` handlers are added without `withSpan(...)` coverage.

## What changed

- Added `lib/markos/observability/otel.cjs` plus `lib/markos/observability/otel.ts`.
  - Exports `initOtel`, `startSpan`, `withSpan`, `recordEvent`, `setTestSink`, and `clearTestSink`.
  - Sink precedence is `Sentry > JSON console lines`, and sink failures are fail-soft.
  - Span/event payloads drop `undefined` values before emission.
- Added `test/observability/otel.test.js` to cover the shared wrapper contract:
  - span creation + `end()`
  - attribute propagation
  - resolve/throw behavior for `withSpan(...)`
  - one-shot `recordEvent(...)`
  - sink-failure isolation
- Wired OTEL into the 10 plan-target handlers and also the remaining matching handler files so the coverage gate is strict instead of partial:
  - `api/webhooks/subscribe.js` -> `webhook.subscribe`
  - `api/webhooks/unsubscribe.js` -> `webhook.unsubscribe`
  - `api/webhooks/list.js` -> `webhook.list`
  - `api/webhooks/test-fire.js` -> `webhook.test_fire`
  - `api/webhooks/rotate-secret.js` -> `webhook.rotate_secret`
  - `api/webhooks/resend-events.js` -> `webhook.resend_events`
  - `api/webhooks/twilio-events.js` -> `webhook.twilio_events`
  - `api/webhooks/vercel-domain.js` -> `webhook.vercel_domain`
  - `api/mcp/session.js` -> `mcp.session`
  - `api/mcp/tools/[toolName].js` -> `mcp.tool`
  - `api/mcp/keys/create.js` -> `mcp.keys.create`
  - `api/mcp/keys/list.js` -> `mcp.keys.list`
  - `api/mcp/keys/revoke.js` -> `mcp.keys.revoke`
  - `api/mcp/session/cleanup.js` -> `mcp.session.cleanup`
- Added explicit one-shot events where the plan required higher-signal audit/telemetry moments:
  - `recordEvent('webhook.test_fired', ...)`
  - `recordEvent('webhook.secret_rotated', ...)`
  - `recordEvent('mcp.tool.invoked', ...)`
- Added `scripts/verify/verify-200-1-09-task-2.cjs` as the Windows-safe standalone verifier for the handler wiring checks.
- Added `scripts/ci/check-otel-coverage.cjs` plus `.github/workflows/otel-coverage.yml` to enforce route-level OTEL coverage on future PRs.

## Attribute schema and gate design

- The shared wrapper now emits the required structural attributes when resolvable: `tenant_id`, `webhook_subscription_id`, `mcp_session_id`, `tool_name`, `cost_cents`, `status_code`, `req_id`, `error`, and `duration_ms`.
- The coverage gate walks `api/webhooks/*.js` and `api/mcp/**/*.js`, asserts `withSpan(` is present, and uses an explicit in-file allowlist policy for any future infra-only exceptions.
- The workflow mirrors the recent eval lane pattern by setting `NODE_OPTIONS=--test-isolation=none`, which keeps the OTEL unit-test step consistent with the repo's other Node test gates.

## Notes

- `api/mcp/session.js` had a latent scope bug in the resource-read path (`supabase` was referenced out of scope). The OTEL pass fixed that to use `auth.supabase`.
- `test/webhooks/ssrf-guard.test.js` had stale reason-code assertions from the earlier validator contract. The closeout updates those checks to the current `ipv6_loopback` and `protocol_http_denied` reason values so the focused regression suite matches the hardened SSRF behavior shipped in `200.1-01`.

## Outcome

`200.1-09` now gives the SaaS hardening phase a shared, fail-soft observability layer instead of route-by-route ad hoc logging. All current webhook and MCP handlers emit named spans, the MCP tool lane records explicit invocation events with cost/session context, and new handlers under `api/webhooks/**` or `api/mcp/**` now fail CI if they ship without OTEL coverage. Review concern `M5` is closed.

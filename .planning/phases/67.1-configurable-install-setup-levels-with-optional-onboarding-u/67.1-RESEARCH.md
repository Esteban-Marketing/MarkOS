## Summary

Phase 67.1 should be an installer-contract phase that adds setup profiles without changing the core vault-first default. Current behavior already has one implicit profile: install protocol files, copy onboarding app if missing, write `.markos-install-manifest.json`, compute install readiness, and auto-launch onboarding unless `--no-onboarding` is provided. This behavior is implemented in `bin/install.cjs`, with shared CLI parsing in `bin/cli-runtime.cjs`, update carry-forward in `bin/update.cjs`, and readiness/status surfaces split between installer output and `/status` handlers.

The safest design is to formalize this implicit behavior into explicit install profiles, persist profile metadata in the manifest, and derive runtime behavior from that metadata. Existing installs with no profile should resolve to current behavior (`full`) to preserve backward compatibility. Non-interactive automation must be deterministic with strict precedence rules across profile flags and onboarding toggles.

**Primary recommendation:** Implement profile selection + manifest persistence first, then runtime gating and status visibility, then migration/update normalization and tests.

## Recommended Profile Model

### Profile Names and Intent

| Profile | Intent | Included by default | Excluded by default |
|---|---|---|---|
| `full` | Current experience parity | Vault bootstrap, onboarding assets, onboarding auto-launch in interactive sessions (unless disabled), full readiness messaging | None (parity profile) |
| `cli` | Vault + CLI daily usage, no browser helper handoff | Vault bootstrap, CLI commands, updater, status surfaces | Onboarding auto-launch, onboarding-required readiness gating |
| `minimal` | Agentic CI/headless with smallest required footprint | Vault bootstrap, manifest, essential CLI/runtime | Onboarding asset copy, onboarding auto-launch, UI/helper messaging except explicit profile summary |

### Rationale

- `full` preserves current contract and satisfies backward compatibility.
- `cli` creates first-class opt-out for onboarding/UI while keeping normal CLI workflows.
- `minimal` supports deterministic headless installs and avoids optional local helper surfaces.
- Three profiles keep behavior understandable without plugin-level complexity.

## Backward-Compatibility Contract

1. Existing installs with no `install_profile` in `.markos-install-manifest.json` resolve to `full`.
2. Existing CLI flags continue to work:
- `--no-onboarding` still suppresses auto-launch in `full`.
- `--yes` keeps non-interactive defaults.
3. No current command path changes are required for users not opting into new profile flags.
4. `npx markos update` should write `install_profile: "full"` when absent and preserve behavior parity.

## Non-Interactive Flag Contract and Precedence

### Proposed Flags

- `--profile <full|cli|minimal>`
- `--cli-only` alias for `--profile cli`
- `--minimal` alias for `--profile minimal`
- Keep `--no-onboarding` as a component override
- Keep `--yes` for acceptance/defaulting behavior

### Precedence Rules (highest to lowest)

1. Explicit profile flags (`--profile`, `--cli-only`, `--minimal`)
2. Explicit component override flags (`--no-onboarding`)
3. Non-interactive mode policy (`--yes`, CI detection, `MARKOS_FORCE_NON_INTERACTIVE`)
4. Manifest remembered profile (update/reinstall continuity)
5. Default fallback (`full`)

### Resolution Rules

- If profile is `cli` or `minimal`, onboarding auto-launch is always `false` regardless of interactivity.
- If profile is `full`, onboarding auto-launch follows current logic (`interactive && !CI && !--no-onboarding`).
- Conflicting flags should fail fast with actionable error (example: `--profile minimal --global` is valid, but multiple profile aliases with different values should error).

## Manifest/Storage Contract

### Manifest Additions

Add fields to `.markos-install-manifest.json`:

- `install_profile`: `full|cli|minimal`
- `profile_schema_version`: `1`
- `components`: object booleans
- `onboarding_enabled`
- `onboarding_auto_launch`

### Suggested Shape

```json
{
  "manifest_schema_version": 3,
  "profile_schema_version": 1,
  "install_profile": "full",
  "components": {
    "vault_bootstrap": true,
    "onboarding_assets": true,
    "onboarding_ui": true,
    "onboarding_server": true,
    "cli_runtime": true
  },
  "onboarding_enabled": true,
  "onboarding_auto_launch": true
}
```

### Storage Decision

- Source of truth: manifest (`.markos-install-manifest.json`).
- Runtime config (`onboarding/onboarding-config.json`) may mirror state for local helper UX, but should not be authoritative for install profile selection.

## Migration/Update Strategy

### Existing Install Migration

1. On `npx markos update`, read manifest.
2. If `install_profile` missing:
- set `install_profile` to `full`
- set `profile_schema_version` to `1`
- derive `components` for parity behavior
3. Preserve existing vault continuity repair behavior in `bin/update.cjs`.
4. Never delete onboarding assets automatically during migration; optional cleanup can be future scope.

### New Install Strategy

1. Parse and resolve profile + overrides.
2. Build install plan object (deterministic component booleans).
3. Execute component plan.
4. Persist resolved profile and plan into manifest.
5. Print profile summary and enabled/disabled components in installer output.

## Common Pitfalls

### Pitfall 1: Hidden profile drift between install and update

- What goes wrong: install writes profile, update ignores it.
- Why it happens: profile logic only implemented in installer.
- How to avoid: centralize profile resolve and manifest normalize logic; call from both install and update.

### Pitfall 2: Flag ambiguity in non-interactive mode

- What goes wrong: CI runs produce different results by environment.
- Why it happens: interactivity checks race with flags.
- How to avoid: enforce explicit precedence and print resolved profile plan in logs.

### Pitfall 3: Runtime crashes when onboarding disabled

- What goes wrong: CLI/status paths assume onboarding files exist.
- Why it happens: current code frequently imports onboarding handlers directly.
- How to avoid: treat onboarding as optional component and gate entrypoints with clear messages.

### Pitfall 4: Breaking old installs with strict schema migration

- What goes wrong: manifest v2 installs fail after profile rollout.
- Why it happens: required fields introduced without defaults.
- How to avoid: additive schema with default `full` and migration-on-read.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Profile logic duplicated across files | Inconsistent behavior | Add shared resolver helper used by install + update + status surfaces. |
| Old manifest installs become invalid | Upgrade failures | Migration-on-read defaults with additive schema only. |
| Optional onboarding introduces dead code paths | Runtime errors | Component gate checks before launch/import; clear operator messages. |
| Documentation lags contract | Operator confusion | Update README install examples and readiness descriptions in same phase. |

## Clear Recommendations and Implementation Order

1. Add profile contract and resolver in CLI layer.
2. Extend installer to resolve profile, apply component plan, and persist profile metadata in manifest.
3. Extend updater to normalize old manifests to `install_profile: full` and preserve profile on update.
4. Add status/readiness exposure for active profile and enabled components.
5. Add profile matrix tests (install + update + selected status checks).
6. Update README install section with profile examples and non-interactive usage contract.

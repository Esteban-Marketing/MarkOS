# Phase 206 — UI Design Contract (no-UI-scope)

> **Phase 206 ships zero UI surfaces.** This document is the explicit no-surface declaration
> for the SOC 2 Type I Foundation phase. It exists so that gsd-ui-checker, gsd-planner, and
> gsd-ui-auditor have an unambiguous contract: there is no `app/`, no `components/`, no
> `*.stories.tsx`, no `page.tsx`, no `*.module.css`, and no `*.css` modified or created in
> any of the seven plans (206-01 through 206-07).
>
> What Phase 206 *does* ship is **control doctrine** (`.planning/compliance/*.md`),
> **typed control modules** (`lib/markos/compliance/*.ts`), **scripts**
> (`scripts/compliance/*.mjs`), and **tests** (`test/compliance/phase-206/**/*.test.js`).
> None of those files compose, import, or render any visual primitive from
> `styles/components.css` or any token from `app/tokens.css`.
>
> However, **every downstream phase (207–217) that consumes a Phase 206 control contract
> WILL eventually need a UI surface** — exception-register approval pages, evidence-export
> consoles, public-claim editor surfaces, etc. This UI-SPEC therefore also serves as a
> forward-looking inheritance map so future UI-SPECs can cite their lineage back to the
> control doctrine defined here.
>
> Authority chain: DESIGN.md v1.1.0 → 213.x adoption-wave decisions (D-08, D-09, D-09b,
> D-13, D-14, D-15) → this document. Generated by gsd-ui-researcher. Status: draft
> (checker upgrades to approved once the no-UI declaration is verified).

## Downstream UI Inheritance Map

This is the load-bearing section of this document. When a future phase ships a UI surface
that consumes a Phase 206 control contract, that phase's UI-SPEC.md MUST cite the row below
that authorizes its surface family. This binds the visual contract back to the control
doctrine and prevents drift.

All downstream UI surfaces below MUST author to **DESIGN.md v1.1.0** and inherit the 213.x
adoption-wave decisions (carried forward from CONTEXT.md decisions D-08 through D-15 of
the 213.3 / 213.4 waves):

| 213.x Decision | Carry-forward rule for any future surface that consumes 206 doctrine |
|----------------|---------------------------------------------------------------------|
| D-08 (token-only) | Zero inline hex literals; every color via `var(--color-*)`; every spacing via `var(--space-*)`; every typography via DESIGN.md `typography.*` token. |
| D-09 (mint-as-text) | Protocol Mint `#00D9A3` allowed as text via `--color-primary-text` for `.c-button--tertiary` link CTAs and `.c-chip-protocol` IDs only; never as fill on surfaces larger than a button or chip. |
| D-09b (`.c-notice` mandatory) | Every status, approval-gate, evidence-pending, exception-pending, or recovery-required notice composes `.c-notice c-notice--{info,warning,success,error}` from `styles/components.css`. No local `.banner`/`.alert`/`.warning`/`.noticeBar` classes. |
| D-13 (`.c-card--feature` reserved) | `.c-card--feature` is reserved for hero panels in `404-workspace` + `213.5` marketing. Any future SOC 2 surface (exception-register, evidence-export console, public-proof editor) uses `.c-card` default — never `.c-card--feature`. |
| D-14 (no `.c-table` primitive) | Any future tabular surface (control-registry browser, evidence-export index viewer, exception-register table) uses vanilla `<table>` semantic + token-only recipe on `<th>`/`<td>` + `.c-badge--{state}` for row state. The `.c-table` primitive remains deferred to Phase 214+. |
| D-15 (selective extraction) | When a future phase extracts a control-doctrine read pattern into a reusable component, the extraction is selective: pages co-locate with their control doctrine first, primitives extract only when reuse is proven across ≥2 surfaces. |

## Copywriting Contract

**End-user / surface copy: not applicable.** Phase 206 emits no copy to any human surface.

**Doctrine prose copy** in `.planning/compliance/*.md` is governed by:
- The CLAUDE.md banned lexicon (`synergy`, `leverage`, `empower`, `unlock`, `transform`,
  `revolutionize`, `supercharge`, `holistic`, `seamless`, `cutting-edge`, `innovative`,
  `game-changer`, `next-generation`, `world-class`, `best-in-class`, `reimagine`,
  `disrupt`, `just` as softener — and no exclamation points).
- Auditor-readability: short sentences, concrete control IDs, named owners. No marketing
  voice. No hedging language ("we strive to", "best-effort"). Every control row asserts
  a specific test procedure or evidence path.
- Pricing-placeholder rule: `{{MARKOS_PRICING_ENGINE_PENDING}}` rendered verbatim in
  `pricing-billing-controls.md` per acceptance criterion `206-03-01`.
- Forbidden architecture-lock strings (rejected by `check-architecture-lock.mjs` per
  acceptance criterion `206-01-00.AC.3`): `shadow compliance store`,
  `implementation ownership equals compliance ownership`,
  `future module treated as implemented`, `enterprise claim without proof`,
  `approval bypass exception`.

These are doctrine-prose rules, not UI copywriting rules — included here only because
the orchestrator's downstream inheritance question requires future UI surfaces consuming
this doctrine to honor the same lexicon discipline when they render any control description
into a UI label, banner, or modal body.

## Destructive Actions

**Not applicable in Phase 206.** No surface ships, so no confirm-modals are authored.

When future phases ship surfaces that mutate any 206-doctrine state (e.g. acknowledging an
exception, marking a control test as passed, exporting a Type I package), each such
mutation MUST:
1. Map to one of the six `mutation_class` literals from 206-02 (`external.send`,
   `billing.charge`, `connector.mutate`, `price.change`, `public.claim`, `data.export`).
   Acknowledging a control or exporting evidence falls under `data.export`.
2. Compose `.c-modal` + `.c-backdrop` + `.c-button--destructive` per the 213.x and 205
   pattern.
3. Honor the `default_approval_mode` declared in `mutation-class-policy.md` for that class.

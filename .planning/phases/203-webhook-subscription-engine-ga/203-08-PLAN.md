<objective>
Ship the **circuit breaker** — Plan 203-04's D-14 (>50% of last 20 deliveries fail = trip) and D-15 (half-open exponential backoff 30s → 60s → 120s → 300s → 600s, capped at 600s) enforced via Upstash Redis sliding-window state per RESEARCH §Pattern 4.

**Revised integration shape** (resolves Wave-4 same-file conflict on `delivery.cjs`):
1. **State library** `lib/markos/webhooks/breaker.cjs` — `recordOutcome` (LPUSH + LTRIM + threshold check), `canDispatch` (state-machine gate), `classifyOutcome` (5xx/timeout → failure; 4xx → NOT failure), `getBreakerState` (read-only for dashboard — Plan 203-09 consumes). Exports both the gate primitives AND the post-fetch outcome primitives (`recordOutcome`, `classifyOutcome`) for Plan 203-10 to consume.
2. **Dispatch-gates extension** — ADD a breaker check as the FIRST gate inside `runDispatchGates` in `lib/markos/webhooks/dispatch-gates.cjs` (the scaffold Plan 203-07 shipped). This is purely additive — Plan 203-07's rate-limit gate becomes the SECOND gate. `delivery.cjs` is NOT touched by this plan (Plan 203-07's single pre-fetch insertion point already routes through `runDispatchGates`).
3. **Integration suite** — `test/webhooks/circuit-breaker.test.js` exercises the full state machine (closed → open → half-open probe success → closed; open → half-open probe fail → open longer) by driving `runDispatchGates` with a seeded Redis mock.

**Wave placement:** This plan moves from Wave 4 to **Wave 5** with `depends_on: [01, 02, 07]`. Plan 203-07 must land first because Plan 203-08 extends `dispatch-gates.cjs` additively. Wave 5 siblings (09, 10) and Plan 08 have non-overlapping `files_modified` after this restructure — no same-wave conflict.

**Post-fetch outcome recording is Plan 203-10's responsibility.** This plan ships `recordOutcome` + `classifyOutcome` as PURE EXPORTS from `breaker.cjs`. Plan 203-10's observability wrapper in `delivery.cjs` imports these and calls them inside its own finally block (co-located with the log-drain emit). That prevents a same-wave `delivery.cjs` conflict between Plan 08 and Plan 10.

Why Redis, not in-memory: RESEARCH §Pitfall 2 — Fluid Compute silently loses in-memory state on instance turnover. Upstash Redis sliding-window already proven via 202-04 pattern.

Key accepted tradeoff: The LPUSH/LTRIM race window for ~20-sample 50% threshold is documented as A3 in Assumptions Log — noise floor of a handful of false-positive trips per thousand invocations is tolerable.

Output: breaker library + dispatch-gates.cjs extension (breaker as FIRST gate) + 2 test suites. Zero edits to `delivery.cjs`.

async function runDispatchGates({ subId, tenantId, eventId, planTier, subscription, redis }) {
  // GATE 1: breaker (this plan adds). On state=open short-circuits everything.
  const brk = await canDispatch(redis, subId);
  if (!brk.can_dispatch) {
    const retryAfterSec = Math.max(1, Math.ceil((brk.probe_at - Date.now()) / 1000));
    return { status: 'breaker_open', retryAfterSec, reason: 'breaker_open', breaker: { state: brk.state, trips: brk.trips, probe_at: brk.probe_at } };
  }
  // GATE 2: rate-limit (Plan 203-07 shipped).
  const rate = await checkWebhookRateLimit(redis, { subscription, plan_tier: planTier });
  if (!rate.ok) return { status: 'rate_limited', retryAfterSec: rate.retry_after, limit: rate.limit, reason: rate.reason || 'sub_rps' };
  return { status: 'allowed' };
}

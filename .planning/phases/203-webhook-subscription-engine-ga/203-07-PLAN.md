<objective>
Ship Wave 4's **per-subscription rate-limit** enforcing **D-13**: plan-tier defaults Free 10rps / Team 60rps / Enterprise 300rps; tenant admin may override per-sub but only to LOWER (never raise) the ceiling.

Two integration points:
1. **At subscribe-time** (`api/webhooks/subscribe.js`) — validate the `rps_override` body param against the plan ceiling; reject with 400 if higher.
2. **At dispatch-time** (`lib/markos/webhooks/delivery.cjs`) — pre-fetch rate-limit check via `@upstash/ratelimit` sliding-window keyed `rl:webhook:sub:{sub_id}`. On cap-breach, postpone the delivery via the existing engine.cjs backoff shape (do NOT mark failed — rate-limit breach is transient, not a DLQ event).

F-100 contract declares the breaker + rate-limit READ surface (exposed via dashboard APIs in Plan 203-09); actual breaker logic ships in Plan 203-08 (parallel Wave 4 sibling).

Purpose: Prevents rogue subscriber endpoints (or misbehaving publishers) from inducing runaway dispatch; lower plan tiers get lower caps aligned with D-13 monetization tier. D-13 explicitly shares the Upstash instance from 202-04 — zero new deps.
Output: rate-limit library + delivery.cjs integration + subscribe.js validation + F-100 contract + 2 test suites.
</objective>

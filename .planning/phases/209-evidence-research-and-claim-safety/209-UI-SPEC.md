# Phase 209 — UI Design Contract (hybrid)

> **Phase 209 ships ONE UI sub-component plus five no-UI backend doctrine plans.** This is
> a hybrid scope. The single UI surface is `evidence-summary.tsx`, mounted inside the
> Phase 208 Approval Inbox detail card. The other five plans (209-01, 209-02, 209-03,
> 209-05, 209-06) ship `.planning/evidence/*.md` doctrine, `lib/markos/evidence/*.ts`
> typed modules, Supabase migration `evidence_map_foundation.sql`, CI assertion scripts
> under `scripts/evidence/`, and tests under `test/evidence/phase-209/**/*.test.js`. None
> of those five plans modify any `app/`, `components/`, `*.stories.tsx`, `page.tsx`,
> `*.module.css`, or `*.css` path.
>
> The single UI sub-component (Plan 209-04) closes the Phase 208 translation gate:
> Phase 208 Surface D (Approval Inbox) renders `<PlaceholderBanner variant="future_phase_209">`
> when `evidence_status == 'waiting_phase_209'`. Phase 209-04 replaces that placeholder
> with a real evidence panel that exposes `EvidenceMap` records, claim risk classification,
> source quality summary, freshness/TTL summary, known-gap list, and the override path
> for unsupported-claim approvals — without changing the parent Approval Inbox shell.
>
> Authority chain: DESIGN.md v1.1.0 → 213.x adoption-wave decisions D-08..D-15 →
> 206-UI-SPEC (mutation-class + evidence-required origin) → 207-UI-SPEC (orchestration
> envelope origin) → 208-UI-SPEC §Surface D (parent Approval Inbox surface) →
> this document. Generated by gsd-ui-researcher. Status: draft (checker upgrades to
> approved once the hybrid declaration is verified and the single-surface contract is
> validated).

## Spacing Scale

Authoring rule: every `padding`, `margin`, `gap`, `inset` in scope MUST cite a
`--space-*` token. No arbitrary px. Off-grid values are auto-FAIL.

The sub-component does NOT introduce a new CSS module. It composes its layout via
`gap` utilities in `styles/components.css` and inherits parent
`app/(markos)/operations/approvals/page.module.css` for outer composition. Tokens
consumed directly:

| Token | Value | DESIGN.md citation | Usage in this surface |
|-------|-------|--------------------|---------------------|
| `--space-none` | 0 | `spacing.none` | Reset margins on `<h3>`, `<h4>`, `<p>`, `<table>`, `<ul>` |
| `--space-xxs` | 2px | `spacing.xxs` | Badge inner padding, status-glyph adjacency, freshness-status chip inner spacing |
| `--space-xs` | 8px | `spacing.xs` | Evidence-row icon-text adjacency, source-quality chip group gap, known-gap inline-meta gap, override CTA pre-spacing |
| `--space-sm` | 16px | `spacing.sm` | Evidence-section vertical rhythm, notice padding-block, table `th`/`td` padding-block, override-reason input top spacing, mobile horizontal sub-component padding (max-width ≤ 640px) |
| `--space-md` | 24px | `spacing.md` | Evidence-summary section gap (between Claim, Sources, Freshness, Known gaps, Inference), inline override-modal padding (modal owned by parent; sub-component only contributes interior padding), evidence-rail outer padding when mounted in detail card |
| `--space-lg` | 32px | `spacing.lg` | Reserved — only used if evidence-summary becomes a standalone page in a future phase (not in 209) |

**Allowed exceptions (DESIGN.md documented):**
1. `1px` for hairline borders (`var(--color-border)`) — preferred over shadows on dark surfaces (DESIGN.md "Borders over shadows"). Used between evidence sections and on table `th`/`td` border-bottom.
2. `2px` for focus ring width and offset (`var(--focus-ring-width)`, `var(--focus-ring-offset)`); never suppressed.
3. `4px` for `.c-notice` `border-inline-start` accent — composed via `.c-notice` primitive; this sub-component never declares this directly.
4. **Mobile touch target `44px`** via `--h-control-touch` on `.c-button` for `(pointer: coarse)` viewports — already declared globally in `styles/components.css`; this sub-component consumes it without re-declaration.

## Destructive Actions

This sub-component ships ONE destructive action: **Override an unsupported-claim
block** ("Continue without evidence" CTA).

| Action | Surface | Confirm pattern | Mutation-class binding |
|--------|---------|-----------------|------------------------|
| Override an unsupported-claim block | evidence-summary.tsx | `.c-input` (override_reason; required, ≥10 chars) + `.c-button--destructive` "Continue without evidence". When the parent `ApprovalItem.mutation_family` maps to a 206 mutation class with `default_approval_mode == 'dual_approval'`, the CTA opens the parent's existing dual-approval modal (`.c-modal` + `.c-backdrop`) per 208-04 — this sub-component does NOT re-author the modal | Underlying mutation class (one of 206's six root classes); override is recorded as a `data.export`-class operator action on the audit log because it produces an audit-log row with override_reason + actor_id; commit emits an `agent_run_close_incomplete` or `agent_run_close_completed` event on the linked run per 207-04 doctrine |

**Override commit lineage (verbatim from 209-04 doctrine):**
1. Operator types reason in `.c-input`.
2. Click `.c-button--destructive` "Continue without evidence".
3. (Dual-approval only) parent's modal opens; second approver issues token.
4. `commitOverride(snapshot, reason, run_id) -> AuditEvent` writes:
   - `ApprovalEvidenceSnapshot.override_reason = reason`
   - `ApprovalEvidenceSnapshot.actor_id = current_actor_id`
   - Audit-log row with mutation-class binding + run_id + reason + actor_id + timestamp.
5. Parent Approval Inbox re-derives `canApprove == true` and Approve CTA enables.
6. Sub-component renders `.c-notice c-notice--warning` "[warn] Approved without evidence. Override recorded …".

**Override blocked path (autonomy-ceiling case):**
- When 206 `autonomy_ceiling` is reached for the mutation class AND override is
  attempted, `claim-blocking.ts` `canOverride(snapshot, mutation_family,
  ceiling_state)` returns `{ allowed: false, reason: 'autonomy_ceiling_reached' }`.
- Sub-component renders `.c-notice c-notice--error` "[block] Override blocked:
  autonomy ceiling reached and evidence missing." above the (disabled)
  override-reason input.
- Compliance-review escalation path is documented in 206 doctrine; this surface
  only displays the block — no escalation UI is shipped in 209.

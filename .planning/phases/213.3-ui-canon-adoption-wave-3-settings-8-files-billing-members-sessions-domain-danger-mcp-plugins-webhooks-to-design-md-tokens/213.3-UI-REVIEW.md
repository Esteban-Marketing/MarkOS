## Top 3 Priority Fixes

1. **`color: var(--color-primary)` on text in webhooks/[sub_id] filter chip and expand chevron (lines 169 and 212)** — On light theme (`[data-theme="light"]`), `--color-primary` resolves to `#00D9A3` on `rgb(0 217 163 / 0.10)` over `#F4F4F0`, yielding approximately 1.6:1 contrast (WCAG FAIL). The rule `--color-primary-text` exists precisely for this — it resolves to `#00805F` on light (4.95:1, AA pass) and `#00D9A3` on dark (10.5:1, AAA). Fix: change both `var(--color-primary)` references at lines 169 and 212 to `var(--color-primary-text)`. This is the one open visual-debt item carried from the verifier.

2. **`height: 6px` for webhooks list success-rate meter track (page.module.css line 152)** — The UI-SPEC Spacing Scale documents `8px` as the canonical exception for progress-bar track height ("cost-meter / seat-bar track height — Claude's Discretion"), and both members and mcp modules correctly use `8px`. The webhooks list deviates at `6px` without a rationale comment and without an explicit AC override. This creates visual inconsistency across the three progress-bar surfaces. Fix: change `height: 6px` to `height: 8px` and add the rationale comment matching the other two modules.

3. **Missing `@media (prefers-reduced-motion: reduce) { .meterFill { transition: none; } }` in members/page.module.css** — The members seat-bar uses `transition: width var(--duration-base) var(--ease-out)`. Global `tokens.css` zeroes `--duration-base` to `0ms` under `prefers-reduced-motion`, so the animation does collapse in practice. However both `mcp/page.module.css` and `webhooks/page.module.css` include an explicit local `@media (prefers-reduced-motion: reduce)` block for their meter fills — this is the pattern the codebase established. The members module is the outlier and lacks the explicit local declaration that peer modules use. Fix: add `@media (prefers-reduced-motion: reduce) { .meterFill { transition: none; } }` to members/page.module.css for parity and explicitness. Low risk; pure consistency improvement.

### Pillar 1: Token Compliance (3/4)

**Evidence:** `grep` on all 9 module.css files confirms zero inline hex (`#[0-9a-fA-F]{3,8}` returns no matches). Zero legacy fonts (Sora, Space Grotesk). Zero gradients active. Zero drop-shadows. Zero `transform: translate` hover-jiggle. Zero off-grid spacing literals except documented exceptions.

**PASS items:**

- All 9 module.css files: 0 inline hex (396 eliminated). Verified via grep.
- All spacing values are `var(--space-*)` tokens. Documented exceptions present for: `1px` borders, `2px` focus offsets, `max-width: 900px` / `max-width: 1040px` layout literals (Phase 203 convention, commented inline), `32px` avatar (icon sizing, not spacing, commented inline), `48px` logo (image dimension, commented inline), `width: 1px; height: 1px` screen-reader-only patterns.
- `height: 8px` for meter tracks in `members/page.module.css` and `mcp/page.module.css` — UI-SPEC Spacing Scale exception #4. Correctly cited with rationale comment.
- `var(--color-primary-text)` correctly used for link text in `webhooks/page.module.css` line 125 (URL cell) and `webhooks/[sub_id]/page.module.css` line 53 (breadcrumb anchor) with correct fallback syntax.

**FLAG items:**

- `webhooks/[sub_id]/page.module.css` line 169: `.filterChipGroup button[data-active='true'] { color: var(--color-primary); }` — raw `--color-primary` on text content. On dark theme this is AAA (10.5:1). On light theme, `#00D9A3` on `rgb(0 217 163 / 0.10)` over `#F4F4F0` yields approximately 1.6:1 — WCAG FAIL. The DESIGN.md "text-on-tinted-surface" rule (cited in CONTEXT D-09) and `tokens.css` comment block at line 175–186 both explicitly require `--color-primary-text` for text contexts. The `--color-primary-subtle` background does not provide sufficient surrounding contrast to rescue the text contrast ratio here — tinted surface contrast requires the darker mint alias. Severity: **FAIL on light theme / FLAG for Phase 213.4 fix.**

- `webhooks/[sub_id]/page.module.css` line 212: `.expandChevron[data-expanded='true'] { color: var(--color-primary); }` — same issue. The expand chevron in its active state uses raw `--color-primary` for its icon/text color. While chevrons are often icon-only elements, this is a `<button>` with `color` driving the rendered chevron character, making it a text-color context. On light theme, same ~1.6:1 contrast against the transparent card background (`--color-surface-raised` = `#FFFFFF` on light). Fix: `var(--color-primary-text)`. Severity: **FLAG (lower risk than line 169 — icon rather than label text, but same token rule).**

- Note: `webhooks/[sub_id]/page.module.css` line 121: `.tabButton[data-active='true'] { border-bottom-color: var(--color-primary); }` — this is a border decoration, not a text context. Raw `--color-primary` for borders and fills is correct per DESIGN.md. Not a flag.

**DEVIATION:**

- `webhooks/page.module.css` line 152: `height: 6px` for `.meterTrack`. The UI-SPEC documents `8px` as the canonical meter track height. Other meter tracks in this wave use `8px`. The `6px` value has no rationale comment and no explicit AC exemption. Minor visual debt; no token violation since track height is a sizing value, not a spacing value. Severity: **WARNING** (inconsistency with peer modules, not a token compliance failure).

### Pillar 4: Accessibility (3/4)

**Evidence:** 91 aria attributes found in non-story settings TSX files. `forced-colors` handled globally via `tokens.css` lines 218–231. `prefers-reduced-motion` handled globally via `tokens.css` lines 207–216. Bracketed-glyph state coding: 70 `[ok]/[warn]/[err]/[info]` references found across 9 TSX files.

**PASS items:**

- `role="alert"` used on urgency notices (danger warning above form; mcp at-cap banner). `role="status"` used on non-urgent notices and toast regions. Distinction is correct per ARIA live-region semantics.
- `role="dialog"` + `aria-labelledby` on confirm modals (danger, mcp revoke). Correct modal ARIA pattern.
- `aria-label` on icon-only buttons: `"Add MCP server"`, `"Refresh MCP usage"`, `"Copy to clipboard"` (mcp/page.tsx lines 182, 191, 252). Icon-only buttons are labelled — passes DESIGN.md "Iconography" rule.
- `role="meter"` + `aria-label="Daily MCP budget"` on the cost meter (mcp/page.tsx line 208, 212). Correct semantic for a progress meter.
- `aria-labelledby` on section landmarks: `danger-heading`, `domain-heading`, `mcp-usage-heading`, `mcp-sessions-heading`, `mcp-breakdown-heading` — landmark regions are labelled.
- Forced-colors: all `1px solid var(--color-border)` hairlines map to `CanvasText`; focus rings map to `Highlight`; mint-as-text maps to `LinkText` — all inherited from `tokens.css` forced-colors block. No module-level forced-colors redeclaration needed.
- Bracketed glyphs (`[ok]`, `[warn]`, `[err]`, `[info]`) are present on all notice bodies and badge labels. State is never color-only.
- `prefers-reduced-motion`: global `tokens.css` zeroes `--duration-fast`, `--duration-base`, `--duration-slow` to 0ms under the media query. Modules that use `var(--duration-*)` in transitions automatically collapse. The three modules with local `@media (prefers-reduced-motion: reduce)` blocks (mcp, webhooks list, webhooks detail) are belt-and-suspenders — they explicitly set `transition: none` on meter fills and interactive elements. This is commendable belt-and-suspenders practice for the three largest surfaces.

**FLAG items:**

- `members/page.module.css` line 41: `transition: width var(--duration-base) var(--ease-out)` on `.meterFill` — global token collapse covers this, but no explicit local `@media (prefers-reduced-motion: reduce)` block exists. Unlike `mcp` and `webhooks` which include the explicit local block, `members` relies entirely on the global token cascade. This is functionally correct but inconsistent with the pattern established by peer modules. Phase 213.4 fix recommendation: add the explicit local block for parity.

- `webhooks/[sub_id]/page.module.css` line 207: `.expandChevron { transition: transform var(--duration-fast) var(--ease-out); }` — this transition animates a `transform: rotate(90deg)` on expand. The `--duration-fast` token collapses to `0ms` under reduced-motion (global), so this is functionally correct. However the local `@media (prefers-reduced-motion: reduce)` block in this file (lines 368–370) only covers `.expandChevron` and `.tabButton` transition: none — which is correct. No issue.

- Focus trapping in modals and the `kernel-pulse` animation freeze on `.c-status-dot--live` under `prefers-reduced-motion` require browser-side verification (human UAT item 3, still open). These cannot be grepped. They are deferred, not flagged as failures.

### Pillar 5: Brand Voice (4/4)

**Evidence:** Zero matches for banned lexicon grep across all 11 TSX files. Zero exclamation points. Zero emoji (bracketed glyphs only: 70 instances found). Zero "Please" softeners.

**PASS items:**

All per-surface copy revisions match or exceed the UI-SPEC Copywriting Contract:

- Billing: `[warn] Payment issue. Resolve to continue using MarkOS.` — imperative, no softener. Invoice headers: `Category / Included / Used / Projected overage / Charge impact` — terse, data-register.
- Members: `[info] Invite pending. Resend or revoke if the recipient has not responded.` — system-state, explicit remediation.
- Sessions: `[ok] Current` via `c-badge--success` — bracketed glyph present on current-session badge.
- Domain: All 4 DNS state notices use correct bracketed glyphs and verb-first remediation copy.
- Danger: `Permanently deletes this workspace and all data. This action cannot be undone.` — explicit consequences, no softener. `[err] Deleting this workspace removes all members, data, API keys, and webhook subscriptions.` — bracketed glyph, enumerated consequences.
- MCP: `[warn] Cost approaching limit. Review tool usage to avoid service interruption.` — remediation explicit. `[err] Cost limit reached. MCP tools are paused. Adjust the budget to resume.` — system-state, remediation explicit.
- Plugins: `[warn] This plugin requires MarkOS v{N} or later.` — version explicit.
- Webhooks: `[ok] Enabled` / `[warn] Disabled` / `[err] Failing` badge copy consistent with UI-SPEC. Signing-rotation timeline notices match the T-7/T-1/T-0 copy specified in UI-SPEC.
- CLI parity: MCP surface uses `[ok] Connected` / `[warn] Disconnected` / `[err] Error` — consistent with `markos status` output per D-23.

**MINOR OBSERVATION (not a score deduction):**

`billing/page-shell.tsx` lines 146–150: the non-hold branch renders a `c-notice c-notice--warning` with copy `[warn] No active billing hold. {holdText}`. Using warning styling on a positive/neutral status (no billing hold = good) is semantically off. The UI-SPEC Billing copywriting table specifies the hold notice only appears when `isHold === true`. The current implementation renders a warning-styled notice in both branches — the non-hold branch would be better expressed as a plain `<p>` inside the card or omitted entirely, since it communicates no actionable state. This is a copy register mismatch, not a banned-lexicon or primitive violation. No AC explicitly requires a non-hold notice; this is implementation choice. Recommendation for Phase 213.4: remove the non-hold `c-notice c-notice--warning` block or replace with a neutral card paragraph.

# Phase 213.3: UI Canon Adoption Wave 3 — settings/* — Research

**Researched:** 2026-04-28
**Domain:** CSS Modules → DESIGN.md v1.1.0 token migration · 8 settings concept-domain surfaces · `.c-notice` primitive adoption · Storybook 8 CSF3 · grep-shape a11y harness
**Confidence:** HIGH (all source files read directly; UI-SPEC.md 49 ACs locked; CONTEXT.md 24 D-XX entries locked; existing test contracts inventoried; all 9 module.css files read; all 213.1 + 213.2 precedent files read)

## Summary

This is **tactical execution research** for Phase 213.3. UI-SPEC.md (49 ACs across 9 surfaces) is canonical for every visual decision; CONTEXT.md (D-01..D-24) is canonical for implementation strategy. This research investigates the exact mechanics of migrating 8 settings surfaces — billing, members, sessions, domain, danger, mcp, plugins, webhooks (list + `[sub_id]` detail) — from legacy teal-Sora-light-bg-drop-shadows to DESIGN.md v1.1.0 tokens, while preserving every wiring contract caught by Phase 200/200.1/201/202/203/204 test suites.

The redesign is pure className-rewrite + CSS-module-rewrite + copy-revision. DOM tree, prop interfaces, server/client boundaries, server actions, and fetch-wiring are preserved verbatim per CONTEXT D-05. The single structural **addition** across all surfaces is that every banner-shaped `<div>` that previously used local `.holdCard`, `.purgeBanner`, `.atCapBanner` classes now composes `<div class="c-notice c-notice--{state}">` directly — the `.c-notice` primitive landed in DESIGN.md v1.1.0 / `styles/components.css` v1.1.0 specifically for this wave.

The highest-complexity surfaces are `settings/mcp` (503 LOC CSS / 339 LOC TSX, 61 hex) and `settings/webhooks/[sub_id]` (762 LOC CSS / 813 LOC TSX, 118 hex). Both are candidates for selective sub-component extraction (MCPToolList, WebhookDeliveryTable) per CONTEXT D-15 carry. The decision is deferred to plan-phase based on JSX complexity per-plan.

**Primary recommendation:** Plan as 9 plans (8 surface + 1 cross-cutting), single wave, sequential execution per CONTEXT D-10. The cross-cutting plan (213.3-09) carries the `test/ui-a11y/213-3-settings-a11y.test.js` (≥30 tests) and the global `(pointer: coarse)` inheritance verification. Wave 0 gap is exclusively the new a11y test file — no new sub-component files are mandatory (sub-component extraction is Claude's Discretion, not a requirement).

### Pattern 1: Module CSS Rewrite Shape

Every `.module.css` rewrite follows this structure:
1. Delete all: font-family literals (Sora, Space Grotesk), all `#hex` values, all `box-shadow`, all `border-radius` literals that have a token equivalent, all per-element button/input/badge styles that primitives now cover.
2. Retain only: layout containers (`.page`, `.contentGrid`, `.cardHeaderRow`, etc.) with token-cited values; meter track/fill classes (no `.c-meter` primitive this wave); empty-state layout; table `th`/`td` token recipe.
3. Add: `.meterTrack` / `.meterFill` token recipe if surface has progress bar (members, mcp, webhooks list).

```css
/* Source: 213.1 chrome pattern + UI-SPEC.md Component Primitive Reuse table */

/* Layout-only classes — token-cited */
.page {
  padding: var(--space-md);
  background: var(--color-surface);
  display: flex;
  flex-direction: column;
  gap: var(--space-md);
}

/* Compose .c-card — override only layout extras */
.contentCard {
  /* inherits from .c-card: surface-raised bg, 1px border, radius-lg, space-md pad */
  max-width: 900px; /* surface-specific width — layout value, not spacing */
}

/* Progress bar recipe — no .c-meter primitive this wave */
.meterTrack {
  height: 8px; /* sizing value, documented exception per UI-SPEC */
  background: var(--color-surface-overlay);
  border-radius: var(--radius-full);
  overflow: hidden;
}
.meterFill {
  height: 100%;
  background: var(--color-success); /* or --warning / --error per value */
  transition: width var(--duration-base) var(--ease-out);
}

/* Table recipe — D-14; no .c-table primitive */
.table th {
  padding: var(--space-xs) var(--space-md);
  color: var(--color-on-surface-muted);
  font-size: var(--fs-label);
  font-weight: var(--fw-semibold);
  text-transform: uppercase;
  letter-spacing: var(--ls-label);
  border-bottom: 1px solid var(--color-border);
}
.table td {
  padding: var(--space-xs) var(--space-md);
  border-bottom: 1px solid var(--color-border);
  color: var(--color-on-surface);
  font-size: var(--fs-body-md);
  vertical-align: middle;
}
```

### Pattern 2: `.c-notice` Composition (THE key new pattern for 213.3)

This is the single biggest delta vs. Phases 213.1 + 213.2. The `.c-notice` primitive from DESIGN.md v1.1.0 covers every banner-shaped surface.

```tsx
{/* Source: styles/components.css .c-notice block + DESIGN.md "Color blindness" rule */}

{/* Warning banner — billing hold, domain rotation grace, MCP cost approaching */}
<div className="c-notice c-notice--warning">
  <strong>[warn]</strong>{" "}Cost approaching limit.{" "}
  Review tool usage to avoid service interruption.
</div>

{/* Error banner — danger zone consequence, MCP cost at limit, domain DNS fail, webhook DLQ */}
<div className="c-notice c-notice--error">
  <strong>[err]</strong>{" "}Deleting this workspace removes all members,
  data, API keys, and webhook subscriptions.
</div>

{/* Info banner — member invite pending, MCP key rotation, domain DNS pending, webhook signing T-7 */}
<div className="c-notice c-notice--info">
  <strong>[info]</strong>{" "}Secret rotation in progress.{" "}
  The previous secret is valid for 30 days (until {date}).{" "}
  Update your webhook consumer before then.
</div>

{/* Success banner — domain DNS verified */}
<div className="c-notice c-notice--success">
  <strong>[ok]</strong>{" "}Domain verified. Your custom domain is active.
</div>
```

**Critical:** The `.c-notice` primitive provides:
- `4px border-inline-start` accent (color per variant) — modules NEVER declare this directly
- `rgb({state-color} / 0.12)` background — modules NEVER declare the alpha formula
- `var(--color-{state})` text color — modules NEVER repeat this
- `a:hover` color shifts via `--color-{state}-hover` tokens (v1.1.0) — inherited

### Pattern 3: Class Rename in TSX (lockstep with module.css rewrite)

Every renamed or deleted class in `.module.css` must produce a corresponding edit in the `.tsx` consumer in the same commit. Zero broken `styles.X` references is the acceptance gate.

```tsx
{/* Before */}
<div className={styles.contentCard}>
  <h2 className={styles.heading}>MCP servers</h2>
  <p className={styles.subheading}>Configure connected AI agents.</p>
  <button className={styles.refreshButton}>Refresh</button>
</div>

{/* After — source: UI-SPEC.md Component Primitive Reuse + Copywriting Contract */}
<div className={`c-card ${styles.contentCard}`}>
  <h2>MCP servers</h2>                {/* globals.css <h2> inherits JetBrains Mono */}
  <p className="t-lead">Configure connected AI agents.</p>
  <button className="c-button c-button--secondary">Refresh</button>
</div>
```

### Anti-Patterns to Avoid

- **Local banner/alert classes:** Never `<div className={styles.holdCard}>` after redesign — use `.c-notice c-notice--warning`.
- **Raw hex in module.css:** Any `#0d9488`, `#0f766e`, `#ffffff`, `#f8fbfd` in a rewritten module is a Gate 4 failure.
- **Off-grid spacing:** `padding: 12px` → `var(--space-xs)` (8px) or `var(--space-sm)` (16px). `12px` is the most common off-grid value across all 8 modules.
- **`c-card--feature` in settings:** Reserved for hero auth panel and landing-page features. Settings panels are `c-card` (8px radius), never `c-card--feature` (32px radius).
- **Re-declaring `(pointer: coarse)` in module.css:** Global rule in `styles/components.css` covers all `.c-button` and `.c-nav-link`. Modules must not redeclare per X-6.
- **Sora/Space Grotesk in `font-family` override:** All 9 module.css files currently have these. Delete all; let globals.css inheritance handle headings.

## Common Pitfalls

### Pitfall 1: Forgetting `.c-notice` requires no class on the parent `.module.css`
**What goes wrong:** Developer adds a local `.banner` class in `page.module.css` and applies it alongside `.c-notice c-notice--warning` in JSX — the local class overrides the primitive's background or color.
**Why it happens:** Old muscle memory from pre-notice pattern (Phase 213.1 used a local `RotationGraceBanner.module.css` pattern).
**How to avoid:** Accept gates explicitly check `grep -nrE 'className=.*banner\|className=.*holdCard' "app/(markos)/settings/"` returns zero (AC X-4).
**Warning signs:** Local class in module.css with `background: rgb(...` or `border-left:` that duplicates primitive behavior.

### Pitfall 2: `box-shadow` on `.c-input` focus state
**What goes wrong:** Developer deletes the teal `outline: 2px solid #0d9488` on inputs but doesn't realize `.c-input:focus-visible` already applies the correct `box-shadow` recipe — so they add nothing, then add a custom outline, which conflicts.
**Why it happens:** The `.c-input` primitive uses `box-shadow` for focus (not `outline`) to stack correctly with the border.
**How to avoid:** Delete ALL local focus styles on inputs. `.c-input:focus-visible` in `styles/components.css` handles this.
**Warning signs:** Any `outline:` declaration in a settings `.module.css` file — Gate 3 grep will catch `outline.*#0d9488` but a new hex might slip through.

### Pitfall 3: `max-width` on `.contentCard` local override clashes with `.c-card` padding
**What goes wrong:** Developer writes `.contentCard { max-width: 900px; }` as a composition override but also duplicates padding from `.c-card`, causing double-padding.
**How to avoid:** The local `.contentCard` class adds ONLY layout-specific overrides (max-width, specific gap) — never re-declare padding, background, border, radius.
**Warning signs:** Local class with `background:`, `border:`, `border-radius:`, `padding:` that duplicates what `.c-card` provides.

### Pitfall 4: Sora/Space Grotesk in legacy `font-family` declarations not deleted
**What goes wrong:** The `font-family` stack is `'Sora', 'Space Grotesk', sans-serif` — developer renames or leaves the fallback fonts.
**Why it happens:** Forgetting there are two font family names in a comma-separated stack.
**How to avoid:** Gate 3 grep: `grep -E "Sora|Space Grotesk"` must return zero. Delete entire `font-family` declarations on `.heading`, `.subsectionHeading` classes and use `<h2>` / `<h4>` inheritance from globals.
**Warning signs:** Any `font-family:` declaration remaining in a settings module that isn't citing `var(--font-mono)` or `var(--font-sans)`.

### Pitfall 5: `12px` — the most common off-grid value in settings
**What goes wrong:** `gap: 12px`, `padding: 12px`, `padding-bottom: 12px` are ubiquitous across all 8 modules. Replacing with `8px` (`var(--space-xs)`) is usually correct but sometimes `16px` (`var(--space-sm)`) is the right choice.
**How to avoid:** Audit each `12px` usage: is it a `gap` between elements (likely `var(--space-xs)` = 8px) or a `padding` on a section element (likely `var(--space-sm)` = 16px)? The off-grid checker (`Gate 2`) catches any remaining literal `12px`.
**Warning signs:** `gap: 12px` (→ `var(--space-xs)`), `padding: 12px` (→ `var(--space-sm)`), `padding-bottom: 12px` on table headers (→ `var(--space-xs)`).

### Pitfall 6: Webhooks `[sub_id]` — largest file, easy to leave stragglers
**What goes wrong:** 762 LOC + 118 hex = highest risk of leaving isolated hex values or off-grid spacing deep in the file. Developer cleans the top sections but misses nested panel or delivery history sections.
**How to avoid:** Run Gate 1 (`grep -nE "#[0-9a-fA-F]{3,8}"`) line-by-line on the completed file and address every match before committing.
**Warning signs:** Any remaining hex in `.secretPanel`, `.dangerCard`, `.codeBlock`, `.deliveryTable` sub-sections.

### Pitfall 7: Test files that grep for legacy class names
**What goes wrong:** `test/mcp/consent-ui-a11y.test.js` and `test/mcp/mcp-settings-ui-a11y.test.js` may grep for legacy class names (`.contentCard`, `.heading`, `.refreshButton`) or legacy hex values (`#0d9488`). After redesign these assertions fail, breaking `npm test`.
**How to avoid:** Explicitly update these test files in the relevant plan (MCP plan 213.3-06 or cross-cutting 213.3-09) to reference new primitives/copy, mirroring the 213.2-05 precedent where `consent-ui-a11y.test.js` was updated.
**Warning signs:** `npm test` failing after a successful module.css rewrite.

## Open Questions (All Resolved)

### Q1: Is `.c-notice` the correct primitive for all 18 banner use cases in settings?

**Answer:** Yes — per CONTEXT D-09b and UI-SPEC Rule 7 (Component Reuse First). DESIGN.md v1.1.0 landed `.c-notice` specifically for this wave. The 4 variants cover all states: `--success` (domain verified), `--warning` (billing hold, domain rotation grace, MCP cost approaching, plugin compatibility, webhook signing T-1), `--error` (danger consequence, MCP cost at limit, domain DNS failed, webhook signing T-0, webhook DLQ), `--info` (member invite pending, MCP key rotation, domain DNS pending, webhook signing T-7). All 18 compositions are covered.

### Q2: Should `plugins/page-shell.module.css` `.brandCard` gradient be eliminated?

**Answer:** Yes — DESIGN.md v1.1.0 (via CLAUDE.md C-11): "No gradients." The `linear-gradient(135deg, #0f766e, #155e75)` is also using legacy teal values. Replacement: flat `background: var(--color-surface-raised); border: 1px solid var(--color-border);`. The brand context (tenant branding card) is served by the logo image and tenant name, not a gradient background.

### Q3: Should `webhooks/[sub_id]` be split into its own plan or combined with webhooks list?

**Answer:** Combined into Plan 213.3-08 per CONTEXT D-10: "webhooks — covers both list `page` and detail `[sub_id]/page`... May internally chunk by sub-region (header / table / detail-sidebar / modal) but commits as one plan." The plan is the largest in the wave (2450 LOC total).

### Q4: Should `test/mcp/mcp-settings-ui-a11y.test.js` be updated in Plan 213.3-06 or Plan 213.3-09?

**Answer:** Plan 213.3-06 (MCP plan) if the test file greps for MCP-specific legacy classes (likely). Plan 213.3-09 if only cross-cutting concerns remain. The executor should run `node --test test/mcp/mcp-settings-ui-a11y.test.js` after the MCP module.css rewrite — if it fails, update the test assertions in the same plan (mirrors 213.2-05-T3 `consent-ui-a11y.test.js` patch precedent).

### Q5: Does the `settings/domain` surface use the same `c-notice` patterns as the chrome-level `RotationGraceBanner`?

**Answer:** They are separate surfaces. The chrome-level `RotationGraceBanner.module.css` (Phase 213.1, already token-compliant) stays unchanged. The `settings/domain/page.tsx` is the deeper drill-down surface and gets its own `.c-notice c-notice--warning` for the rotation-grace state (per CONTEXT "Specific Ideas" section). They intentionally overlap in copy because both surfaces communicate the same system state — this is correct per CONTEXT D-23 (Phase 204 CLI parity: conceptual model consistency).

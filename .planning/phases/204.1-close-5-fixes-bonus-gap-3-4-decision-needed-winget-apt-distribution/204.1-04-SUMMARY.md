# Phase 204.1 Plan 04 Summary

Plan `204.1-04` closes the two decision-gated review gaps: the env preview mask now follows the last-4 contract, and the status subscription panel now surfaces the pricing placeholder when the API explicitly says pricing is still unresolved. This slice also picks up the deferred `status` spinner wiring from Plan 02.

## What changed

- Added `supabase/migrations/78_markos_cli_env_mask_last4.sql`.
  - Replaces the `set_env_entry(...)` preview logic so secrets with length `>= 8` store `'*'.repeat(len - 4) + last4`.
  - Fully masks shorter secrets.
  - Documents the deliberate no-backfill behavior for pre-78 rows.
- Extended `lib/markos/cli/env.cjs`.
  - Added `maskSecret(value)` as the JS parity twin for the SQL preview algorithm.
- Updated `bin/commands/status.cjs`.
  - Reuses `PRICING_PLACEHOLDER_SENTINEL` from `lib/markos/cli/runs.cjs`.
  - Shows `{{MARKOS_PRICING_ENGINE_PENDING}}` when `pricing_recommendation_id` is explicitly `null`.
  - Keeps the existing tier rendering when the field is non-null, and also preserves the old tier rendering for envelopes that omit the field entirely so current callers do not regress before the API rolls out the field consistently.
  - Wraps the status fetch path in `createSpinner({ label: 'fetching status' })`, which also covers `--watch` refresh cycles because they reuse the same fetch helper.
- Added `test/cli/204.1/env-mask-last4.test.js` and `test/cli/204.1/status-pricing-placeholder.test.js`.
  - Covers last-4 masking, family-prefix protection, short-secret masking, placeholder rendering, JSON-mode non-injection, and the deferred status spinner behavior.

## Repo-reality notes

- The plan packet referenced `upsert_markos_cli_tenant_env_v1(...)`, but this repository's real RPC surface is still `set_env_entry(...)` from migration 76. Migration 78 updates the real function instead of introducing a second, unused name.
- The placeholder branch is intentionally null-specific. Existing `status` fixtures in this repo do not always include `pricing_recommendation_id`, and treating every falsy/omitted value as pending would have regressed the current one-shot status output before the server contract is updated everywhere.

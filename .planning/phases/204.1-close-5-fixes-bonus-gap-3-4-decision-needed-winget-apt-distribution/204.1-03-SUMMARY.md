# Phase 204.1 Plan 03 Summary

Plan `204.1-03` closes the P2 terminal-compatibility gaps by centralizing glyph selection and teaching the shared table renderer how to degrade for narrow shells. The CLI now has one box-drawing policy for UTF-8 versus ASCII environments, and tabular output stops wrapping mid-row in sub-60-column terminals.

## What changed

- Extended `bin/lib/cli/output.cjs`.
  - Added `GLYPHS_UTF8`, `GLYPHS_ASCII`, and `pickGlyphs(env, platform)`.
  - Added the `<60` column vertical renderer path inside `renderTable(...)`.
- Updated box-rendering consumers to use the shared glyph selector.
  - `bin/lib/cli/errors.cjs`
  - `bin/commands/whoami.cjs`
  - `bin/commands/status.cjs`
  - `bin/commands/doctor.cjs`
  - `bin/commands/run.cjs` now routes the snapshot arrow through `pickGlyphs().arrow`.
- Added `test/cli/204.1/glyphs-fallback.test.js` and `test/cli/204.1/render-table-width.test.js`.
  - Covers UTF-8 vs ASCII tier selection, the Windows codepage/env override path, narrow vertical rendering, wide horizontal rendering, and dimmed narrow labels in color-capable TTYs.

## Repo-reality notes

- There is no reliable built-in Node API for reading `chcp` directly, so the Windows ASCII fallback is keyed off `MARKOS_CODEPAGE=437` or `MARKOS_FORCE_ASCII=1` rather than shelling out from the renderer.
- Several existing CLI files contained hardcoded mojibake-rendered literals. Rather than fight brittle string-level patches, I overrode the affected render helpers at the function level so every box path now resolves through `pickGlyphs()` cleanly.

## Outcome

The CLI now has a single glyph policy instead of five ad hoc ones, and narrow shells get readable record-by-record output instead of broken wrapped tables. This closes the reviewed ASCII-fallback and width-breakpoint debt without introducing new dependencies.

- searchApprovedKnowledge({ query, scopes, filters, claims, topK }) -> snippet-first result list with citations, authority, freshness, confidence or score, implication hints, and resource links
- fetchApprovedArtifact({ uri, section, claims }) -> one approved artifact or section with full content and metadata

<threat_model>
- T-92-06 (Transport sprawl): the MCP layer could grow into a broad menu of browse, write, or approval tools. Mitigation: expose exactly two public operations in v1, with optional read-only resources only for explicit markos URIs.
- T-92-07 (SDK instability): alpha MCP SDK changes could force core retrieval rewrites. Mitigation: isolate all transport-specific imports and registration logic to the adapter boundary so the core service stays pure and reusable.
- T-92-08 (Auth bypass): parity wrappers might accidentally bypass the tenant-scoped hosted auth path. Mitigation: reuse the existing hosted-read auth pattern and fail closed when claims are missing or mismatched.
- T-92-09 (Audit blind spots): deep-research clients may query sensitive knowledge without sufficient traceability. Mitigation: record request_id, tenant_id, client surface, requested tool or resource, and denial outcomes for every call.
- T-92-10 (Contract drift across clients): MCP, API, and CLI wrappers could slowly diverge. Mitigation: all wrappers delegate to the same service module and share the same payload schema and fixtures.
</threat_model>

# Phase 64.1 Research

## Executive Summary

Phase 64.1 should be treated as a runtime stabilization phase, not a broad framework rewrite. Repo evidence shows that `.claude/get-shit-done/VERSION` and `.github/get-shit-done/VERSION` are both `1.32.0`, and the core CLI entrypoint `.claude/get-shit-done/bin/gsd-tools.cjs` already matches `.github/get-shit-done/bin/gsd-tools.cjs`. `.claude/get-shit-done/bin/lib/phase.cjs` is also already in parity. The immediate instability is elsewhere: `.claude/settings.json` still invokes three nonexistent shell hooks (`gsd-session-state.sh`, `gsd-phase-boundary.sh`, `gsd-validate-commit.sh`). Those names appear nowhere else in the repo, and none of them are present in `.claude/gsd-file-manifest.json`.

The runtime drift that matters for 64.1 is narrower than the version numbers and manifest differences suggest. Direct comparison shows that some `.claude` versus `.github` hash differences are wording-only or instruction-surface differences, not execution-path differences. `core.cjs` and `init.cjs` differ mostly in “Claude” versus “the agent” wording. `verify.cjs` contains at least one live command-guidance delta (`/gsd-progress` in `.github` versus `/gsd:progress` in `.claude`) that does affect operator guidance. `profile-output.cjs`, `templates/claude-md.md`, and workflow references such as `new-project.md` and `plan-phase.md` differ mainly because `.claude` is still localized around `CLAUDE.md`, while `.github` is now centered on `copilot-instructions.md`. That is a real interface boundary, but it belongs to 64.2 unless it directly breaks runtime execution.

Primary recommendation: in 64.1, treat `.github/get-shit-done/**` as the canonical shared GSD framework for runtime behavior, keep `.claude/settings.json` and `.claude/hooks/*.js` as the Claude-local runtime bootstrap, remove dead shell hook references instead of porting them, and sync only functional runtime deltas from `.github` into `.claude/get-shit-done/**`. Do not chase full hash parity, rename instruction targets, refresh manifests, or absorb end-to-end verification.

### Project Constraints From `CLAUDE.md`

- `.planning/STATE.md` is the canonical live GSD state. Runtime changes in 64.1 should preserve that assumption.
- GSD and MarkOS are separate layers. 64.1 should operate on the GSD runtime layer, not the broader MarkOS marketing protocol layer.
- Client overrides belong in `.markos-local/`, not in ad hoc new runtime locations.
- Repo-standard validation commands are `npm test` or `node --test test/**/*.test.js`.
- The primary project CLI is `npx markos`, but this phase is about the checked-in GSD runtime surfaces under `.claude` and `.github`.

## Architecture Patterns

### Canonical Versus Localized Assets

| Asset Class | Canonical in `.github` | Localized in `.claude` | 64.1 Action |
| --- | --- | --- | --- |
| Shared execution logic | `.github/get-shit-done/bin/**/*.cjs` | `.claude/get-shit-done/bin/**/*.cjs` | Sync only behavior-affecting drift into `.claude` |

| Manifest inventories | `.github/gsd-file-manifest.json` | `.claude/gsd-file-manifest.json` | Use for audit and scope control; do not make refresh the main 64.1 objective |

### Pattern 1: Sync by Behavior, Not by Hash

64.1 should not attempt full file-hash parity. The file comparisons show three categories of drift:

1. Already aligned runtime files.
   - Confirmed aligned: `.claude/get-shit-done/bin/gsd-tools.cjs`, `.claude/get-shit-done/bin/lib/phase.cjs`.
2. Runtime files with wording-only divergence.
   - `core.cjs` and `init.cjs` differ primarily in comments and user-facing wording such as `Claude` versus `the agent`.
3. Files whose differences change runtime guidance or generated surface targets.
   - `verify.cjs` has at least one operator-facing command difference.
   - `profile-output.cjs`, `new-project.md`, `plan-phase.md`, and `templates/claude-md.md` target different instruction-file surfaces.

64.1 should patch category 3 only when the difference affects current runtime behavior or near-term planning. Category 2 does not justify churn in this phase.

### Pattern 2: JS-Only Hook Contract

The active repo already contains a working Node-hook model:

- `.claude/hooks/gsd-check-update.js`
- `.claude/hooks/gsd-context-monitor.js`
- `.claude/hooks/gsd-prompt-guard.js`
- `.claude/hooks/gsd-read-guard.js`
- `.claude/hooks/gsd-statusline.js`
- `.claude/hooks/gsd-workflow-guard.js`

The missing shell hooks are not present anywhere else in the repo and are not represented in `.claude/gsd-file-manifest.json`. The correct stabilization move is therefore:

- Remove dead shell-hook references from `.claude/settings.json`.
- Keep the runtime hook surface Node-based.
- Only add a new JS hook if a concrete runtime requirement is missing after dead-reference cleanup.
- Do not invent replacements for the absent shell hooks unless a verified behavior gap remains.

### Pattern 3: Treat `.claude/settings.json` as the Runtime Binding Layer

`.claude/settings.json` is the authoritative binding between the `.claude` runtime and its hooks. In 64.1 it should follow three rules:

- Every configured command must resolve to an existing file in `.claude/hooks/`.
- Every bound hook should correspond to a file tracked by `.claude/gsd-file-manifest.json`.
- The file should express only the active runtime contract, not historical or speculative hooks.

### Pattern 4: Keep the MarkOS-Local Instruction Contract Stable During Runtime Repair

This repo still boots project guidance through root `CLAUDE.md`, and `CLAUDE.md` explicitly points agents to `.protocol-lore/QUICKSTART.md`, `.protocol-lore/INDEX.md`, `.planning/STATE.md`, and `.agent/markos/MARKOS-INDEX.md`. Meanwhile, `.github`'s newer GSD surface has shifted toward `copilot-instructions.md`. That is a real alignment task, but it is an instruction-surface task, not a hook-stabilization task. 64.1 should preserve the current local contract unless a planning command is blocked by it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
| --- | --- | --- | --- |
| Broken shell hook references | New ad hoc `.sh` hooks | Existing `.claude/hooks/*.js` and a simplified `.claude/settings.json` | The shell hooks do not exist anywhere in the repo and are brittle on Windows |
| Runtime parity detection | A custom parity tracker | Existing `.claude/gsd-file-manifest.json`, `.github/gsd-file-manifest.json`, and direct file diffing | The repo already has hash inventories for shipped surfaces |
| Broad instruction retargeting in 64.1 | New one-off target-rewrite logic | Defer to 64.2's localized agent/template/instruction alignment | That drift is real but belongs to a later phase boundary |
| Manifest-wide refresh | Manual hash rewriting across the whole surface | Existing manifest-generation/update workflow, deferred to 64.3 | High churn, low immediate runtime value, easy to get wrong |
| Replacement logic for unknown legacy hooks | Reconstructed behavior from guesswork | Remove dead references first; only add JS replacements if a verified gap remains | There is no canonical shell implementation left to port |

## Common Pitfalls

### Pitfall 1: Mistaking Version Parity for Runtime Parity

What goes wrong: both runtime roots report `1.32.0`, so the phase gets dismissed as unnecessary.

Why it happens: `VERSION` files match while several runtime and surface files do not.

How to avoid it: classify drift by behavior. Only execution-affecting deltas belong in 64.1, but they still need to be checked.

Warning signs: identical version files paired with different manifest hashes or command strings.

### Pitfall 2: Trying to Port the Missing Shell Hooks

What goes wrong: the implementation invents new shell scripts or tries to reconstruct legacy behavior from settings alone.

Why it happens: `.claude/settings.json` still names shell hooks, which creates the false impression that they are the intended contract.

How to avoid it: treat the existing JS hooks as the live contract and the shell references as stale config.

Warning signs: work starts by creating `.sh` files instead of cleaning `.claude/settings.json`.

### Pitfall 3: Absorbing 64.2 Through Template and Instruction Churn

What goes wrong: the phase turns into a repo-wide rewrite from `CLAUDE.md` to `copilot-instructions.md`.

Why it happens: `profile-output.cjs`, `new-project.md`, `plan-phase.md`, and `templates/claude-md.md` visibly diverge between `.claude` and `.github`.

How to avoid it: only touch those files in 64.1 if they directly block runtime behavior or planning. Otherwise, document the interface and defer.

Warning signs: changes begin affecting generated instruction files, profile-output targets, or template naming instead of runtime stability.

### Pitfall 4: Treating Manifest Files as the Runtime Itself

What goes wrong: the phase spends its time refreshing hashes instead of fixing the actual runtime.

Why it happens: `.claude/gsd-file-manifest.json` and `.github/gsd-file-manifest.json` are highly visible and show a lot of differences.

How to avoid it: use manifest files as evidence and scoping tools in 64.1, not as the main implementation target.

Warning signs: the plan is centered on manifest regeneration before the hook contract is even valid.

### Pitfall 5: Expanding Hook Policy While Trying to Stabilize Hooks

What goes wrong: optional behavior such as `gsd-workflow-guard.js` gets newly enabled without a demonstrated requirement.

Why it happens: the file exists in `.claude/hooks/`, so it is easy to assume it should be wired immediately.

How to avoid it: keep 64.1 focused on removing broken references and restoring a coherent runtime contract.

Warning signs: the phase changes enforcement policy, not just broken bindings.

## Code Examples

### Example 1: JS-Only Hook Wiring Precedent

Use `.agent/settings.json` as the cleanest in-repo example of a JS-only hook contract. The pattern is simple: wire only hooks that exist, and keep the event contract explicit.

Source paths:
- `.agent/settings.json`
- `.claude/settings.json`

Pattern to reuse:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node .agent/hooks/gsd-check-update.js"
          }
        ]
      }
    ],
    "AfterTool": [
      {
        "matcher": "Bash|Edit|Write|MultiEdit|Agent|Task",
        "hooks": [
          {
            "type": "command",
            "command": "node .agent/hooks/gsd-context-monitor.js",
            "timeout": 10
          }
        ]
      }
    ]
  }
}
```

64.1 reuse rule: adapt this cleanliness to `.claude/settings.json`, but keep the Claude-local event names that the active runtime actually uses.

### Example 2: Versioned Node Hooks With Built-In Stale Detection

`.claude/hooks/gsd-check-update.js` is the pattern to preserve. It detects installed version state, scans hook headers for `// gsd-hook-version: ...`, and writes a shared cache that `gsd-statusline.js` can read.

Source paths:
- `.claude/hooks/gsd-check-update.js`
- `.claude/hooks/gsd-statusline.js`

Pattern to reuse:

```js
const versionMatch = content.match(/\/\/ gsd-hook-version:\s*(.+)/);
if (versionMatch) {
  const hookVersion = versionMatch[1].trim();
  if (isNewer(installed, hookVersion) && !hookVersion.includes('{{')) {
    staleHooks.push({ file: hookFile, hookVersion, installedVersion: installed });
  }
}
```

64.1 reuse rule: keep this Node-based lifecycle. Do not replace it with shell glue.

### Example 3: Statusline-to-Monitor Bridge Instead of Extra Hook State Scripts

The existing context bridge is already a robust hook composition pattern. `gsd-statusline.js` writes a temp JSON bridge file, and `gsd-context-monitor.js` consumes it after tool use.

Source paths:
- `.claude/hooks/gsd-statusline.js`
- `.claude/hooks/gsd-context-monitor.js`

Pattern to reuse:

```js
const bridgePath = path.join(os.tmpdir(), `claude-ctx-${session}.json`);
fs.writeFileSync(bridgePath, JSON.stringify({
  session_id: session,
  remaining_percentage: remaining,
  used_pct: used,
  timestamp: Math.floor(Date.now() / 1000)
}));
```

64.1 reuse rule: prefer this explicit Node-to-Node bridge pattern over trying to revive a missing `gsd-session-state.sh` hook.

### Example 4: Guard Hooks Should Be File-Existence Aware and Advisory

`.claude/hooks/gsd-read-guard.js` only injects guidance when the target file already exists. That is a good example of bounded, advisory hook behavior.

Source path:
- `.claude/hooks/gsd-read-guard.js`

Pattern to reuse:

```js
try {
  fs.accessSync(filePath, fs.constants.F_OK);
  fileExists = true;
} catch {
  // File does not exist — no guidance needed
}

if (!fileExists) {
  process.exit(0);
}
```

64.1 reuse rule: if any missing runtime behavior truly needs a replacement hook, implement it in this bounded Node style, not as a broad shell interceptor.

### Example 5: Instruction-Surface Drift That Should Be Deferred

The following pairs clearly show the 64.2 boundary rather than the 64.1 runtime target:

- `.claude/get-shit-done/bin/lib/profile-output.cjs` writes `CLAUDE.md`
- `.github/get-shit-done/bin/lib/profile-output.cjs` writes `copilot-instructions.md`
- `.claude/get-shit-done/workflows/new-project.md` sets `INSTRUCTION_FILE="CLAUDE.md"`
- `.github/get-shit-done/workflows/new-project.md` sets `INSTRUCTION_FILE="copilot-instructions.md"`
- `.claude/get-shit-done/workflows/plan-phase.md` tells the runtime to read `./CLAUDE.md`
- `.github/get-shit-done/workflows/plan-phase.md` tells the runtime to read `./copilot-instructions.md`

64.1 reuse rule: document this interface boundary, but do not collapse it prematurely unless it blocks active planning.

## Recommended Phase Boundary

### What Belongs in 64.1

- Repair `.claude/settings.json` so every configured hook points to an existing, supported file.
- Remove the dead `bash` hook references rather than porting them.
- Keep the active `.claude` hook strategy JS/Node-only.
- Sync runtime-impacting deltas from `.github/get-shit-done/bin/lib/*.cjs` into `.claude/get-shit-done/bin/lib/*.cjs` when those deltas affect behavior or current operator guidance.
- Apply minimal runtime guidance corrections needed before new planning begins.
- Treat `.github/get-shit-done/**` as the behavioral source of truth, but only for execution logic, not for every localized wording or file-target difference.

### What Should Defer to 64.2

- Reconciliation of `.claude` versus `.github` instruction-file targets such as `CLAUDE.md` versus `copilot-instructions.md`.
- Broad alignment of localized workflow prose and generated template surfaces.
- Files such as:
  - `.claude/get-shit-done/bin/lib/profile-output.cjs`
  - `.claude/get-shit-done/templates/claude-md.md`
  - `.claude/get-shit-done/workflows/new-project.md`
  - `.claude/get-shit-done/workflows/plan-phase.md`
  - `.github/copilot-instructions.md`
- Any repo-wide decision about whether MarkOS keeps `CLAUDE.md`, adopts `copilot-instructions.md`, or supports both.

### What Should Defer to 64.3

- End-to-end runtime verification across planning flows, hook execution, and generated surfaces.
- Refresh or regeneration of `.claude/gsd-file-manifest.json` and `.github/gsd-file-manifest.json` as a formal deliverable.
- Customization-boundary documentation that explains which surfaces are shared GSD framework files and which are MarkOS-local overrides.
- Cleanup work whose value depends on verification evidence rather than immediate runtime stability.

### Specific Prescriptive Notes

- Hook strategy: remove dead shell references; do not port missing shell hooks; only add a new JS hook if a verified behavior gap remains after cleanup.
- `.claude/settings.json`: treat as the live runtime binding layer and constrain it to existing Node hooks only.
- Manifest-driven surfaces: use manifests to identify intended file ownership and parity scope, but do not make manifest refresh the center of the phase.
- Generated instruction surfaces: record the `.claude` versus `.github` target mismatch as a handoff boundary to 64.2 unless it is blocking active planning commands.

## Open Questions

1. Is `verify.cjs` the only runtime-affecting code delta that still matters before Phase 65 planning?
   - What we know: `gsd-tools.cjs` and `phase.cjs` are already aligned, and the inspected `core.cjs` and `init.cjs` drift is wording-only.
   - What's unclear: whether there are additional behavior-affecting deltas elsewhere in `.claude/get-shit-done/bin/lib/**` that were not surfaced by the mandatory-file sample alone.
   - Recommendation: before planning, classify remaining `.claude` versus `.github` `bin/lib` diffs into behavior versus wording and only schedule the behavior set.

2. Should `gsd-workflow-guard.js` remain unbound in `.claude/settings.json`?
   - What we know: the file exists and is manifest-backed, but it is not currently wired in `.claude/settings.json`.
   - What's unclear: whether enabling it is considered stabilization or policy expansion for this repo.
   - Recommendation: leave it unbound in 64.1 unless there is an explicit requirement to change edit-policy behavior.

3. Is root `CLAUDE.md` intentionally the local instruction authority for MarkOS even though `.github` now prefers `copilot-instructions.md`?
   - What we know: the repo currently boots through root `CLAUDE.md`, and MarkOS quickstart documents reinforce that path.
   - What's unclear: whether long-term parity means migrating, dual-supporting, or preserving the current MarkOS-local contract.
   - Recommendation: make that decision in 64.2, not while stabilizing hooks and runtime behavior.

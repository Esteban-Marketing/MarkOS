# Phase 64.1: GSD Runtime Parity and Hook Stabilization - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Align the active `.claude` GSD runtime with the newer `.github` 1.32.0 surface by repairing the live hook contract, syncing the full executable runtime subtree under `.claude/get-shit-done/bin/**`, and correcting user-facing workflow or help command guidance where it affects active repo behavior. This phase does not rewrite generated instruction targets, collapse `CLAUDE.md` into `copilot-instructions.md`, refresh manifests, or perform end-to-end verification.

</domain>

<decisions>
## Implementation Decisions

Hook Contract decisions:
- **D-01:** Remove the dead shell-hook references from `.claude/settings.json` and keep the active `.claude` runtime JS-only.
- **D-02:** Do not recreate the missing `.sh` hooks.
- **D-03:** Add a new JS hook only if cleanup reveals a verified runtime gap that the existing `.claude/hooks/*.js` set does not cover.

Runtime parity scope decisions:
- **D-04:** 64.1 should sync the full executable runtime subtree under `.claude/get-shit-done/bin/**` against `.github/get-shit-done/bin/**`.
- **D-05:** 64.1 should also update user-facing workflow and help command guidance where it affects active runtime behavior or operator instructions.
- **D-06:** 64.1 must stop before template rewrites, generated instruction-target changes, or broader `.claude/get-shit-done/**` surface alignment that belongs in 64.2.

Command guidance decisions:
- **D-07:** Where 64.1 touches user-facing runtime guidance for this repo, standardize on the VS Code command form `/gsd:...`.
- **D-08:** Command-guidance fixes are part of runtime parity in this repo and should be included when they affect active workflows or user correction paths.

Optional hook activation decisions:
- **D-09:** Leave currently unbound optional hooks, including `gsd-workflow-guard.js`, unbound in 64.1 unless a verified runtime need is discovered during implementation.
- **D-10:** 64.1 is a stabilization phase, not a hook-policy expansion phase.

the agent's Discretion:

- Which touched workflow or help markdown files require command-guidance correction once the runtime-impacting set is identified.
- Whether any verified missing runtime behavior truly warrants a new JS hook after dead-reference cleanup.

</decisions>

## Existing Code Insights

### Reusable Assets
- `.agent/settings.json`: Demonstrates the cleanest in-repo JS-only hook binding pattern.
- `.claude/hooks/gsd-check-update.js`: Existing version-aware hook that already manages stale-hook signaling.
- `.claude/hooks/gsd-statusline.js` plus `.claude/hooks/gsd-context-monitor.js`: Existing Node-to-Node bridge pattern for session context, making a replacement shell state hook unnecessary.
- `.claude/hooks/gsd-read-guard.js`: Existing bounded, file-aware guard behavior that can serve as the model if a new JS hook becomes necessary.

### Established Patterns
- `.claude/settings.json` is the active binding layer for Claude-local runtime hooks and should only point at existing executable files.
- `.github/get-shit-done/bin/**` is the canonical shared runtime behavior source; `.claude/get-shit-done/bin/**` is the localized runtime copy.
- Root `CLAUDE.md` remains the local instruction surface in this repo; template or target-surface rewrites are intentionally deferred.

### Integration Points
- Hook cleanup lands in `.claude/settings.json` and must stay consistent with `.claude/hooks/*.js`.
- Runtime parity work connects `.github/get-shit-done/bin/**` to `.claude/get-shit-done/bin/**`.
- Command-guidance parity touches the active `.claude` runtime and any user-facing workflow/help docs that shape how this repo invokes GSD commands.

</code_context>

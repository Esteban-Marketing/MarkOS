# Phase 64.1: GSD Runtime Parity and Hook Stabilization - Validation Strategy

**Purpose:** Define the Nyquist validation contract for Phase 64.1 so execution can prove the active .claude runtime uses a valid JS-only hook contract, the executable .claude bin subtree is back in parity with the canonical .github runtime where 64.1 allows it, and the touched workflow or help guidance shows the repo-standard /gsd:... commands without crossing the 64.2 or 64.3 boundary.

## Phase Goal

Align the active .claude GSD runtime with the newer .github 1.32.0 surface by removing dead shell-hook references, synchronizing executable runtime behavior under .claude/get-shit-done/bin, and correcting active workflow or help command guidance where it affects live repo behavior, while preserving CLAUDE.md as the local instruction contract and deferring broader surface alignment, manifest refresh, and end-to-end verification.

## Deliverables and Verification Methods

| Deliverable | Verification Method | Responsible | Dimension |
| --- | --- | --- | --- |
| .claude/settings.json contains only live JS hook bindings | Node config validation script plus file inspection | AI | Hook Contract |
| No missing .sh hooks remain in the active runtime path | Shell-reference scan and existing-hook existence check | AI | Bootstrap Stability |
| Optional hooks such as gsd-workflow-guard.js remain unbound unless a verified runtime need appears | Binding scan over .claude/settings.json | AI | Scope Control |
| .claude/get-shit-done/bin/lib/core.cjs and init.cjs match the canonical .github runtime | Exact file parity check | AI | Runtime Parity |
| Touched operator-facing bin files preserve only approved repo-local command-guidance deltas | Targeted string assertions over `init.cjs` and `verify.cjs` plus 64.2-boundary checks | AI | Local Contract Fidelity |
| Touched workflow and help files show /gsd:... instead of legacy /gsd-... command guidance | Full touched-surface syntax scan and spot review | AI / Human | Operator Guidance |
| The repaired `.claude` runtime still executes coherently after hook and bin repair | Repo-local smoke command through `.claude/get-shit-done/bin/gsd-tools.cjs` | AI / Human | Runtime Smoke |

## Human Testing Items

1. Open .claude/settings.json and confirm the active hook contract contains only node-based .claude/hooks/*.js commands.
2. Run the localized help or progress workflow in the active repo runtime and confirm the next-step guidance shown to operators uses /gsd:... syntax.
3. Trigger a runtime-health or progress correction path and confirm the repo points the operator to /gsd:progress rather than a legacy /gsd-progress command.
4. Run one repo-local smoke invocation through `.claude/get-shit-done/bin/gsd-tools.cjs` and confirm it resolves Phase 64.1 context without shell-hook or command-guidance regressions.

## Exit Conditions

Phase 64.1 is only complete when:

1. GSD-ALIGN-01 is satisfied by a JS-only .claude hook contract with no dead shell-hook references.
2. GSD-ALIGN-02 is satisfied by executable runtime parity across the touched .claude bin files, with only the approved repo-local /gsd:... deltas preserved.
3. GSD-ALIGN-03 is satisfied by keeping CLAUDE.md as the local instruction contract and avoiding template rewrites, manifest refresh, optional hook activation, and end-to-end verification work reserved for 64.2 or 64.3.
4. Every touched workflow or help surface in 64.1 presents /gsd:... command guidance to the operator.
5. The phase can be executed and verified without inventing missing shell hooks, silently widening repo-local hook policy, or pulling instruction-target work forward from 64.2.

# Phase 207 — UI Design Contract (no-UI-scope)

> **Phase 207 ships zero UI surfaces.** This document is the explicit no-surface declaration
> for the AgentRun v2 Orchestration Substrate phase. It exists so that gsd-ui-checker,
> gsd-planner, and gsd-ui-auditor have an unambiguous contract: there is no `app/`, no
> `components/`, no `*.stories.tsx`, no `page.tsx`, no `layout.tsx`, no `*.module.css`, and
> no `*.css` modified or created in any of the six plans (207-01 through 207-06).
>
> What Phase 207 *does* ship is **orchestration doctrine** (`.planning/orchestration/*.md`),
> a **contract lock** (`207-01-CONTRACT-LOCK.md`), **typed runtime modules**
> (`lib/markos/contracts/agent-run-v2.ts`, `lib/markos/agents/*.ts`), **billing-bridge
> modules** (`lib/markos/billing/agent-run-cost-bridge.ts`, `usage-normalizer.ts`),
> **compatibility adapters** (`lib/markos/crm/playbook-run-adapter.ts`,
> `lib/markos/mcp/run-emitter.cjs`, `onboarding/backend/agents/run-engine.cjs` shim),
> **Node API handlers** (`api/tenant/runs/**`, `api/tenant/chains/**`,
> `api/tenant/agents/**`, `api/tenant/run-handoffs/**`, `api/internal/scheduler/tick.js`),
> **migrations** (`supabase/migrations/101..106*.sql` + rollbacks), **F-contracts**
> (`contracts/F-106..F-111-*.yaml`), **CI assertion scripts**
> (`scripts/orchestration/*.mjs`), and **tests** (`test/agentrun/phase-207/**/*.test.js`,
> `test/contracts/agent-run-v2.test.js`, `test/integration/{crm,mcp}-to-agentrun-v2.test.js`,
> `test/migrations/10[1-6]_*.test.js`). None of those files compose, import, or render any
> visual primitive from `styles/components.css` or any token from `app/tokens.css`. The
> Node API handlers under `api/tenant/**` and `api/internal/**` are server-side route
> modules (CommonJS or ESM, no JSX), not Next.js App Router pages.
>
> However, **every downstream phase (208–228) that consumes a Phase 207 orchestration
> contract WILL eventually need a UI surface** — run viewer dashboards, run-event SSE
> consumers, DLQ inspectors, approval-token viewers, cost-posture previews, chain
> visualizers, scheduler queue boards, registry rollout consoles, handoff inboxes, and
> retry-policy editors. This UI-SPEC therefore also serves as a forward-looking
> inheritance map so future UI-SPECs can cite their lineage back to the orchestration
> doctrine defined here.
>
> Authority chain: DESIGN.md v1.1.0 → 213.x adoption-wave decisions (D-08, D-09, D-09b,
> D-13, D-14, D-15) → 206-UI-SPEC (compliance doctrine origin for approval / data-export
> mutation classes) → this document. Generated by gsd-ui-researcher. Status: draft
> (checker upgrades to approved once the no-UI declaration is verified).

## Copywriting Contract

**End-user / surface copy: not applicable.** Phase 207 emits no copy to any human surface.

**Doctrine prose copy** in `.planning/orchestration/*.md` and `207-01-CONTRACT-LOCK.md`
is governed by:
- The CLAUDE.md banned lexicon (`synergy`, `leverage`, `empower`, `unlock`, `transform`,
  `revolutionize`, `supercharge`, `holistic`, `seamless`, `cutting-edge`, `innovative`,
  `game-changer`, `next-generation`, `world-class`, `best-in-class`, `reimagine`,
  `disrupt`, `just` as softener — and no exclamation points).
- Engineering-readability: short sentences, concrete contract field names, named owning
  plans. No marketing voice. No hedging language ("we strive to", "best-effort"). Every
  contract row asserts a specific field, literal, or test-gate procedure.
- Pricing-placeholder rule: `{{MARKOS_PRICING_ENGINE_PENDING}}` appears verbatim in
  `run-cost-billing-bridge.md` and as an accepted JSONB value on
  `markos_agent_runs.pricing_engine_context` per CLAUDE.md Pricing Engine Canon.
- Forbidden architecture-lock strings (rejected by `check-run-architecture-lock.mjs` per
  acceptance criterion `207-01-00.AC.3`): `second run substrate`,
  `client-owned task persistence`, `approval bypass`, `unmetered run close`.
- Task-ownership boundary copy: every reference to the human task system in 207 doctrine
  prose must explicitly defer ownership to Phase 208 (per HIGH#2 review concern). The
  contract-lock §9 includes the verbatim phrase `Phase 208`. No 207 doctrine claims to
  "own the task system" — only "expose handoff records and optional `task_ref` links".

**Node API error envelopes:** When `api/tenant/runs/**`, `api/tenant/chains/**`,
`api/tenant/run-handoffs/**`, `api/tenant/agents/**`, or `api/internal/scheduler/tick.js`
return error responses, the JSON envelope `{ error_code, error_message }` follows these
rules:
- `error_code` is a stable kebab-case identifier (e.g. `run-not-found`,
  `approval-required`, `concurrency-cap-exceeded`, `dlq-poison-immediate`,
  `task-handoff-missing`, `cost-actualizer-divergence`, `registry-not-runnable`,
  `chain-edge-deadlock`).
- `error_message` is a short engineering-grade sentence. No marketing voice. No
  hedging. No banned-lexicon terms. No exclamation points. No emoji. Bracketed-glyph
  prefixes (`[err]`, `[warn]`) reserved for downstream CLI / UI surfaces — JSON envelopes
  themselves carry only the structured code + message fields.
- These rules apply to API envelopes shipped by 207 because consumers (CLI / future UI)
  may surface `error_message` verbatim in their UX, and surfacing a banned-lexicon
  string would violate the downstream surface's DESIGN.md Pillar 1 audit.

These are doctrine-prose rules and API-envelope rules, not UI copywriting rules — included
here only because the orchestrator's downstream inheritance question requires future UI
surfaces consuming this orchestration substrate to honor the same lexicon discipline when
they render any run state, error message, or contract field name into a UI label, banner,
or modal body.

## Destructive Actions

**Not applicable in Phase 207.** No surface ships, so no confirm-modals are authored.

The Node API handlers DO ship destructive endpoints (`POST /api/tenant/runs/{run_id}/retry`,
`POST /api/tenant/runs/{run_id}/pause`, run-cancel via state transition to `canceled`,
DLQ-move via state transition to `dlq`), but these endpoints emit JSON envelopes only —
they do not author confirm-modals. Confirm-modal authoring is the responsibility of
downstream UI consumers per the carry-forward rules below.

When future phases ship surfaces that mutate any 207-substrate state (e.g. retry a DLQ

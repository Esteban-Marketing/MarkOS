# Plan 207-03 — Priority, Concurrency, Retry, Timeout, Pause, Cancel, DLQ Policy

## Objective

Ship the durable scheduler, priority/concurrency enforcer, retry-policy engine, DLQ table, and chain DAG execution. Complete migrations 102 (chains + edges) and 103 (failures + DLQ). Complete F-107 + F-108 YAMLs.

## Current code path this migrates

- `lib/markos/mcp/pipeline.cjs` — existing middleware pipeline has cost/approval/timeout per-tool-call. This plan **preserves** it; the new scheduler calls MCP pipeline under the hood for MCP-triggered runs but does NOT bypass its middleware.
- `onboarding/backend/agents/run-engine.cjs` — stays; does NOT own scheduling. New `lib/markos/agents/scheduler.ts` owns queue+priority+concurrency.
- `lib/markos/webhooks/breaker.cjs` (Phase 203 sliding-window breaker) — pattern reused for per-tenant concurrency tracking.

## must_haves

- Migration 102 complete: `markos_agent_chains` + `markos_agent_chain_edges` with RLS, tenant FK, indexes.
- Migration 103 complete: `markos_agent_failures` + `markos_dlq_agent_runs` view; RLS; purge retention index.
- Priority P0-P4 enforced with PRIORITY_CONCURRENCY_DEFAULTS from CONTRACT-LOCK §6.
- Retry-with-jitter implemented per RETRY_POLICY from CONTRACT-LOCK §6.
- DLQ created on `poison` (1 hit) or `quota` exhaustion (10 hits).
- Pause + cancel endpoints work; paused runs can resume.
- Chain engine executes `sequential`, `parallel_fan_out`, `all_of`, `any_of`, `conditional` edges.
- Starvation protection: P4 run blocked > `starvation_floor_seconds` is promoted by 1 tier.

  <action>
    Create **`lib/markos/agents/retry-policy.ts`** exporting:
    - `computeBackoffMs(retryCount: number): number` returning `min(BACKOFF_CAP_MS, BACKOFF_BASE_MS * 2^retryCount) * (1 + random jitter within BACKOFF_JITTER_FRACTION)`.
    - `classifyError(error: unknown): AgentFailureClass` — given an error code/message, returns one of the 7 classes. Mapping: timeout/network/5xx → `transient`; 4xx/validation → `user_input`; policy-block → `policy`; rate-limit/quota → `quota`; upstream-dep → `upstream`; parser-crash/infinite-loop → `poison`; cancel-token → `cancel`.
    - `shouldRetry(failure: { failure_class: AgentFailureClass; retry_count: number }): { retry: boolean; dlq: boolean; backoff_ms: number }`:
      - `poison` → `{ retry: false, dlq: true, backoff_ms: 0 }` (DLQ after 1 hit per CONTRACT-LOCK).
      - `transient|upstream` → retry if `retry_count < MAX_RETRIES_DEFAULT`; else `dlq: true`.
      - `quota` → retry with backoff if `retry_count < DLQ_AFTER_QUOTA_EXHAUSTION_HITS` (10); else DLQ.
      - `policy|user_input|cancel` → `{ retry: false, dlq: false, backoff_ms: 0 }` (task-only).

    Create **`lib/markos/agents/dlq.ts`** exporting:
    - `moveToDlq({ runId, tenantId, failureClass, errorCode, errorMessage, retryCount })` — inserts `markos_agent_failures` row with `dlq_reason = <failure_class>:<error_code>`, transitions run to `dlq`, creates a recovery task (`task_kind: 'recovery'`, status `open`) and writes `agent_run_dlq_moved` event. Returns `{ failure_id, recovery_task_id }`.
    - `scheduleRetry({ runId, tenantId, retryCount, backoffMs })` — sets run `retry_count = retryCount + 1`, `retry_after = now() + backoff_ms`, transitions run to `retry_wait`, emits `agent_run_retry_scheduled` event.
  </action>

  <action>
    Create **`lib/markos/agents/scheduler.ts`** exporting:

    - `pickNextRunnable(tenantId: string, limit: number = 10): Promise<AgentRun[]>`:
      - Returns up to `limit` runs in state `queued` OR `retry_wait` where `retry_after <= now()`, ordered by priority (P0 first) then `created_at asc`.
      - Enforces `PRIORITY_CONCURRENCY_DEFAULTS[priority].tenant_max_concurrent` — subtract the count of runs currently in `executing|awaiting_approval|context_loaded|accepted` for this tenant at this priority.
      - Applies starvation promotion: any `queued` run with `created_at < now() - starvation_floor_seconds` for its priority gets promoted by one tier (P4→P3, P3→P2, ...) by updating `priority` column and emitting `agent_run_transitioned` with reason `starvation_promoted`.
      - Emergency override: P0 ignores `tenant_max_concurrent` when `emergency: true`.

    - `claimRun({ runId, workerId }): Promise<{ claimed: boolean }>` — atomic UPDATE to transition `queued → accepted` with `actor_id = workerId` using `WHERE state = 'queued'` optimistic lock; returns `claimed: false` if another worker beat it.

    - `applyTimeoutPolicy({ runId, tier }): Promise<void>` — sets a timer per `RETRY_POLICY.TIMEOUT_MS[tier]`; on expiry, calls `classifyError('timeout')` → `shouldRetry` → either `scheduleRetry` or `moveToDlq`.

    - `tickScheduler(tenantId?)` — main loop: loads eligible runs, dispatches via injected `runnerFn(run) → Promise<{ok, error?}>`, applies timeout + retry policy to each result. Designed to be called by `/api/internal/scheduler/tick` (cron) OR a background worker.

    Create **`api/internal/scheduler/tick.js`** — POST-only, protected by `MARKOS_INTERNAL_SCHEDULER_TOKEN` header. Calls `tickScheduler()`. Returns `{ processed, completed, failed, dlq_moved }`.

    Create **`lib/markos/agents/chain-engine.ts`** exporting:
    - `createChain({ tenantId, actorId, chainName, strategy, priority })` → inserts `markos_agent_chains`, returns chain.
    - `addChainEdge({ chainId, fromRunId, toRunId, dependencyType, fanInRequirement, failurePolicy })` → inserts `markos_agent_chain_edges`.
    - `evaluateDownstream(runId)`: on a run entering terminal state (`completed`, `failed`, `canceled`, `rejected`), finds outgoing edges and applies dependency semantics:
      - `sequential` → single downstream becomes `queued` when upstream is `completed`.
      - `all_of` → downstream becomes `queued` when **all** upstreams satisfying `fan_in_requirement` are `completed`.
      - `any_of` → downstream becomes `queued` when any upstream is `completed`.
      - `conditional` → evaluate `payload.condition` JS expression against upstream output; queue if true.
    - Apply `failure_policy` per edge when upstream is non-completed terminal:
      - `abort_chain` → transition chain + all non-terminal runs in chain to `canceled`.
      - `skip_downstream` → downstream run is set to `canceled` with reason `upstream_failed`.
      - `retry_branch` → call `scheduleRetry` on downstream once, max 1 retry per branch.
      - `continue` → downstream proceeds regardless.
    - Emits `agent_run_chain_edge_satisfied` event on each satisfaction.
    - `pauseChain(chainId)` + `cancelChain(chainId)` — transition chain state + cascade pause/cancel to all non-terminal runs.

    Create **`api/tenant/chains/index.js`** (POST create chain) and **`api/tenant/chains/[chain_id]/index.js`** (GET chain + runs; POST `/pause` + `/cancel` via body `action` field OR query param).
  </action>

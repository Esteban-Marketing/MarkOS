# Plan 207-04 — ApprovalGate and Task Handoff Contracts

## Objective

Ship the durable `markos_agent_tasks` table (Task handoff record), ApprovalGate contract, side-effect idempotency guard integration, and 4 HTTP endpoints. Complete migration 104 + F-109 YAML. Connect every `awaiting_approval`, `rejected`, `failed` (non-retry), and `dlq` run to a Task so no run disappears silently per RUN-05.

## Current code path this migrates

- `app/(markos)/operations/tasks/task-types.ts` + `task-machine.ts` + `task-store.tsx` — existing Phase 208 task UI. **Becomes a read-only consumer of `markos_agent_tasks`**, NOT a sibling store. The pure reducer in task-machine.ts remains valid (it reduces task events); but the source of truth migrates from Zustand/client state to Supabase `markos_agent_tasks`.
- `supabase/migrations/53_agent_run_lifecycle.sql` — `markos_agent_side_effects` table stays; idempotency UNIQUE (tenant_id, run_id, step_key, effect_hash) preserved. Migration 104 adds approval linkage.
- `lib/markos/mcp/approval.cjs` — existing MCP approval-token system stays intact; this plan's `lib/markos/agents/approval-gate.ts` is a higher-level run-level gate that may (optionally) call into mcp/approval.cjs for tool-level approvals.

## must_haves

- Migration 104 complete: `markos_agent_tasks` with RLS + FK + 4 task_kinds + 5 statuses; `markos_agent_runs.task_id` FK added.
- `markos_agent_side_effects` ALTER: add `approval_id TEXT REFERENCES markos_agent_tasks(task_id) ON DELETE SET NULL` and `target_system TEXT NOT NULL DEFAULT 'unknown'`.
- Every `awaiting_approval`, `rejected`, `failed (non-retry)`, `dlq`, `paused` run transition automatically creates/links a `markos_agent_tasks` row.
- ApprovalGate enforces: external side effects blocked unless `task_id.status = 'approved'` OR run has `approval_policy.mode = 'autonomous'`.
- 4 endpoints: list, get, approve, reject.
- F-109 YAML complete.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Mismatched closing tag on `<output>` element**
- **Found during:** Task 3 (build-storybook)
- **Issue:** Replaced `<div role="status">` with `<output>` but left `</div>` closing tag — esbuild parse error.
- **Fix:** Changed `</div>` to `</output>` to match opening tag.
- **Files modified:** app/(markos)/operations/page.tsx
- **Commit:** 9222c6e (amended into Task 3 commit)

**2. [Rule 2 - Banned Lexicon] Replaced "unlock" with "advance" in panel copy**
- **Found during:** Task 2 (automated banned-lexicon scan)
- **Issue:** "only unlock later steps" — `unlock` is in CLAUDE.md banned lexicon.
- **Fix:** Replaced with "only advance to later steps".
- **Files modified:** app/(markos)/operations/page.tsx
- **Commit:** 7243fc8

**3. [Rule 2 - A11y] `<output>` over `<div role="status">`**
- **Found during:** Task 2 (S6819 IDE diagnostic)
- **Issue:** `role="status"` on `<div>` triggers S6819 — use semantic `<output>` element instead per WCAG 2.1 AA.
- **Fix:** Changed element to `<output>` — matches DESIGN.md accessibility requirement.
- **Files modified:** app/(markos)/operations/page.tsx
- **Commit:** 7243fc8

**4. [Rule 2 - Missing prop] Added `canAccess` to AuthContext interface**
- **Found during:** Task 3 (Storybook story arg shape)
- **Issue:** UI-SPEC O-8 stories pass `canAccess: true/false` via `authOverride` args; original `AuthContext` interface lacked the field, so prop would be silently dropped.
- **Fix:** Added `canAccess?: boolean` to `AuthContext` interface; updated `canAccess` derivation to use `??` (nullish coalescing) — preserves operator role-gate fail-closed logic for the runtime path.
- **Files modified:** app/(markos)/operations/page.tsx
- **Commit:** 7243fc8

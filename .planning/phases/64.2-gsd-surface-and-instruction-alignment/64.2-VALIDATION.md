# Phase 64.2: GSD Surface and Instruction Alignment - Validation Strategy

**Purpose:** Define the validation contract for Phase 64.2 so execution can prove the repo now has intentional dual-root instruction support, clear shared-versus-local ownership across `.github` and `.claude`, and focused automated coverage for those policy seams without absorbing the 64.3 manifest-refresh or end-to-end verification scope.

## Phase Goal

Reconcile MarkOS-localized `.claude` agents, templates, and instruction surfaces with the refreshed `.github` framework so Copilot, Claude, and shared GSD workflows stay behaviorally aligned through one shared dual-root instruction policy, a real root `copilot-instructions.md` artifact, preserved root `CLAUDE.md` locality, and explicit ownership boundaries.

## Deliverables and Verification Methods

| Deliverable | Verification Method | Responsible | Dimension |
| --- | --- | --- | --- |
| Shared dual-root output policy exists and both generation trees consume it | Node module assertions over shared and localized policy helpers | AI | Policy Foundation |
| Root `copilot-instructions.md` exists as a managed project artifact and remains distinct from `.github/copilot-instructions.md` | File existence and content scan | AI | Root Contract |
| `GSD-ALIGN-04` and `GSD-ALIGN-05` exist in `.planning/REQUIREMENTS.md` without broader requirement invention | Registry text scan | AI | Requirements Traceability |
| Canonical `.github` workflows and core agents consistently point at root `./copilot-instructions.md` | Static file scans and targeted unit tests | AI | Shared Ownership |
| Localized `.claude` workflows and core agents consistently keep root `./CLAUDE.md` as the local contract | Static file scans and targeted unit tests | AI | Local Runtime Contract |
| Shared and localized execute-plan, UI, update, and profile-skill surfaces stay aligned to their respective root contracts | Static file scans and targeted unit tests | AI | Execution and Drift Guard |
| `.claude/skills/**` stay pointed at localized `.claude/get-shit-done/**` paths | Recursive skill-manifest scan | AI | Dispatch Boundary |
| 64.2 gains focused automated proof without bundling manifest refresh or end-to-end verification | New node:test suite plus explicit no-64.3 assertions in validation review | AI / Human | Scope Control |

## Human Testing Items

1. Open root `CLAUDE.md` and root `copilot-instructions.md` and confirm they read like sibling project instruction contracts rather than one file replacing the other.
2. Spot-check shared execute-plan or UI workflow or agent wording and confirm it references root `./copilot-instructions.md`, not hidden `.github/copilot-instructions.md`, when talking about project instructions.
3. Spot-check localized execute-plan or UI workflow or agent wording and confirm it still references root `./CLAUDE.md` as the localized runtime contract.
4. Confirm the new automated coverage stays fast and narrow and does not attempt manifest refresh, cross-machine checks, or end-to-end runtime verification that belongs in 64.3.

## Exit Conditions

Phase 64.2 is only complete when:

1. GSD-ALIGN-04 is satisfied by a real dual-root instruction contract that preserves root `CLAUDE.md`, adds or manages root `copilot-instructions.md`, and does not break localized `.claude` runtime expectations.
2. GSD-ALIGN-05 is satisfied by explicit ownership across generation, templates, workflows, and core agents: `.github/**` stays canonical shared content and `.claude/**` stays the localized runtime or dispatch layer.
3. `.claude/skills/**` still point at localized `.claude/get-shit-done/**` paths, proving the phase did not absorb runtime migration or customization-boundary closure.
4. The only requirements-registry change is the narrow addition of `GSD-ALIGN-04` and `GSD-ALIGN-05` using the locked roadmap or research wording.
5. The new automated coverage proves the policy seams touched by 64.2, including execute-plan, UI, update, and profile-skill surfaces, without bundling manifest refresh, generated-file rehashing, or full end-to-end verification reserved for 64.3.

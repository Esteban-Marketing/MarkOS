## Summary

Phase 64.2 is not a version-parity problem. It is a contract split. Repo evidence shows that the active MarkOS-local runtime still boots from root `CLAUDE.md`: `.claude/settings.json` places `CLAUDE.md` first in `sourceOfTruthOrder`, `.claude/agents/*.md` instruct subagents to read `./CLAUDE.md`, `.claude/get-shit-done/workflows/new-project.md` targets `CLAUDE.md` for non-Codex runtimes, and the root `bin/install.cjs` still treats `CLAUDE.md` as a first-class AI doc. At the same time, the refreshed shared GSD framework under `.github` has moved its non-Codex contract to root `copilot-instructions.md`: `.github/agents/*.agent.md`, `.github/get-shit-done/workflows/new-project.md`, `.github/get-shit-done/workflows/plan-phase.md`, and `.github/get-shit-done/bin/lib/profile-output.cjs` all expect that file.

That means 64.2 should align instruction policy and generation, not attempt a one-way migration. Preserving only `CLAUDE.md` leaves the `.github` planner/checker/executor surfaces without project-instruction compliance because they explicitly read `./copilot-instructions.md`. Migrating only to `copilot-instructions.md` would break the active `.claude` runtime, local boot docs, and localized skill routing. The stable answer is dual support: keep root `CLAUDE.md` as the live Claude-local contract, introduce and manage root `copilot-instructions.md` as the Copilot/shared-framework project contract, and ensure both files are produced from the same managed content policy so they do not drift.

Phase 64.2 should also formalize ownership. `.github/get-shit-done/**`, `.github/skills/**`, and `.github/agents/**` should be treated as the canonical shared GSD framework surfaces. `.claude/settings.json`, `.claude/hooks/*.js`, `.claude/skills/**`, and `.claude/get-shit-done/**` should remain the localized Claude runtime and dispatch layer, because local skills are hard-wired to `@~/.claude/get-shit-done/workflows/...` and the repo boot flow still centers on `CLAUDE.md`. 64.2 should align policy, targets, and generation points across these surfaces. 64.3 should verify the result end-to-end, refresh manifests or generated artifacts that need re-hashing, and document the final customization boundary.

**Primary recommendation:** Support both root `CLAUDE.md` and root `copilot-instructions.md`, generated from one shared policy, while keeping `.github/**` canonical for shared framework content and `.claude/**` localized for active Claude runtime behavior.

## Architecture Patterns

### Recommended Project Structure

```text
repo-root/
|- CLAUDE.md                     # Active Claude-local project instructions
|- copilot-instructions.md      # Project-root Copilot/shared-framework instructions
|- .github/
|  |- copilot-instructions.md   # Global/shared Copilot guidance artifact
|  |- agents/                   # Canonical shared agent definitions
|  |- skills/                   # Canonical shared skill manifests
|  `- get-shit-done/            # Canonical shared workflows, templates, and tooling
`- .claude/
   |- settings.json             # Local Claude boot order and hook/runtime bindings
   |- hooks/                    # Local Claude runtime hooks
   |- agents/                   # Localized Claude agent definitions
   |- skills/                   # Localized skill entrypoints and dispatch
   `- get-shit-done/            # Localized workflow/tooling copy used by .claude surfaces
```

### Pattern 1: Dual-Root Instruction Contract

**What:** Treat root `CLAUDE.md` and root `copilot-instructions.md` as sibling project instruction surfaces with the same managed content policy.

**When to use:** Any change to profile generation, new-project scaffolding, plan/research agent instructions, or template naming.

**Example:**

```js
// Localized Claude generator target
targetPath = path.join(cwd, 'CLAUDE.md');

// Shared Copilot generator target
targetPath = path.join(cwd, 'copilot-instructions.md');
```

**Prescriptive rule:** Do not make one of these files the hand-maintained source and the other a best-effort copy. Align the generator/template policy so both are produced deliberately.

### Pattern 2: Canonical Shared Content, Localized Dispatch

**What:** `.github/**` is the canonical shared framework surface. `.claude/**` is the localized runtime and dispatch layer that can wrap or mirror canonical content where the Claude runtime needs local paths, settings, or filenames.

**When to use:** Aligning workflows, skills, templates, and agent instructions without flattening the runtime model.

**Example:**

```md
<execution_context>
@~/.claude/get-shit-done/workflows/plan-phase.md
@~/.claude/get-shit-done/references/ui-brand.md
</execution_context>
```

**Prescriptive rule:** Preserve `.claude` path locality where skill manifests and runtime settings already require it. Align behavior and content, not by deleting the localized layer.

### Pattern 3: Separate Project Instructions from Global Instructions

**What:** Root `copilot-instructions.md` is a project file. `.github/copilot-instructions.md` is a global/shared-user artifact in the current `.github` profile flow. They are not interchangeable.

**When to use:** Aligning profile generation, planner compliance checks, and installer/update messaging.

**Example:**

```text
Project copilot-instructions.md  -> ./copilot-instructions.md
Global copilot-instructions.md   -> .github/copilot-instructions.md
```

**Prescriptive rule:** 64.2 must not treat the existing `.github/copilot-instructions.md` as satisfying `.github` agent requirements for root `./copilot-instructions.md`.

### Pattern 4: Policy Alignment in 64.2, Verification in 64.3

**What:** 64.2 should align file targets, generator ownership, template intent, and agent/workflow language. 64.3 should verify the aligned system end-to-end, refresh manifests, and document the final customization boundary.

**When to use:** Scoping tasks so 64.2 does not become a second stabilization-plus-verification phase.

### Anti-Patterns to Avoid

- **Repo-wide `CLAUDE.md` to `copilot-instructions.md` replacement:** breaks localized Claude runtime assumptions and confuses project versus global artifacts.
- **Treating `.github/copilot-instructions.md` as the project instruction file:** `.github` agents read root `./copilot-instructions.md`, not the hidden `.github` file.
- **Collapsing `.claude` into `.github` during this phase:** local skills and settings are intentionally wired to `.claude` paths today.
- **Manual duplication of template content:** the drift root is generator/template ownership, so hand-edits make the problem worse.

## Common Pitfalls

### Pitfall 1: Treating the Problem as a Filename Migration

**What goes wrong:** The phase tries to replace every `CLAUDE.md` reference with `copilot-instructions.md`.

**Why it happens:** The `.github` surface is newer and consistently targets `copilot-instructions.md`.

**How to avoid:** Model the repo as a dual-contract system: localized `.claude` still depends on `CLAUDE.md`, while shared `.github` expects root `copilot-instructions.md`.

**Warning signs:** Proposed changes touch `.claude/settings.json`, `.protocol-lore/*`, and `.claude/skills/**` as if those files were legacy leftovers.

### Pitfall 2: Confusing `.github/copilot-instructions.md` with `./copilot-instructions.md`

**What goes wrong:** Work assumes the hidden `.github` file already satisfies `.github` agent requirements.

**Why it happens:** Both files have the same name but different scopes.

**How to avoid:** Keep project-root and global/shared instruction artifacts distinct in code, docs, and tests.

**Warning signs:** Plans say "the file already exists" without specifying whether they mean the root file or the hidden `.github` file.

### Pitfall 3: Repointing Localized Skills During Policy Alignment

**What goes wrong:** `.claude/skills/**` are repointed to `.github/get-shit-done/**` during the same phase.

**Why it happens:** The shared framework looks like the cleaner source of truth.

**How to avoid:** Keep localized dispatch localized. Align content and targets first; consider deeper dispatch refactors only if a later phase explicitly scopes them.

**Warning signs:** Skill manifests start changing from `@~/.claude/get-shit-done/...` to `.github/...` paths.

### Pitfall 4: Pulling 64.3 Work Forward

**What goes wrong:** 64.2 expands into manifest refresh, broad generated-artifact rewrites, and end-to-end verification.

**Why it happens:** Instruction-target changes naturally touch generated files and tests.

**How to avoid:** Restrict 64.2 to policy and ownership alignment. Leave proof, refresh, and closure work to 64.3.

**Warning signs:** The plan spends more time on verification workflows than on ownership decisions.

### Pitfall 5: Letting the Missing Requirement Registry Create Scope Guesswork

**What goes wrong:** The planner invents new requirement text because `GSD-ALIGN-04` and `GSD-ALIGN-05` are absent from `.planning/REQUIREMENTS.md`.

**Why it happens:** The IDs exist in `ROADMAP.md`, so the missing registry looks like a planning vacuum.

**How to avoid:** Use the roadmap goal as the interim requirement source and treat the registry update as a parallel documentation cleanup.

**Warning signs:** New requirement language appears that is broader than the 64.2 roadmap goal.

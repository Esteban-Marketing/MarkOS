# Plan 205-01 - Pricing Contracts, Schema, and ID Allocation

**Phase:** 205 - Pricing Engine Foundation + Billing Readiness  
**Status:** Planned

## Objective

Create the contract and database foundation for the Pricing Engine without reusing occupied incoming F-IDs or stale billing assumptions.

## Scope

- Reserve fresh contract IDs for pricing intelligence, competitive matrix, cost model, recommendations, price tests, watch list, alerts, and billing handoff.
- Reserve fresh migration IDs and table names.
- Define tenant-isolated tables for `markos_pricing_knowledge`, `markos_cost_models`, `markos_pricing_recommendations`, `markos_price_tests`, `markos_pricing_watch_list`, and `markos_pricing_alerts`.
- Add append-only history/diff fields for pricing intelligence.
- Add RLS policy plan and audit events for pricing-sensitive objects.
- Add forward-compatible references for future SaaS Suite plan, subscription, churn, support, product usage, MRR, discount, and save-offer signals without implementing SaaS Suite tables in Phase 205.

## Acceptance Criteria

- No incoming F-90 through F-100 collision is reused.
- Every pricing table has tenant isolation, history strategy, and audit posture.
- Contract schemas separate evidence from inference.
- Contract schemas can later link SaaS Suite signals to PricingRecommendation records without bypassing approval.
- Migration and rollback strategy is documented before code execution.

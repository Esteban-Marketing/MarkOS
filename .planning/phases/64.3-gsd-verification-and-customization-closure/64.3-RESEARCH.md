## Summary

Phase 64.3 is a closure phase, not a design phase. The dual-root policy is already implemented and verified at the policy seam. Repo evidence shows that the remaining work is exactly what 64.2 deferred: manifest freshness, generated-artifact freshness, end-to-end proof that the aligned `.github` and `.claude` surfaces behave coherently as one system, and durable documentation of who owns what. The most important new finding is that the checked-in GSD file manifests are already stale relative to the 64.2 changes. Direct hash comparison on 2026-04-06 showed mismatches for both shared and localized touched files, including `.github/get-shit-done/bin/lib/profile-output.cjs`, `.github/get-shit-done/workflows/new-project.md`, `.claude/get-shit-done/bin/lib/profile-output.cjs`, and `.claude/get-shit-done/workflows/new-project.md`, while both manifest timestamps remain on 2026-04-05.

The repo already has most of the verification pieces needed for 64.3. `test/gsd/dual-root-generation.test.js` proves deliberate two-root output behavior. `test/gsd/instruction-surface-alignment.test.js` proves the shared/localized wording split. `bin/install.cjs` already treats `CLAUDE.md` and `copilot-instructions.md` as first-class root artifacts. `bin/update.cjs` and `bin/cli-runtime.cjs` already show the established hash-refresh pattern for `.markos-install-manifest.json`. What is missing is the closure glue: one deterministic way to refresh `.github/gsd-file-manifest.json` and `.claude/gsd-file-manifest.json`, one end-to-end test or command path that exercises both generators and the real root artifacts together, and one codebase-level document that records the boundary between shared `.github`, localized `.claude`, and client-owned `.markos-local` surfaces.

The correct 64.3 strategy is therefore prescriptive and narrow. Refresh the stale manifests with deterministic hashes, regenerate or verify the root instruction artifacts through the existing `generate-claude-md` flows, run a focused end-to-end verification bundle that spans both roots and both generator lanes, and then document the ownership boundary in `.planning/codebase/` so Phase 65 and later phases do not revisit the same policy question.

**Primary recommendation:** Treat 64.3 as a three-deliverable closure phase: `1)` refresh stale GSD manifests and any generated root instruction artifacts, `2)` add an end-to-end verification gate that spans `.github` and `.claude`, and `3)` promote the shared-versus-localized-versus-client ownership boundary into durable codebase documentation.

- Localized `.claude` workflows, agents, and profile skill still point at root `./CLAUDE.md`.

**Layer 2: Generator behavior proof**

- Localized generator creates or refreshes project-root `CLAUDE.md`.
- Hidden `.github/copilot-instructions.md` remains distinct.

**Layer 3: Manifest freshness proof**
- Every tracked file hash in `.github/gsd-file-manifest.json` matches the live `.github/get-shit-done/**` tree.
- Every tracked file hash in `.claude/gsd-file-manifest.json` matches the live `.claude/get-shit-done/**` tree.

**Prescriptive rule:** 64.3 is not complete until all three layers pass in sequence.

### Pattern 2: Refresh by Deterministic Hashing, Not Manual JSON Edits

**What:** Recompute manifest hashes from disk and write the manifest in one pass.

**When to use:** Refreshing `.github/gsd-file-manifest.json` and `.claude/gsd-file-manifest.json`.

**Example:**

```js
// Pattern reference: bin/cli-runtime.cjs buildFileHashes()
function buildFileHashes(dir, baseDir = dir, hashes = {}) {
  for (const entry of fs.readdirSync(dir, { withFileTypes: true })) {
    const full = path.join(dir, entry.name);
    if (entry.isDirectory()) {
      buildFileHashes(full, baseDir, hashes);
    } else {
      const rel = path.relative(baseDir, full).replace(/\\/g, '/');
      hashes[rel] = crypto.createHash('sha256').update(fs.readFileSync(full)).digest('hex');
    }
  }
  return hashes;
}
```

**Prescriptive rule:** Reuse the repo's established hashing pattern and JSON shape. Do not hand-edit hash strings.

## Common Pitfalls

### Pitfall 1: Assuming the GSD Manifests Are Fine Because Tests Are Green

**What goes wrong:** The phase stops after the 64.2 test bundle passes.

**Why it happens:** The policy tests are meaningful and already green.

**How to avoid:** Add an explicit manifest-freshness check to the 64.3 gate.

**Warning signs:** `.github/gsd-file-manifest.json` and `.claude/gsd-file-manifest.json` timestamps predate the last instruction-surface edits.

### Pitfall 2: Manually Rewriting Hashes

**What goes wrong:** A person copies new hash strings into JSON by hand.

**Why it happens:** No checked-in GSD manifest generator is currently discoverable.

**How to avoid:** Create or use one deterministic refresh helper in-repo and record that command in 64.3 docs.

**Warning signs:** The plan says "update hashes" without naming a command or helper.

### Pitfall 3: Confusing Project-Root and Hidden Copilot Artifacts Again

**What goes wrong:** 64.3 verification proves only that `.github/copilot-instructions.md` exists.

**Why it happens:** Both files share the same name.

**How to avoid:** Every end-to-end check must assert root `./copilot-instructions.md` and hidden `.github/copilot-instructions.md` are distinct artifact classes.

**Warning signs:** Verification language says "copilot-instructions exists" without naming the path.

### Pitfall 4: Trying to Prove User-Home Global Artifacts from Repo-Local Commands

**What goes wrong:** The report claims global artifact parity for `~/.claude/CLAUDE.md` or `~/.claude/copilot-instructions.md` without actually controlling the home directory.

**Why it happens:** `getProjectInstructionTargets({ global: true })` advertises those targets in code.

**How to avoid:** Keep repo proof and home-directory proof separate. Use human-UAT or a temp-home simulation if needed.

**Warning signs:** The plan treats `global: true` outputs as repo files.

### Pitfall 5: Burying the Boundary in Phase Artifacts Only

**What goes wrong:** 64.3 writes a strong verification report, but nothing in `.planning/codebase/` explains the ownership split.

**Why it happens:** Phase artifacts are the easiest place to write.

**How to avoid:** Promote the boundary into codebase docs before closing the phase.

**Warning signs:** A future reader can still not answer "who owns `.github`, `.claude`, and `.markos-local`?" without reading 64.2 and 64.3 artifacts.

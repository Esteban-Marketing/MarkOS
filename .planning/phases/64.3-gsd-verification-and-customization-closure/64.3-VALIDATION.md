# Phase 64.3: GSD Verification and Customization Closure - Validation Strategy

**Purpose:** Define the validation contract for Phase 64.3 so execution can prove the dual-root GSD system closes cleanly after 64.2 through deterministic manifest refresh, end-to-end shared and localized generator proof, and durable documentation of the MarkOS customization boundary without reopening the ownership model.

## Phase Goal

Verify the aligned GSD surfaces end-to-end, refresh manifests or generated artifacts, and document the MarkOS customization boundary before Phase 65 planning resumes.

## Deliverables and Verification Methods

| Deliverable | Verification Method | Responsible | Dimension |
| --- | --- | --- | --- |
| Reusable repo-local helper refreshes `.github/gsd-file-manifest.json` and `.claude/gsd-file-manifest.json` deterministically | Direct helper invocation plus hash-match assertions against live files | AI | Manifest Closure |
| Root `copilot-instructions.md` and root `CLAUDE.md` are regenerated or verified through the real generator lanes | Generator command execution plus file-content and path-distinction assertions | AI | Root Artifact Closure |
| Hidden `.github/copilot-instructions.md` remains distinct from root `copilot-instructions.md` after closure | Static file-path and content assertions in focused tests | AI | Artifact Boundary |
| Shared `.github` and localized `.claude` instruction surfaces still honor the locked 64.2 ownership model | Static scans and focused node:test assertions | AI | Policy Integrity |
| `GSD-ALIGN-06` and `GSD-ALIGN-07` exist in `.planning/REQUIREMENTS.md` with narrow wording aligned to the roadmap goal | Registry text scan | AI | Requirements Traceability |
| One canonical `.planning/codebase/` ownership-boundary doc exists and names shared, localized, and client-owned surfaces | File existence and content scan | AI | Documentation Closure |
| `.planning/codebase/FILES.md` and `.planning/codebase/STRUCTURE.md` surface the new ownership model | Static file scans | AI | Inventory Visibility |
| Repo-local verification closes the system without overclaiming user-home or global artifact proof | Verification report plus explicit human follow-through artifact or section | AI / Human | Honest Scope Boundary |

## Human Testing Items

1. Review the new `.planning/codebase/` ownership-boundary document and confirm it clearly distinguishes `.github/**`, `.claude/**`, and `.markos-local/**` responsibilities.
2. Spot-check root `CLAUDE.md`, root `copilot-instructions.md`, and hidden `.github/copilot-instructions.md` and confirm the report describes them as three deliberate artifact roles, not interchangeable files.
3. If the phase records user-home or global artifact follow-through, confirm those steps are clearly labeled as manual evidence and not mixed into repo-local automated claims.

Verifiable via CLI or file reads:

## Exit Conditions

Phase 64.3 is only complete when:

1. `GSD-ALIGN-06` is satisfied by a real repo-local closure path: stale shared and localized manifests are refreshed deterministically, root artifacts are regenerated or proven current through the generator lanes, and focused automated proof covers both lanes and manifest freshness together.
2. `GSD-ALIGN-07` is satisfied by durable documentation: one canonical `.planning/codebase/` boundary document exists and the existing codebase inventories point to the same ownership model.
3. The locked 64.2 ownership model remains intact: `.github/**` stays canonical shared framework content, `.claude/**` stays the localized runtime and dispatch layer, root `CLAUDE.md` stays the localized contract, and root `copilot-instructions.md` stays the shared project contract.
4. Any user-home or global artifact proof is either explicitly simulated in-repo or clearly separated into human follow-through rather than overclaimed as automated closure.
5. The final automated gate is green for the focused dual-root bundle and the broader repo suite before phase verification begins.

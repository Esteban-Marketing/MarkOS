# Phase 64.3: GSD Verification and Customization Closure - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Verify the aligned dual-root GSD system end-to-end, refresh stale `.github` and `.claude` GSD manifests and any generated root instruction artifacts that now lag the 64.2 policy, and document the long-lived MarkOS customization boundary before Phase 65 planning resumes. This phase proves and closes the already-decided 64.2 policy; it does not redesign ownership, collapse `.claude` into `.github`, or broaden into general installer or runtime migration work.

</domain>

<decisions>
## Implementation Decisions

### Manifest refresh mechanism
- **D-01:** 64.3 should add one reusable repo-local helper to refresh `.github/gsd-file-manifest.json` and `.claude/gsd-file-manifest.json` deterministically.
- **D-02:** Manifest refresh must reuse the repo's existing hash-generation pattern and JSON shape rather than relying on manual hash edits or ad hoc JSON surgery.

### End-to-end verification depth
- **D-03:** 64.3 verification should fully automate all repo-local dual-root proof, including generator behavior, manifest freshness, shared-versus-localized instruction surfaces, and root-versus-hidden artifact distinction.
- **D-04:** Any proof for user-home or other global artifact lanes should be documented as explicit human follow-through, not claimed as fully automated repo-local proof unless the phase adds a deliberate simulation harness.

### Customization-boundary documentation
- **D-05:** 64.3 should add one new canonical `.planning/codebase/` document that explicitly defines ownership boundaries between shared `.github`, localized `.claude`, and client-owned `.markos-local` surfaces.
- **D-06:** 64.3 should also patch existing canonical inventories, specifically `.planning/codebase/FILES.md` and `.planning/codebase/STRUCTURE.md`, so the ownership model is visible from the normal codebase maps rather than hidden only in one new document.

### Requirements registry handling
- **D-07:** The missing `GSD-ALIGN-06` and `GSD-ALIGN-07` entries in `.planning/REQUIREMENTS.md` should be patched inside 64.3 as a narrow registry fix rather than left as a planning-only assumption.

### Scope guardrails
- **D-08:** 64.3 should keep the 64.2 ownership model closed: root `CLAUDE.md` stays the localized contract, root `copilot-instructions.md` stays the shared project contract, `.github/**` remains canonical shared framework content, and `.claude/**` remains the localized runtime and dispatch layer.
- **D-09:** 64.3 should not repoint `.claude/skills/**` into `.github/**`, convert `.github/copilot-instructions.md` into the project contract, or expand into broader installer or runtime redesign.

### the agent's Discretion
- Exact helper filename and placement for deterministic manifest refresh, so long as it is repo-local, reusable, and aligned with existing hash-generation patterns.
- Exact verification test layout, so long as it proves the three required closure layers: static contract scans, real generator behavior, and manifest freshness.
- Exact name and section structure for the new `.planning/codebase/` ownership document, so long as it becomes the durable canonical boundary reference and the existing inventories are updated to point at it.

</decisions>

<specifics>
## Specific Ideas

- Keep 64.3 as a closure phase with three deliverables only: deterministic manifest and generated-artifact refresh, end-to-end dual-root verification, and durable customization-boundary documentation.

- Make the ownership model obvious from the normal `.planning/codebase/` maps so later phases do not reopen the `.github` versus `.claude` versus `.markos-local` question.

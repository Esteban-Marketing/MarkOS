<objective>
Ship tenant env var sync: `markos env list | pull | push | delete` + server-side encrypted storage (migration 76) + 3 endpoints + F-104 contract.

The key security property: values stored encrypted at rest via pgcrypto pgp_sym_encrypt keyed on MARKOS_ENV_ENCRYPTION_KEY (server config — not a tenant secret); list endpoint returns only `value_preview` (first 4 chars + ellipsis) so members can see keys without secret exposure; pull endpoint is owner/admin-only.

CLI side: `env pull` lands values in `.markos-local/.env` with 0o600 perms, refuses to clobber uncommitted local vars unless `--force`, and supports `--diff` / `--merge` workflows (RESEARCH §Pitfall 4).

Purpose: `vercel env pull`-style workflow is a top-3 discoverability expectation for any SaaS CLI. Ships in Wave 2 so operators can self-serve deploys without touching dashboard.
Output: 1 migration + 1 library + 3 endpoints + CLI command with 4 subcommands + F-104 + 3 test files.
</objective>

<!-- RPC functions in migration 76 (simpler than bare SQL from JS): -->
```sql
create or replace function set_env_entry(
  p_tenant_id text, p_key text, p_value text, p_user_id text, p_encryption_key text
) returns void
language plpgsql security definer as $$
begin
  insert into markos_cli_tenant_env (tenant_id, key, value_encrypted, value_preview, updated_by)
  values (
    p_tenant_id, p_key,
    pgp_sym_encrypt(p_value, p_encryption_key),
    substring(p_value from 1 for 4) || '…',
    p_user_id
  )
  on conflict (tenant_id, key) do update set
    value_encrypted = pgp_sym_encrypt(p_value, p_encryption_key),
    value_preview   = substring(p_value from 1 for 4) || '…',
    updated_at      = now(),
    updated_by      = p_user_id;
end; $$;

create or replace function get_env_entries(p_tenant_id text, p_encryption_key text)
returns table (key text, value text) language sql security definer as $$
  select key, pgp_sym_decrypt(value_encrypted, p_encryption_key)::text
  from markos_cli_tenant_env where tenant_id = p_tenant_id;
$$;
```

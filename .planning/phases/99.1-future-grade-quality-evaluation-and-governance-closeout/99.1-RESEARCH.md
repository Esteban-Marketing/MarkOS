## Summary

Phase 99.1 should be implemented as an **additive premium-quality gate layer**, not as a new platform or a replacement for the existing acceptance evaluator. The repo already has the core pieces: Phase 95 provides a portable evaluation envelope and ranked review packaging, Phase 98 provides the governed shortlist-plus-winner reasoning contract, and Phase 99 already enforces shared tailoring alignment and rewrite-required behavior. The missing piece is a **measurable, machine-readable scorecard plus a closeout gate** that turns those ingredients into trusted premium-quality promotion criteria.

The best implementation path is to keep the current decision model and transport shape intact while adding one new nested quality section. In practice, keep the existing Phase 95 ranking fields and decision states, then add a `premium_quality` or `quality_dimensions` block that scores the six NLI-11 dimensions, preserves the NLI-12 ethics rules, and records the NLI-13 non-regression evidence. Promotion should remain fail-closed: no output is premium-ready unless the existing governance baseline stays green and the new dimensions all clear their floors.

**Primary recommendation:** extend the existing evaluation envelope with a portable premium-quality scorecard and a hard closeout gate, while preserving the current Phase 95/98/99 contract semantics unchanged for all supported surfaces.

## Repo-Native Implementation Path

### 1) Keep the current envelope, add one nested quality contract
Do **not** replace the current Phase 95 shape. Keep:
- `decision`
- `best_candidate`
- `runner_up`
- `score_breakdown`
- `artifact_flags`
- `governance_diagnostics`
- `review`
- `surfaces.{api,mcp,cli,editor,internal_automation}`

Add one nested quality section while keeping the existing top-level `decision` as the **overall verdict**:

```json
{
  "decision": "promotable|review_required|blocked",
  "quality_dimensions": {
    "personalization_depth": 0,
    "icp_fit": 0,
    "neuro_fit": 0,
    "naturality": 0,
    "specificity": 0,
    "grounded_usefulness": 0
  },
  "quality_gate": {
    "review_status": "passed|warnings|rewrite_required",
    "pass_floors": {},
    "failed_dimensions": [],
    "blocking_reasons": [],
    "required_fixes": []
  },
  "baseline_regressions": {
    "preview_safe": true,
    "provenance_required": true,
    "read_safe": true,
    "write_disabled": true,
    "route_trace_present": true,
    "provider_audit_present": true
  },
  "override": {
    "entry_type": "manual_override_note",
    "append_only": true,
    "actor_id": "operator-id",
    "rationale": "Required for audited human override only.",
    "evidence_refs": []
  }
}
```

Use the existing top-level `decision` as the single overall verdict, keep the new scorecard additive, and persist any human intervention through the append-only override note path already present in the repo. This preserves backward compatibility and satisfies NLI-14 by avoiding a rewrite.

### 3) Reuse the Phase 99 tailoring gate as the first blocker
Phase 99 already blocks generic, template-sounding, and ungrounded output through `rewrite_required`. Phase 99.1 should **not** duplicate that logic. Instead:
- run the shared tailoring gate first;
- if the result is `rewrite_required`, short-circuit premium promotion;
- otherwise compute the measurable quality dimensions;
- only outputs that clear both the tailoring gate and the numeric quality gate can stay `promotable`.

### 4) Add one repo-native governance closeout gate
Use the Phase 88 closure-gate pattern as the model for a quality closeout gate. The new gate should check:
1. all six NLI-11 dimensions are present and above floor for promotable outputs;
2. NLI-12 ethics and brand-safety diagnostics are clear;
3. NLI-13 preview-safe, provenance, and audit baselines are still true;
4. NLI-14 portability evidence exists across all supported surfaces.

The output should be a small read-safe closeout artifact, not a new service.

## Measurable Evaluation Dimensions

### Recommended dimensions and thresholds

| Dimension | What it measures | Recommended pass floor | Block when |
|----------|------------------|------------------------|------------|
| Personalization depth | How clearly the output uses the matched pain, desired outcome, objection, and trust posture | 80 | below 50 or no visible ICP evidence |
| ICP fit | Whether the content clearly follows the Phase 98 winner, audience stage, and rationale | 80 | winner is missing or the content could fit any audience |
| Neuro-fit | Whether the trigger and persuasion posture are governed, appropriate, and stage-aligned | 75 | unsupported or manipulative neuro language appears |
| Naturality | Whether the draft is plainspoken, human, concrete, and anti-hype | 75 | robotic template language or hype dominates |
| Specificity | Whether the draft names concrete mechanics, proof, and real constraints | 80 | output stays generic or abstract |
| Grounded usefulness | Whether the output gives actionable, evidence-aware help rather than vague polish | 80 | no clear mechanism, no next step, or no grounding |

### Decision rule
Use the current Phase 95 `decision` values unchanged as the **overall verdict**:
- **blocked** when any hard governance blocker exists, any baseline regression is present, or any dimension falls below the block floor;
- **review_required** when the output is directionally good but misses one or more pass floors;
- **promotable** only when the existing Phase 95 ranking is strong, the Phase 99 tailoring gate is not blocking, and all six quality dimensions clear the pass floors.

Human override is allowed only with an explicit audit note, rationale, and evidence references via the existing override-log seam. It must remain visible in the envelope and review package, and it must **not** erase blocker evidence or baseline-regression evidence.

### Recommended weighting posture
Keep the current Phase 95 ranking weights for candidate comparison and add the new quality scorecard as a **gate**, not a replacement. This avoids breaking existing consumers and keeps the upgrade additive.

## Architecture Patterns

### Pattern 1: Rank first, gate second
**What:** keep the Phase 95 ranking model for candidate ordering, then apply the new premium-quality gate as a second pass.  
**Why:** this is the cleanest additive upgrade and avoids breaking current transport and decision logic.

### Pattern 2: One portable quality contract across all surfaces
**What:** quality metrics and governance diagnostics travel inside the same read-safe payload, while each surface only changes presentation.  
**Why:** this directly satisfies the portability requirement and reuses the Phase 95 adapter pattern.

### Pattern 3: Fail closed on ethics and governance
**What:** unsupported trigger language, missing evidence, missing route trace, or preview-safety violations should block promotion immediately.  
**Why:** NLI-12 and NLI-13 are about trust and non-regression, not best-effort warnings.

### Pattern 4: Closeout is computed evidence, not a manual opinion
**What:** closeout readiness should be emitted from test-backed facts and machine-readable diagnostics.  
**Why:** this phase is about measurable governance confidence, so the evidence bundle must be deterministic.

### Anti-Patterns to Avoid
- replacing the existing Phase 95 scoring fields instead of extending them;
- using average-only quality scores that let one weak dimension hide behind a strong total;
- allowing overrides to silently bypass blockers;
- inventing new neuromarketing heuristics outside `MARKOS-REF-NEU-01`;
- creating per-surface scoring logic that drifts between API, MCP, CLI, editor, and internal automation.

## Common Pitfalls

### Pitfall 1: Replacing the existing score policy instead of extending it
**What goes wrong:** existing ranking, packaging, or consumer logic breaks.  
**How to avoid:** keep the old fields intact and append an additive quality block.

### Pitfall 2: Treating naturality as a simple banned-word list
**What goes wrong:** the metric becomes brittle and easy to game.  
**How to avoid:** combine phrase-pattern checks with proof density, ICP references, and plainspoken specificity.

### Pitfall 3: Confusing neuro-fit with stronger persuasion pressure
**What goes wrong:** outputs become manipulative or hype-heavy.  
**How to avoid:** score for governed fit and appropriateness, not for aggression or urgency.

### Pitfall 4: Letting a strong average hide a critical weakness
**What goes wrong:** one output gets promoted even though it is generic, unsafe, or poorly grounded.  
**How to avoid:** use per-dimension floors and hard blockers before any promotable decision is allowed.

### Pitfall 5: Forgetting the v3.6.0 non-regression posture
**What goes wrong:** new quality logic weakens preview safety, provenance, or auditability.  
**How to avoid:** keep `read_safe`, `write_disabled`, route trace, provider audit, and `provenance_required` as explicit carried baselines.

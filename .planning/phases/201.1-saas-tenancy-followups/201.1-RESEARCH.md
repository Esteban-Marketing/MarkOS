### Plan 01 — D-103 Canonical-JSON Spec (H4)

**Files touched:** `lib/markos/audit/canonical.cjs` (41 LoC, verified), `canonical.ts`, `canonicalizer-rpc.cjs/ts` (new), migration 90, `test/audit/canonical-spec.test.js` (new), `test/audit/canonical-fuzzer.test.js` (new), `docs/canonical-audit-spec.md` (new)

**Critical gap confirmed [VERIFIED]:** Current `canonical.cjs` line 29 uses `Object.keys(value).sort()` — JS's default Unicode code point sort. This is NOT binary-lex UTF-8 byte sort. For keys containing non-ASCII characters (e.g., `é` U+00E9 vs `e` U+0065), JS sort and Buffer.compare diverge. The fix (using `Buffer.compare(Buffer.from(a, 'utf8'), Buffer.from(b, 'utf8'))`) is spec-correct.

**NFC gap confirmed [VERIFIED]:** Current `canonical.cjs` has NO Unicode normalization. String values pass through `JSON.stringify(value)` without NFC normalization. Decomposed `é` (U+0065 + U+0301) and composed `é` (U+00E9) produce different bytes.

**Number handling:** `-0` currently falls to `JSON.stringify(value)` which serializes as `0` in most V8 versions — but `Object.is(value, -0)` check should be explicit. `NaN` and `Infinity`: current line 12-13 handles `!Number.isFinite(value) → return 'null'` which is correct [VERIFIED from canonical.cjs].

**Postgres function note:** The plan's inline SQL for `markos_canonicalize_jsonb_recursive` uses `p_payload` in the object case but the parameter name at that function scope is `p_value` (the recursive helper takes `p_value jsonb`). Plan 01 Task 2 action block line 289 references `p_payload -> v_key` which is a bug — should be `p_value -> v_key`. Planner should verify/fix before execution.

**Fuzzer test environment gate:** `canonical-fuzzer.test.js` requires `MARKOS_TEST_SUPABASE_URL` or it SKIPs. This is correct per existing pattern in `test/audit/hash-chain.test.js`. The fuzzer is NOT a blocking CI gate until Plan 11 wires staging Supabase.

**Test coverage gap:** Existing audit tests = `canonical.test.js`, `chain-checker.test.js`, `hash-chain.test.js` (3 files, ~23 tests) [VERIFIED: filesystem ls]. Plan adds `canonical-spec.test.js` (≥6 tests) + `canonical-fuzzer.test.js` (gated). All pre-existing tests must remain green.

### Plan 06 — D-106 Audit PII Erasure (M4)

**Files touched:** Migration 93, `lib/markos/audit/erasure.cjs/ts` (new), `lib/markos/audit/chain-checker.cjs/ts` (modified), `api/tenant/audit/erase-pii.js` (new), 4 test files

**Depends on:** Plan 01 (erasure uses `markos_canonicalize_audit_payload` to re-canonicalize the redacted payload)

**Hash-chain redaction edge case (critical):** The tombstone approach modifies the original row's `row_hash` (because the payload bytes changed). This means the chain from that row forward is technically broken if the verifier uses `prev_hash → row_hash` strictly. The chain-checker must be taught: "if `audit.pii_erased` row references original_row_id AND the hash mismatch is at that row, classify as `redaction_tombstone_detected` not `tampering_suspected`." The critical ordering constraint is: the tombstone audit row MUST have a `created_at` / `id` BEFORE or equal to the mutation — hence "writes tombstone row FIRST, then mutates." If this ordering is violated (e.g., concurrent erasure), the chain verifier could misclassify.

**Atomicity:** The erasure SQL fn must perform both writes (tombstone INSERT + original row UPDATE) in a single SQL transaction. This is achievable via a Postgres function with `SECURITY DEFINER` and explicit `BEGIN`/`COMMIT` (Postgres functions are implicitly transactional). The plan's must-have truth confirms this.

**RLS:** Only `service_role` + tenant-admin can call erasure fn. The fn itself emits an audit row — this is a recursive audit write (the audit of the audit erasure). The chain-checker should not special-case this; it is a normal audit row with `action='audit.pii_erased'`.

**GDPR Art. 17 compliance:** The tombstone approach is a recognized pattern for append-only ledgers under GDPR. The key invariant is that the original actor reference in the tombstone row preserves auditability of who triggered the erasure, while the erased row's payload no longer carries PII. [ASSUMED: this pattern is compliant with GDPR Art. 17 read with Recital 65 regarding pseudonymization in audit contexts — formal legal sign-off is out of scope for this phase per CONTEXT deferred items.]

**Test coverage:** 4 new test files (erasure, erasure-chain-tombstone, chain-checker-tombstone, erasure-atomicity). The atomicity test must verify that a simulated mid-fn crash (via mock) does not leave the DB in a state where the tombstone exists but the payload was NOT redacted (or vice versa).

## Risks and Pitfalls

1. **Hash-chain redaction tombstone race:** If two concurrent erasure calls target the same row, both may read the row before either writes the tombstone, resulting in a double-tombstone without a verified ordering. Mitigation: the erasure SQL fn must use `SELECT FOR UPDATE` on the target row to serialize erasure calls at DB level.

2. **Migration 96 gap:** Slot 96 is taken by `96_neuro_literacy_metadata.sql`. Plan 09 references migration 97 (not 96), so there is no direct collision — but the gap in the sequence could confuse tooling or future migration auditors. Document explicitly in Plan 09's SUMMARY.

3. **`markos_canonicalize_jsonb_recursive` variable name bug:** In the Plan 01 Task 2 SQL action block, the recursive helper uses `p_value` as the parameter but the object branch references `p_payload -> v_key` (wrong variable). This will cause a Postgres compile error. Must be fixed to `p_value -> v_key` before execution.

4. **Supabase transaction boundaries:** `approve.js` and `submit.js` use separate PostgREST calls. "Same DB txn as business write" requires either: (a) a stored procedure wrapping both writes, or (b) using Supabase's `begin()`/`end()` if supported in the project's client version. Plan 04 must verify which mechanism is available and document the transaction model.

5. **`obscenity` npm API surface:** The `obscenity` package API changed between 0.3.x and 0.4.x. The plan correctly adds a runtime adapter that checks `typeof built.hasMatch !== 'function'` before constructing `RegExpMatcher`. Executor must verify installed version's TypeScript declarations before writing integration (Plan 03 Task 2 action explicitly requires this).

6. **Edge-config single-flight map memory leak:** The module-scope `Map<slug, Promise>` accumulates entries across requests in long-lived Vercel function instances (Fluid Compute). Entries must be deleted after promise resolution. Plan 05 must include map cleanup on both resolve and reject.

7. **GDPR nonce race (consume-and-respond):** If the proxy response fails after nonce consumption, the user cannot retry — nonce is already consumed. Consider a "consumed-but-delivery-failed" state or a short re-issue window (e.g., 60s) for failed proxies. Plan 08 should address this edge case.

8. **WCAG CHECK constraint in Postgres:** Pure SQL relative-luminance computation requires: (1) parsing hex string to RGB, (2) linearizing sRGB channels, (3) computing weighted sum. This is feasible in PL/pgSQL but non-trivial. Plan 07 must implement the SQL luminance function correctly — floating-point precision differences between Node and Postgres may cause borderline colors to pass in one and fail in the other.

9. **Playwright virtual authenticator dependency:** The headless WebAuthn ceremony in Plan 10 uses Playwright's CDP virtual authenticator. This requires `--enable-webauthn-simulator` flag in newer Chromium versions. Verify the Playwright version supports this API without additional flags.

10. **`bad-words` package `Filter` constructor:** The plan references `new (Bad.default || Bad)()` — this is the correct CJS interop pattern for a package that might export as ESM default. Verify `bad-words ^4.0.x` exports correctly via `require()`.

11. **k6 for middleware load smoke:** Plan 11 references k6 100 RPS for 60s. k6 is not a Node package — it is a standalone binary. The staging-smoke CI workflow must install k6 separately or use a Docker-based k6 image.

12. **Canonical spec backward-compat:** The spec lock changes key sort from JS `.sort()` to `Buffer.compare()`. For ASCII-only keys (which all existing Phase 201 audit rows should have), these produce identical ordering. The backward-compat assertion (golden hash) in the fuzzer verifies this. If any existing row contains non-ASCII keys, the chain verifier will produce a false positive tamper alert — the migration must include a recanonicalize-on-mismatch sweep or document the expected mismatch as a known exception.

13. **Plan 11 openapi regen order:** `contracts/openapi.json` regen requires F-106 (Plan 03) and F-107 (Plan 08) YAML files to exist. Plan 11 must run after both Plans 03 and 08 are complete. The `depends_on` list in Plan 11 includes all 10 prior plans — correct.

14. **M1 passkey RP-ID unresolved:** While out of scope, the unresolved M1 (passkey on BYOD vanity host may fail because rpId cannot change without invalidating existing credentials) represents a real user-facing breakage when a tenant adds a BYOD domain and tries to use passkey. A runbook note or error message clarifying this limitation should be added even if the code fix is deferred.

15. **M5 ZIP export size bomb unaddressed:** If a malicious (or simply large) tenant has multi-GB attachment blobs, the GDPR export PassThrough streaming (Phase 201 Plan 07) will stream the full bundle to the signed URL. Consider adding a `GDPR_EXPORT_MAX_BYTES` guard in the streaming archiver.

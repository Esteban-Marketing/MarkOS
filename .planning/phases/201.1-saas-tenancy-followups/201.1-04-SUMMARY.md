## Diff Summary: Phase 201 → Phase 201.1

### api/approve.js (61 LoC → 36 LoC)

**Removed (Phase 201 post-res.end wrapper):**
```javascript
res.end = function patchedEnd(chunk, ...rest) { ... };
await handleApprove(req, res);
try { await enqueueAuditStaging(...); } catch { /* swallowed */ }
```

**Added (Phase 201.1 inline pattern):**
```javascript
const captured = await runWithDeferredEnd(req, res, handleApprove);
if (captured.status >= 200 && captured.status < 300) {
  try { await emitInlineApprovalAudit(req, captured, { action: 'approve' }); }
  catch (err) { return res.end(JSON.stringify({ error: 'audit_emit_failed', ... })); }
}
res.statusCode = captured.status; res.end(captured.body);
```

Same structure for `api/submit.js`.

## Same-Txn Caveat (Compensation Marker)

`handleApprove` / `handleSubmit` (in `onboarding/backend/handlers.cjs`, ~3000 LoC, not touched by D-101) write to potentially different schemas/Supabase clients than the audit staging table. A truly atomic "business + audit in one txn" requires refactoring that handler — out of scope for this delta phase.

D-101 implements the next-best fail-closed semantics: emit-before-response + explicit error path + `compensation: 'business_write_succeeded_audit_pending'` marker in the 500 body. This makes the cross-schema gap visible to callers and downstream observability tooling rather than silently swallowing it.

Full atomicity is deferred to a future handlers.cjs refactor phase.

<objective>
Eliminate the rate-limit race condition in `lib/markos/auth/rate-limit.cjs` by replacing the JS read-then-write upsert with an atomic Postgres function `increment_signup_rate(ip_hash text, hour_bucket timestamptz) returns int`. Closes review concern **H6** ("Plan 03 rate-limit race deployed before Plan 08 — concurrent attackers can exceed 5/hour").

Purpose: Phase 201 Plan 03 shipped the rate-limit library with an explicit TODO at line 71-72 of `lib/markos/auth/rate-limit.cjs`: *"Postgres-level increment on conflict isn't a one-liner in PostgREST — use RPC in prod. For now: best-effort upsert + RPC increment lives in Plan 08."* Plan 08 shipped without delivering the RPC. The current code uses `.upsert({ attempt_count: 1 })` with `ignoreDuplicates: false` — under concurrent fires from the same IP, two parallel requests both read `attempt_count=4`, both write `attempt_count=5`, and the limit is bypassed.

Output: (1) Migration 91 publishing `increment_signup_rate` as an atomic SQL function (single transaction, INSERT ... ON CONFLICT DO UPDATE SET attempt_count = attempt_count + 1 RETURNING attempt_count); (2) rewritten `recordSignupAttempt` calling the RPC; (3) test asserting 50 concurrent calls never produce attempt_count > expected — the limit holds at the DB layer.
</objective>

async function recordSignupAttempt(client, input) {
  const ip_hash = hashIp(input.ip);
  const ipWindow = windowStart(Date.now(), RATE_LIMITS.ip_hourly.window_ms);
  const { data, error } = await client.rpc('increment_signup_rate', {
    p_ip_hash: ip_hash, p_hour_bucket: ipWindow,

  if (error) throw new Error(`recordSignupAttempt: rpc failed: ${error.message}`);

  return { attempt_count: data };
}

-- Phase 201.1 D-105: atomic rate-limit increment for signups.
-- Closes review concern H6: eliminates the read-then-write race in lib/markos/auth/rate-limit.cjs.
-- The pre-existing markos_signup_rate_limits table (migration 83) ships with
-- columns (ip_hash text, window_start timestamptz, attempt_count int, updated_at timestamptz)
-- and unique constraint on (ip_hash, window_start). This migration only adds the SQL fn.

create or replace function increment_signup_rate(
  p_ip_hash text,
  p_hour_bucket timestamptz
) returns int
  language sql
  volatile
as $$
  insert into markos_signup_rate_limits (ip_hash, window_start, attempt_count, updated_at)
    values (p_ip_hash, p_hour_bucket, 1, now())
  on conflict (ip_hash, window_start) do update
    set attempt_count = markos_signup_rate_limits.attempt_count + 1,
        updated_at = now()
  returning attempt_count;
$$;

```javascript
async function recordSignupAttempt(client, input) {
  if (!client || typeof client.rpc !== 'function') {
    throw new Error('recordSignupAttempt: supabase client required');
  }
  const { ip, email } = input || {};
  if (typeof ip !== 'string' || typeof email !== 'string') {
    throw new Error('recordSignupAttempt: ip + email required');
  }

  const now = Date.now();
  const ipWindow = windowStart(now, RATE_LIMITS.ip_hourly.window_ms);
  const ip_hash = hashIp(ip);

  // Phase 201.1 D-105 (closes H6): atomic UPSERT+increment via SQL fn — no read-then-write race.
  const { data, error } = await client.rpc('increment_signup_rate', {
    p_ip_hash: ip_hash,
    p_hour_bucket: ipWindow,
  });
  if (error) throw new Error(`recordSignupAttempt: rpc failed: ${error.message}`);

  // Email throttle (1-per-minute per email) — not race-prone, single-PK last-write-wins.
  const { error: emailErr } = await client
    .from('markos_signup_email_throttle')
    .upsert(
      { email: email.trim().toLowerCase(), last_sent_at: new Date(now).toISOString() },
      { onConflict: 'email', ignoreDuplicates: false }
    );
  if (emailErr) throw new Error(`recordSignupAttempt: email upsert failed: ${emailErr.message}`);

  return { attempt_count: typeof data === 'number' ? data : 0 };
}
```

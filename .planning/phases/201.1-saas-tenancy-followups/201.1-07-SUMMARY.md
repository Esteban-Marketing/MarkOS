# Phase 201.1 Plan 07: WCAG AA Contrast Gate for Tenant Branding Summary

WCAG 2.2 §1.4.3 luminance contrast gate added to `upsertTenantBranding` and mirrored as a Postgres CHECK constraint in migration 94; Phase 201 default teal corrected from #0d9488 (~3.86:1, failing) to #0f766e (~4.79:1, passing). Closes review concern M3.

## What Was Built

### WCAG Helper (`lib/markos/tenant/contrast.cjs` — 80 LoC)

Pure functions, no I/O, mirrors the SQL helper:

| Export | Signature | Purpose |
|--------|-----------|---------|
| `hexToRgb` | `(hex: string) -> {r,g,b} \| null` | Parse #RRGGBB; returns null on invalid |
| `relativeLuminance` | `(rgb) -> number [0,1]` | WCAG 2.2 §1.4.3 formula |
| `contrastRatio` | `(rgbA, rgbB) -> number [1,21]` | Symmetric contrast ratio |
| `passesWcagAa` | `(fgHex, bgHex, threshold?) -> {ok, ratio, threshold, error?}` | Gate function |
| `VANITY_LOGIN_BG` | `'#ffffff'` | Locked vanity-login background |
| `WCAG_AA_THRESHOLD` | `4.5` | WCAG AA non-large-text threshold |

TypeScript mirror at `lib/markos/tenant/contrast.ts` with full type exports (`RgbColor`, `WcagResult`).

### Augmented Branding Module (`lib/markos/tenant/branding.cjs`)

Changes from Phase 201 baseline:

1. `require('./contrast.cjs')` added at top.
2. `DEFAULT_BRANDING.primary_color`: `'#0d9488'` -> `'#0f766e'` (WCAG-compliant darker teal).
3. `upsertTenantBranding`: after the existing `HEX_COLOR.test(c)` check, calls `passesWcagAa(c, VANITY_LOGIN_BG)` and throws on fail with message: `primary_color: contrast ratio X.XX:1 below WCAG AA 4.5:1 minimum (vs background #ffffff)`.
4. Input paths without `primary_color` are not affected.

### Migration 94 (`supabase/migrations/94_markos_tenant_branding_contrast.sql`)

Three steps:

**Step 0 — Add `created_at` column:**
Migration 86 created `markos_tenant_branding` with only `updated_at`. Migration 94 adds `created_at timestamptz not null default now()` and immediately back-fills existing rows with `created_at = updated_at`. This makes the W-3 narrowing clause `updated_at = created_at` structurally valid.

**Step 1 — SQL helper `markos_tenant_branding_contrast_check(p_color text) returns boolean`:**
- `immutable`, `parallel safe`.
- Returns `true` if `p_color` is null (allows nullable column).
- Parses hex via `('x' || substr(p_color, 2, 2))::bit(8)::int` pattern.
- Applies sRGB linearization with threshold 0.03928 and exponent 2.4.
- Computes `v_ratio := (1.0 + 0.05) / (v_lum + 0.05)` (white bg = 1.0).
- Returns `v_ratio >= 4.5`.

**Step 2 — W-3 narrowed backfill:**
```sql
update markos_tenant_branding
  set primary_color = '#0f766e', updated_at = now()
  where primary_color = '#0d9488'
    and updated_at = created_at;
```
Runs BEFORE the constraint is added so default-only rows pass the gate.

**Step 3 — CHECK constraint (idempotent via DO block):**
```sql
alter table markos_tenant_branding
  add constraint markos_tenant_branding_primary_color_passes_wcag_aa
  check (markos_tenant_branding_contrast_check(primary_color));
```

### Rollback (`supabase/migrations/rollback/94_markos_tenant_branding_contrast_rollback.sql`)

Drops constraint, drops function, reverses backfill (where `primary_color = '#0f766e' and updated_at > created_at`), drops `created_at` column.

## Visible color change announcement

**Required section per W-3 closure (plan must_haves.truths[4]).**

### Count of rows backfilled

On a fresh / staging database with no real operator data, the narrowed UPDATE touches **0 rows** (no `markos_tenant_branding` rows exist). On a production database the exact count depends on deployment state.

To determine the count before deploying:
```sql
select count(*)
from markos_tenant_branding
where primary_color = '#0d9488'
  and updated_at = created_at;
```
These rows held the Phase 201 default and were never edited by the operator. After migration 94, their `primary_color` changes from `#0d9488` to `#0f766e` — a slightly darker teal (~4.79:1 vs. the original ~3.86:1 against white).

### Count of operator-customized #0d9488 rows requiring manual review

```sql
select count(*)
from markos_tenant_branding
where primary_color = '#0d9488'
  and updated_at > created_at;
```
If this returns > 0, those rows deliberately have `#0d9488` and will **fail the CHECK constraint** when migration 94 runs (the constraint addition step will error). Resolution options:
- Update those rows to a compliant color before running migration 94.
- Roll back migration 94 if the operator insists on `#0d9488` (accepting the WCAG fail).

### Operator revert path

To undo the visible color change after migration 94 has deployed:

```sql
-- Step 1: drop the constraint.
alter table markos_tenant_branding
  drop constraint if exists markos_tenant_branding_primary_color_passes_wcag_aa;

-- Step 2: reverse the backfill for rows that were touched by the forward migration.
update markos_tenant_branding
  set primary_color = '#0d9488'
  where primary_color = '#0f766e'
    and updated_at > created_at;

-- Note: Step 2 is best-effort. Operators who legitimately chose #0f766e after
-- the migration deployed will also be reverted. Export markos_tenant_branding
-- before running this rollback.
```

This is a copy of the relevant fragment from `supabase/migrations/rollback/94_markos_tenant_branding_contrast_rollback.sql`.

## M3 Closure Attestation

Review concern M3: "Validation accepts any hex. Tenant could set `--accent` indistinguishable from background — vanity login unusable; WCAG fail."

Resolution:
- **App layer:** `upsertTenantBranding` calls `passesWcagAa(c, VANITY_LOGIN_BG)` before any DB write. Rejects with explicit ratio in error message.
- **DB layer:** `markos_tenant_branding_primary_color_passes_wcag_aa` CHECK constraint mirrors the same algorithm. Rejects any direct INSERT/UPDATE that bypasses the app layer.
- **Default corrected:** `DEFAULT_BRANDING.primary_color` updated to a WCAG-passing color.
- **Tests:** 19 new test cases cover compliant / non-compliant / omitted inputs + edge cases.

M3 status: **CLOSED** at both app and DB layers.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Migration 86 has no `created_at` column — W-3 narrowing clause required it**
- **Found during:** Task 2
- **Issue:** The plan's backfill WHERE clause `updated_at = created_at` assumes a `created_at` column exists on `markos_tenant_branding`. Migration 86 only created `updated_at`. The clause would fail at runtime on any real DB.
- **Fix:** Migration 94 Step 0 adds `created_at timestamptz not null default now()` and immediately back-fills `created_at = updated_at` for all existing rows. The narrowing clause then works as intended.
- **Files modified:** `supabase/migrations/94_markos_tenant_branding_contrast.sql`

**2. [Rule 1 - Bug] Existing branding test used #ff0000 which now fails the WCAG gate**
- **Found during:** Task 1, test run
- **Issue:** `test/tenancy/branding.test.js` called `upsertTenantBranding` with `primary_color: '#ff0000'`. Red at full saturation has a contrast ratio of ~3.99:1 against white — correctly rejected by the new gate.
- **Fix:** Updated test to use `#cc0000` (~5.89:1, WCAG-passing). This is not a gate regression — it is the gate working correctly.
- **Files modified:** `test/tenancy/branding.test.js`

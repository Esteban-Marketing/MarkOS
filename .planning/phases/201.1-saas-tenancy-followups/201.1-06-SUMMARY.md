# Phase 201.1 Plan 06: GDPR Art. 17 Erasure — Hash Chain Reconciliation Summary

Pseudonymize-with-tombstone policy: PII in audit log rows redacted in place, row_hash recomputed, tombstone audit row emitted, chain verifier cross-references tombstones to distinguish documented erasure from tampering. Closes review concern M4.

## What Was Built

### Migration 93 (`supabase/migrations/93_markos_audit_pii_erasure.sql`)

- Adds `redacted_at timestamptz` and `redacted_by text` columns to `markos_audit_log` (operator hints; NOT part of the canonicalized payload envelope; no chain impact).
- Defines `erase_audit_pii(p_audit_row_id bigint, p_redaction_marker text, p_actor_id text) returns table(id, tombstone_id, new_row_hash)`:
  - 5-step D-106 protocol: fetch + advisory lock + tombstone INSERT via `append_markos_audit_row` + deterministic redacted payload + `markos_canonicalize_audit_payload` recompute + UPDATE.
  - `SECURITY DEFINER` bypasses the migration-82 RESTRICTIVE block-UPDATE RLS policy.
  - Advisory lock `pg_advisory_xact_lock(hashtext('audit:' || tenant_id))` serializes against concurrent writers.
  - **ATOMICITY CONTRACT**: PL/pgSQL implicit transaction guarantees tombstone INSERT and payload UPDATE roll back together on any failure.
  - Double-redaction guard: raises if payload already contains `__redacted`.
  - Grants: `revoke from public; grant execute to service_role`.
- Defines `recanonicalize_legacy_audit_row(p_audit_row_id bigint, p_actor_id text) returns table(id, tombstone_id, old_row_hash, new_row_hash)`:
  - One-shot backward-compat: re-runs payload through `markos_canonicalize_audit_payload` and recomputes `row_hash`. Emits `'audit.recanonicalized'` tombstone.
  - Gated at the application layer via env flag `MARKOS_AUDIT_RECANONICALIZE_ENABLED=1`.
  - Same grants pattern as `erase_audit_pii`.

### Rollback (`supabase/migrations/rollback/93_markos_audit_pii_erasure_rollback.sql`)

Drops both functions, the index, and the two columns.

### Node Wrapper (`lib/markos/audit/erasure.cjs` + `.ts`)

- `eraseAuditPii(client, { row_id, reason, actor_id })`: validates input (positive integer row_id, non-empty reason ≤ 256 chars, non-empty actor_id), calls `client.rpc('erase_audit_pii', ...)`, returns `{ row_id, tombstone_id, new_row_hash }`.
- `recanonicalizeLegacyRow(client, row_id, actor_id)`: env-flag gated (`MARKOS_AUDIT_RECANONICALIZE_ENABLED=1` required), calls `recanonicalize_legacy_audit_row` RPC.

### Augmented Chain Verifier (`lib/markos/audit/chain-checker.cjs`)

Key changes to `verifyTenantChain`:

- **Tombstone pre-load**: scans all rows for `action IN ('audit.pii_erased', 'audit.recanonicalized')` and builds `tombstoneByOriginalId: Map<number, number>` (original_row_id → tombstone_row_id).
- **Row_hash mismatch classification**:
  - Mismatch + `tombstoneByOriginalId.has(row.id)` → `reason: 'redaction_tombstone_detected'`, `tombstone_row_id` populated. Chain continues.
  - Mismatch with no tombstone → `reason: 'tampering_suspected'`.
- **`ok` computation**: `breaks.every(b => b.reason === 'redaction_tombstone_detected')` — not `breaks.length === 0`.
- **Return shape extension**: `breaks[]` entries now include `tombstone_row_id?: number` when `reason === 'redaction_tombstone_detected'`.
- **Backward compat**: on chains with no redactions, `tombstoneByOriginalId` is empty and all mismatch paths fall through to `tampering_suspected`; behavior is identical to Phase 201.

### Tenant Endpoint (`api/tenant/audit/erase-pii.js`)

- `POST /api/tenant/audit/erase-pii` only (405 on other methods).
- Auth: `X-Markos-User-Id` + `X-Markos-Tenant-Id` headers required.
- RBAC: fetches `markos_tenants.org_id`, then checks `markos_org_memberships.role = 'owner'` (mirrors `api/tenant/audit/list.js` pattern).
- Defense-in-depth (T-201.1-06-04): fetches `markos_audit_log.tenant_id` for the supplied `row_id` and rejects on mismatch (403 `tenant_mismatch`).
- Delegates to `eraseAuditPii(client, { row_id, reason, actor_id: userId })`.
- Returns `200 { ok: true, tombstone_id, new_row_hash }` | 400 | 401 | 403 | 404 | 500.

## Chain Cascade Documentation (Test C)

After `erase_audit_pii` runs in production:
1. Tombstone is appended via `append_markos_audit_row` (captures old row2 hash as its `prev_hash`).
2. Row 2's `row_hash` is recomputed to match the redacted payload.
3. The verifier walks: row1 clean, row2 clean (hash recomputed to match), row3/tombstone `prev_hash mismatch` (its stored `prev_hash = old_row2_hash` != new expected `new_row2_hash`).

This cascade `prev_hash mismatch` on the tombstone row is a documented expected artifact — it does not indicate tampering. It is exercised by Test C in `erasure-chain-tombstone.test.js`. The tombstone-detection branch (`redaction_tombstone_detected`) is exercised by Test A, which simulates the scenario where the row_hash was NOT recomputed (e.g., legacy recanonicalization path).

## Check A1: Plugin settings UI accessibility

**Requirement coverage:** WL-01, WL-02, WL-04
**Goal:** Confirm the tenant settings surface exposes the Digital Agency toggle and related branded controls in a live environment.

### Steps

1. Sign in as a tenant admin.
2. Navigate to Tenant Settings -> Extensions.
3. Confirm the Digital Agency plugin is visible.
4. Toggle the plugin on and off without console or server errors.
5. If branding controls are present on the same surface, confirm they render with tenant-specific labels, theme tokens, or metadata.

### Commands

Optional endpoint smoke check before UI interaction:

```powershell
Invoke-WebRequest -Uri "http://localhost:3000/api/tenant-plugin-settings" -Method GET -Headers @{ Authorization = "Bearer <tenant-admin-token>" }
```

Optional server log capture during interaction:

```powershell
Get-Content -Path .\test_output.txt -Tail 50
```

### Pass criteria

- Digital Agency toggle is visible.
- Toggle interaction succeeds without a 4xx/5xx error.
- Surface remains tenant-scoped and branded correctly.

### Evidence fields

- Screenshot of the Extensions page before interaction
- Screenshot after toggle interaction
- Response payload or network trace for the settings API call
- Any console errors
- Any server-side errors
- Final verdict: Pass or Fail

## Check A2: Plugin disable gate in live environment

**Requirement coverage:** WL-03, WL-04
**Goal:** Prove that disabling the plugin blocks its tenant route in the running environment.

### Steps

1. Ensure the plugin is enabled.
2. Confirm the dashboard route is reachable while enabled.
3. Disable the plugin in settings.
4. Re-request the dashboard route.
5. Confirm the route now fails closed with `404` or `PLUGIN_DISABLED` semantics.

### Commands

Enabled-state probe:

```powershell
Invoke-WebRequest -Uri "http://localhost:3000/plugins/digital-agency/dashboard" -Headers @{ Authorization = "Bearer <tenant-admin-token>" }
```

Disable the plugin through the UI or, if needed, through the action/API surface used by the settings page.

Disabled-state probe:

```powershell
try {
  Invoke-WebRequest -Uri "http://localhost:3000/plugins/digital-agency/dashboard" -Headers @{ Authorization = "Bearer <tenant-admin-token>" }
} catch {
  $_.Exception.Response.StatusCode.value__
}
```

### Pass criteria

- Route resolves while enabled.
- Route fails closed immediately after disablement.
- Failure path is explicit and tenant-safe.

### Evidence fields

- Screenshot of enabled state
- Screenshot of disabled state
- Response body or status code while enabled
- Response body or status code while disabled
- Timestamp of disable action
- Final verdict: Pass or Fail

## Check A3: Tenant and operator billing UX review

**Requirement coverage:** BIL-01
**Goal:** Confirm the billing surfaces use reconciled billing vocabulary and evidence-first drilldown in a live session.

### Steps

1. Sign in as a tenant billing admin and open Tenant Settings -> Billing.
2. Review plan, usage, entitlement, and invoice/evidence sections.
3. Sign in as an operator or governance-capable user and open Admin -> Billing.
4. Compare tenant and operator vocabulary for plans, entitlements, evidence, and invoice line items.
5. Confirm the UI exposes drilldown or evidence references without falling back to raw internal-only terminology.

### Commands

Tenant summary endpoint:

```powershell
Invoke-WebRequest -Uri "http://localhost:3000/api/billing/tenant-summary" -Headers @{ Authorization = "Bearer <tenant-billing-token>" }
```

Operator reconciliation endpoint:

```powershell
Invoke-WebRequest -Uri "http://localhost:3000/api/billing/operator-reconciliation" -Headers @{ Authorization = "Bearer <operator-token>" }
```

### Pass criteria

- Tenant and operator billing pages render.
- Shared billing vocabulary is coherent across both surfaces.
- Evidence and reconciliation details are human-readable.
- No missing-state or error-state regressions block the review.

### Evidence fields

- Screenshot of tenant billing surface
- Screenshot of operator billing surface
- API response snapshots for both billing endpoints
- Notes on vocabulary mismatches, if any
- Notes on evidence readability, if any
- Final verdict: Pass or Fail

## Check A4: Real enterprise IdP callback validation

**Requirement coverage:** IAM-04
**Goal:** Confirm enterprise SSO callback behavior in a real non-production IdP flow and validate canonical-role mapping plus escalation denial.

### Steps

1. Configure a non-production enterprise IdP against Supabase Enterprise SSO.
2. Sign in with a test user whose external claim should map to a permitted canonical role.
3. Confirm the mapped canonical role after successful login.
4. Repeat with a test user whose external claim should be rejected or escalated.
5. Confirm the rejected or escalated login fails closed and emits the expected denial behavior.

### Commands

Optional local log capture before login:

```powershell
Get-Content -Path .\test_output.txt -Tail 100
```

Optional callback endpoint observation after the IdP redirect:

```powershell
Invoke-WebRequest -Uri "http://localhost:3000/api/auth/sso/callback?code=<captured-code>" -Method GET
```

Optional governance evidence capture after login attempt:

```powershell
Invoke-WebRequest -Uri "http://localhost:3000/api/governance/evidence" -Headers @{ Authorization = "Bearer <operator-token>" }
```

### Pass criteria

- Permitted external claim maps to the expected canonical role.
- Escalating or unmapped claim fails closed.
- Evidence of the mapping or denial path is retrievable.

### Evidence fields

- IdP name and environment
- Screenshot or trace of successful login
- Screenshot or trace of rejected/escalated login
- Resulting canonical role for successful path
- Denial reason or status for rejected path
- Governance or audit evidence reference
- Final verdict: Pass or Fail

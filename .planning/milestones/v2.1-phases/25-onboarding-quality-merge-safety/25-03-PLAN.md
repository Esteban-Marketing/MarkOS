# Plan 25-03: Regenerate and Approve Reporting

## Wave 2 — Make Outcomes Understandable

<task id="1" name="Clarify backend response contracts for regenerate and approve">
  <read_first>
    - onboarding/backend/handlers.cjs
    - onboarding/backend/agents/orchestrator.cjs
    - test/onboarding-server.test.js
  </read_first>
  <action>
    Review regenerate and approve responses and define clearer outcome states for:
    - full success
    - partial success with warnings
    - degraded fallback output
    - hard failure
    Ensure the response shape gives callers enough detail to handle these outcomes explicitly.
  </action>
  <acceptance_criteria>
    - Tests can distinguish warning-bearing success from true success and failure
    - Backend responses expose structured outcome data instead of only freeform error strings
  </acceptance_criteria>
</task>

<task id="2" name="Align onboarding UI statuses to the clearer contract">
  <read_first>
    - onboarding/onboarding.js
    - onboarding/index.html
  </read_first>
  <action>
    Update the onboarding UI flow so user-visible statuses, banners, or inline messages can represent warning states and degraded outcomes without forcing the operator to inspect logs.
  </action>
  <acceptance_criteria>
    - UI status handling can reflect warning-level outcomes separately from hard failures
    - Operators can tell when output was generated with fallback behavior or partial degradation
  </acceptance_criteria>
</task>

<task id="3" name="Lock the new outcome model with tests">
  <read_first>
    - test/onboarding-server.test.js
    - .planning/research/PITFALLS.md
  </read_first>
  <action>
    Add regression coverage around the new response contract so future onboarding changes cannot silently collapse warning states back into ambiguous success responses.
  </action>
  <acceptance_criteria>
    - The response and status model is enforced by tests, not only implementation intent
    - Later quality work can build on explicit outcome states instead of re-inventing them
  </acceptance_criteria>
</task>

## Verification

Targeted onboarding tests and a manual UI pass should be able to distinguish full success, warning-bearing success, degraded fallback output, and hard failure without reading server logs.

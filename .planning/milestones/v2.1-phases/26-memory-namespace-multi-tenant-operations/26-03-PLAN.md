# Plan 26-03: Chroma Mode and Health Reporting

## Wave 2 — Make Memory Mode Visible

<task id="1" name="Define a mode-aware Chroma health contract">
  <read_first>
    - onboarding/backend/chroma-client.cjs
    - onboarding/backend/handlers.cjs
    - bin/ensure-chroma.cjs
  </read_first>
  <action>
    Design a health contract that distinguishes at minimum:
    - local daemon available
    - cloud backend configured and reachable
    - cloud backend configured but unavailable
    - local boot attempted and failed
    - memory-backed features degraded or disabled
  </action>
  <acceptance_criteria>
    - Operators can tell which memory mode the system is using and why
    - Health failures are actionable instead of generic connection errors
  </acceptance_criteria>
</task>

<task id="2" name="Wire mode-aware reporting into runtime surfaces">
  <read_first>
    - onboarding/backend/handlers.cjs
    - README.md
    - .protocol-lore/WORKFLOWS.md
  </read_first>
  <action>
    Expose the new health and mode semantics through runtime status endpoints and operator docs so hosted and local memory behavior is understandable without reading source code.
  </action>
  <acceptance_criteria>
    - Runtime status outputs expose meaningful memory-mode information
    - Documentation explains the supported local and cloud operating modes clearly
  </acceptance_criteria>
</task>

<task id="3" name="Cover failure-reporting behavior with tests">
  <read_first>
    - test/onboarding-server.test.js
    - .planning/research/PITFALLS.md
  </read_first>
  <action>
    Add or extend tests so health and failure reporting remain stable across local-only, cloud-configured, and degraded-memory scenarios.
  </action>
  <acceptance_criteria>
    - Memory-mode reporting is protected against regression
    - Later runtime changes cannot silently collapse meaningful health states into generic errors
  </acceptance_criteria>
</task>

## Verification

Manual review and targeted runtime tests should be able to distinguish local, cloud, degraded, and unavailable memory states and explain their effect on onboarding behavior.

# Milestone v3.0 — MarkOS Literacy System

## Mission

Evolve the MarkOS Literacy system from bare infrastructure (Phase 32 plumbing) into a production-grade, business-model-aware knowledge base that grounds every generated artifact in validated marketing standards. The system follows a **pain-points-first** architecture: audience pain points captured during onboarding drive content retrieval ranking, skeleton template selection, and discipline prioritization — ensuring the first generated drafts attack the client's highest-leverage problems rather than generic best practices.

## Gap Analysis

1. **No content corpus.** Infrastructure is built; nothing to retrieve.
2. **Single-discipline retrieval.** Orchestrator hardcodes `Paid_Media`; other disciplines (`Content_SEO`, `Lifecycle_Email`, `Social`, `Landing_Pages`) are dead paths.
3. **No pain-points routing.** Seed captures `audience.pain_points` (min 2), but retrieval does not weight results by pain-point relevance.
4. **No dynamic skeleton templates.** Example files exist per business model, but there is no skeleton generator that produces a starter content pack tailored to a specific model + pain-points intersection.
5. **No secure DB provisioning flow.** Connection credentials are assumed preconfigured; no guided setup, validation, or isolation for new client environments.
6. **No onboarding→literacy activation.** Seed submission stores seed sections in vector memory, but does not trigger literacy content provisioning or skeleton hydration.

## Phase 39 — Pain-Points-First Content Corpus

### Objective

Author and ingest the foundational marketing literacy corpus organized by pain-point taxonomy, covering all five MSP disciplines with business-model-aware metadata.

### Key Components

- **Pain-Point Taxonomy.** Define canonical pain-point categories mapped to disciplines: e.g., "unclear attribution" → Paid Media tracking, "low organic visibility" → Content SEO, "high churn / low retention" → Lifecycle Email.
- **Discipline Coverage Matrix.** One literacy document set per MSP discipline (`Paid_Media`, `Content_SEO`, `Lifecycle_Email`, `Social`, `Landing_Pages`), each containing definition, evidence, tactic, benchmark, counter-indicator, and vocabulary chunks.
- **Business-Model Annotations.** Every literacy document carries frontmatter with `business_model` applicability tags (may be `["all"]` or specific subsets like `["B2B", "SaaS"]`), enabling filter-based retrieval.
- **Pain-Point Weighting Metadata.** Frontmatter includes a `pain_point_tags` array linking each document to the taxonomy. Retrieval can boost documents matching the client's declared pain points.

### Technical Requirements

- `literacy/` directory tree: `literacy/{discipline}/{doc_id}.md` with standardized frontmatter schema.
- Frontmatter schema extension: add `pain_point_tags: string[]` to `parseLiteracyFrontmatter()` in `literacy-chunker.cjs`.
- Ingestion validation: `ingest-literacy.cjs` rejects documents missing required frontmatter fields (`discipline`, `business_model`, `pain_point_tags`, `status`).
- Minimum corpus: 5 disciplines × 3 core documents = 15 documents; each producing ≥3 chunks.
- Tests: chunker frontmatter parsing for new fields; ingestion dry-run with sample corpus; vector store round-trip retrieval by pain-point filter.

## Phase 40 — Multi-Discipline Orchestrator Retrieval

### Objective

Replace the hardcoded single-discipline literacy fetch with a dynamic multi-discipline retrieval pipeline that selects disciplines based on the client's seed data and ranks results by pain-point relevance.

### Key Components

- **Discipline Router.** Map seed fields (`audience.pain_points`, `company.business_model`, `content.primary_channels`) to a ranked discipline list. Example: pain point "poor email engagement" → `Lifecycle_Email` top, `Content_SEO` secondary.
- **Pain-Point Boosted Retrieval.** Extend `getLiteracyContext()` to accept an optional `pain_point_tags` filter array. When provided, results matching declared pain points receive score amplification before merge into the prompt context.
- **Parallel Discipline Fetch.** Orchestrator issues `Promise.all()` across the top-N ranked disciplines (default 3) instead of a single hardcoded call.
- **Context Budget.** Enforce a token-budget cap on literacy context injected into LLM prompts. Rank and truncate across disciplines rather than per-discipline.
- **Telemetry.** Emit `literacy_retrieval_observed` event with `{ disciplines_queried, total_hits, pain_point_match_count, context_tokens }`.

### Technical Requirements

- New module `onboarding/backend/agents/discipline-router.cjs`: `rankDisciplines(seed) → string[]`.
- Extend `buildLiteracyFilter()` in `vector-store-client.cjs` to support `pain_point_tags CONTAINS ANY` filter condition.
- Refactor `orchestrator.cjs` lines 130–146: replace single `getLiteracyContext('Paid_Media', ...)` with `discipline-router` → parallel fetch → merge → budget trim.
- Backward compatibility: when no literacy content exists, behavior degrades identically to current (empty `standards_context`).
- Tests: discipline-router unit tests (seed→discipline ranking); orchestrator integration test with mocked multi-discipline hits; token budget enforcement test.

## Phase 41 — Dynamic Skeleton Template Generator

### Objective

Build a skeleton generator that produces a starter content pack — pre-filled template stubs customized to the client's business model and top pain points — that populates the `.markos-local/` workspace immediately after onboarding approval.

### Key Components

- **Skeleton Registry.** Define skeleton templates per business model × discipline intersection. Each skeleton is a markdown stub with sections pre-labeled for the model's typical focus areas (e.g., B2B/Paid Media → "ABM Campaign Brief," "LinkedIn Ads Audience Build"; DTC/Social → "UGC Brief," "Influencer Outreach Sequence").
- **Pain-Point Injection.** Skeleton output dynamically inserts the client's top 2–3 pain points as starter headings within each template, ensuring generated content targets those problems immediately.
- **Approval-Triggered Hydration.** After `POST /approve` completes MIR/MSP draft writes, a post-approval hook invokes the skeleton generator. Output lands in `.markos-local/MSP/{discipline}/SKELETONS/`.
- **Example Resolver Extension.** Extend `example-resolver.cjs` to also resolve skeleton base templates (not just LLM examples) using the same `MODEL_SLUG` map and file naming convention.

### Technical Requirements

- Skeleton templates stored in `.agent/markos/templates/SKELETONS/{discipline}/_SKELETON-{model_slug}.md`.
- New module `onboarding/backend/agents/skeleton-generator.cjs`: `generateSkeletons(seed, approvedDrafts) → { discipline, files[] }[]`.
- Post-approval hook in `handlers.cjs` `approve` handler: call `skeleton-generator` after draft writes succeed.
- Pain-point interpolation: `{{pain_point_1}}`, `{{pain_point_2}}` placeholders in skeleton templates replaced at generation time.
- Minimum skeleton coverage: 7 business models × 5 disciplines = 35 skeleton base templates.
- Tests: skeleton generation for each business model; pain-point injection produces valid markdown; post-approval hook integration test.

## Phase 42 — Secure Database Provisioning Flow

### Objective

Implement a guided, safe database connection and provisioning workflow that validates credentials, creates required tables and RLS policies, and isolates per-client data — without risking existing production data.

### Key Components

- **Connection Wizard.** Interactive CLI flow (`npx markos db:setup`) that collects Supabase URL + service role key and Upstash REST URL + token, validates connectivity, and persists credentials to `.env` (gitignored).
- **Schema Provisioning.** Idempotent migration runner that applies required tables (`markos_literacy_chunks`, `markos_artifacts`, `markos_vector_index_config`) only if missing. Uses `IF NOT EXISTS` semantics — never destructive.
- **RLS Policy Verification.** Post-migration check confirming row-level security policies are active and that anon-role access is denied on literacy tables.
- **Namespace Isolation Audit.** Verify Upstash Vector namespaces are client-slug-prefixed and that standards namespaces (`markos-standards-*`) are read-only for non-admin callers.
- **Health Snapshot.** After provisioning, emit a structured health report (reusing `vectorStore.healthCheck()`) confirming both providers are reachable and schema is correct.

### Technical Requirements

- New CLI command in `bin/install.cjs` or standalone `bin/db-setup.cjs`: guided prompts via `readline` (no external deps).
- Credential validation: test Supabase connection with a lightweight query (`SELECT 1`); test Upstash with `__health__` namespace probe.
- Migration runner: read `supabase/migrations/*.sql` files, apply in order, skip already-applied (track in `markos_migrations` table).
- Secret handling: credentials written to `.env` only; never logged, never committed. `.gitignore` enforcement check on setup completion.
- Rollback safety: provisioning creates but never drops. If a migration fails, the runner stops and reports the failing migration with actionable output.
- Tests: mock-based provisioning flow test; RLS verification test; health check assertion; idempotent rerun test.

## Phase 43 — Onboarding-to-Literacy Activation Pipeline

### Objective

Close the gap between onboarding seed submission and literacy system activation by wiring a post-submission pipeline that provisions literacy context, triggers skeleton hydration, and confirms the client workspace is literacy-ready.

### Key Components

- **Post-Submit Literacy Hook.** After successful seed validation and vector store upsert in the submit handler, trigger a literacy activation sequence: (1) determine applicable disciplines from seed, (2) verify literacy content exists for those disciplines, (3) flag missing coverage as `literacy_gaps` in the submission response.
- **Literacy Readiness Contract.** Define a `literacy_readiness` status (`ready`, `partial`, `unconfigured`) returned in the submit and status API responses. `ready` = all ranked disciplines have ≥1 literacy document ingested. `partial` = some disciplines covered. `unconfigured` = no literacy content or vector store not configured.
- **Activation Telemetry.** Emit `literacy_activation_observed` event on every submission, capturing `{ readiness_status, disciplines_available, disciplines_missing, business_model, pain_point_count }`.
- **Status Endpoint Extension.** Extend `GET /status` to include `literacy.readiness`, `literacy.disciplines_available[]`, and `literacy.last_ingestion_at` in the response payload.

### Technical Requirements

- Modify `onboarding/backend/handlers/submit.cjs`: after successful `upsertSeed()`, invoke discipline-router (Phase 40) to determine required disciplines, then probe vector store for each.
- New field in submit response: `literacy: { readiness: 'ready'|'partial'|'unconfigured', disciplines_available: string[], gaps: string[] }`.
- Extend `api/status.js` response shape with `literacy` block.
- Readiness probe: call `getLiteracyContext(discipline, 'health', {}, 1)` for each required discipline — if at least one result returns, discipline is available.
- Graceful degradation: if vector store is unconfigured, return `unconfigured` without error. Submission still succeeds.
- Tests: submit with fully stocked literacy → `ready`; submit with partial coverage → `partial` with correct gaps; submit with no vector store → `unconfigured`; status endpoint literacy block shape assertion.

## Phase 44 — End-to-End Literacy Integration Verification

### Objective

Validate the complete literacy lifecycle from content authoring through onboarding activation to draft generation, confirming that pain-points-first retrieval measurably improves output relevance.

### Key Components

- **E2E Test Suite.** Automated test that: (1) ingests a test literacy corpus, (2) submits a seed with declared pain points, (3) runs orchestrator draft generation, (4) asserts `standards_context` in drafts contains pain-point-relevant content from the correct discipline.
- **Literacy Coverage Dashboard Data.** Expose `GET /api/literacy/coverage` returning `{ disciplines: { [name]: { doc_count, chunk_count, last_updated, business_models[] } } }` for operator visibility.
- **Regression Guard.** CI gate that fails if literacy retrieval returns zero hits when a test corpus is present. Prevents silent regressions to empty standards context.
- **Operations Runbook.** Document the full operator workflow: install → db:setup → ingest content → verify coverage → onboard client → confirm literacy-grounded output.

### Technical Requirements

- New test file `test/literacy-e2e.test.js`: full lifecycle with test fixtures.
- New API route `api/literacy/coverage.js` (or extend `api/status.js`): discipline-by-discipline coverage report.
- CI workflow step: after test suite, assert non-zero literacy hits from test corpus.
- Runbook location: `.planning/codebase/LITERACY-OPERATIONS.md` linked from `TECH-MAP.md`.
- Test corpus fixture: `test/fixtures/literacy/` with ≥3 discipline directories, each containing 1 valid literacy markdown file.
- Tests: E2E lifecycle test; coverage endpoint shape test; regression gate test with empty and populated corpus.

## Risk Register

| Risk | Mitigation |
|------|------------|
| Literacy content authoring bottleneck delays Phase 39 | Start with 3 priority disciplines; expand in follow-on |
| Upstash Vector namespace proliferation | Enforce naming convention via `buildStandardsNamespaceName()`; audit in Phase 42 |
| Token budget overflow with multi-discipline context | Phase 40 enforces explicit context cap before merging into prompt |
| Skeleton templates become stale as business models evolve | Skeleton registry versioned; admin CLI `literacy-admin deprecate` covers retirement |
| RLS misconfiguration exposes cross-client data | Phase 42 includes explicit RLS verification step; test coverage required |
| Pain-point taxonomy drift across disciplines | Taxonomy defined once in Phase 39 and referenced by all subsequent phases |

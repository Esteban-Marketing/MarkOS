Source: Milestone discussion captured in `v3.3.1-OBSIDIAN-MIND-PIVOT-CONTEXT.md`

## Recommended Insertion Strategy

1. **Do not interrupt Phase 65.** It is a narrow hosted closeout lane for v3.3.0 and should remain untouched.
2. **Do not repurpose Phase 66.** Its context explicitly forbids redesigning the package, installer UX, or product identity before the `3.3.0` release is honestly hardened.
3. **Start `v3.3.1` only after Phase 66.** Treat Phase 66 as the final release-prep lane for the MIR/MSP-first model, then begin the vault-first replacement as the next milestone.
4. **Use whole-number phases rather than decimal insertion.** This is a milestone pivot, not an urgent patch between existing v3.3.0 plans.

Recommended first phase number: **67**

## Milestone Goal

Replace the MIR/MSP-first MarkOS operating model with a vault-first knowledge system inspired by Obsidian Mind, making the local Obsidian vault the canonical source of context, memory, workflows, and operator support.

## Milestone Constraints

- Obsidian becomes a required dependency for the new primary path.
- QMD remains optional.
- MIR/MSP stop being first-class canonical structures.
- Existing `.markos-local/MIR` and `.markos-local/MSP` users get an importer, not a long-lived sync bridge.
- The onboarding UI may survive only if it writes directly into the vault.
- MarkOS should prefer Obsidian Mind-native commands, hooks, and workflow patterns, with marketing overlays added on top.

## Proposed Phases

### Phase 67: Vault Taxonomy and Replacement Contract
**Goal:** Define the canonical MarkOS vault taxonomy that replaces MIR/MSP, lock the compatibility boundary, and specify how legacy artifacts import into the new structure.

**Depends on:** Phase 66

**Covers:**
- New top-level vault information architecture for marketing strategy, execution, people, evidence, and operator memory
- Explicit retirement contract for MIR/MSP terminology and paths
- Mapping spec from legacy `.markos-local/` content into vault-native note families
- Command, hook, and dashboard inventory that MarkOS will adopt from Obsidian Mind versus extend

### Phase 68: Vault-First Installer and Bootstrap Rewrite
**Goal:** Rework `npx markos` and update/bootstrap flows around vault-first setup, Obsidian dependency checks, optional QMD enablement, and project bootstrap into the new knowledge structure.

**Depends on:** Phase 67

**Covers:**
- Obsidian-aware install contract
- Vault bootstrap scaffolding instead of MIR/MSP template cloning
- Update-path behavior for vault-first installations
- Required dependency checks for Obsidian and optional QMD

### Phase 69: Vault-Native Onboarding and Legacy Importer
**Goal:** Replace publish-to-MIR onboarding behavior with vault-native authoring, while providing a one-way importer for existing `.markos-local/MIR` and `.markos-local/MSP` users.

**Depends on:** Phase 68

**Covers:**
- UI flow changes so onboarding writes into the vault
- Structured migration/importer from legacy trees into the new vault taxonomy
- Migration validation and operator-facing outcomes
- Retirement path for `write-mir.cjs` as canonical persistence

### Phase 70: Command, Hook, and Agent Surface Remap
**Goal:** Replace the current MarkOS-first protocol surface with an Obsidian Mind-native operating surface plus explicit marketing overlays.

**Depends on:** Phase 69

**Covers:**
- Session lifecycle hook adoption and adaptation
- Native command-family alignment
- Marketing-specific overlay commands and subagents
- Agent boot docs and protocol references updated to the new workflow model

### Phase 71: Bases, Evidence, and Marketing Performance Graph
**Goal:** Adapt Obsidian Mind dashboard, evidence, and performance-review concepts to MarkOS marketing execution, campaign memory, and operator reporting.

**Depends on:** Phase 70

**Covers:**
- Home dashboard and Base-style views for marketing operations
- Campaign evidence, review, wins, and outcomes graph
- Marketing-specific equivalents of performance/review structures
- Frontmatter and note-linking conventions for operational reporting

### Phase 72: Legacy Surface Demotion, Documentation Rewrite, and Milestone Validation
**Goal:** Demote MIR/MSP-era surfaces to legacy or migration status, rewrite public and internal docs around the vault-first product, and validate the new milestone end-to-end.

**Depends on:** Phase 71

**Covers:**
- README, protocol lore, and architecture rewrite
- Legacy compatibility boundaries and migration guidance
- Package positioning and public messaging rewrite
- End-to-end validation of install, onboarding, command lifecycle, dashboards, and importer flows

## Why This Breakdown

1. It keeps **taxonomy** ahead of **bootstrap**, so install work does not guess the target structure.
2. It keeps **bootstrap** ahead of **onboarding**, so UI or import flows have a real vault destination.
3. It separates **workflow remap** from **dashboard/evidence adaptation**, which reduces scope explosion.
4. It reserves **legacy demotion and docs rewrite** for the end, when the new canonical path already exists.

## Suggested Milestone Acceptance Criteria

1. A fresh `npx markos` install produces a usable Obsidian-based MarkOS vault without MIR/MSP-first scaffolding.
2. Existing `.markos-local/MIR` and `.markos-local/MSP` installs can be imported into the new vault with a documented one-way migration.
3. Session hooks, commands, and subagent behavior follow the Obsidian Mind-native operating pattern with explicit MarkOS marketing overlays.
4. Marketing dashboards, evidence capture, and review/win tracking work inside the vault-first model.
5. MIR/MSP-era docs and workflows are clearly marked legacy or removed from the canonical path.

## Not Recommended

- Inserting this work before Phase 66 as a decimal patch.
- Preserving long-lived MIR/MSP and vault dual-write behavior.
- Treating this as only a storage migration while leaving the product/installer/command surface conceptually unchanged.

# v3.9.0 Plugin Literacy Libraries Context

Status: Discussed, candidate next milestone
Date: 2026-04-14
Discussion mode: Milestone-level expansion discussion

## Intent

Define the next MarkOS milestone around plug-and-play literacy template libraries that are specialized by business model and industry so a new workspace can initialize with stronger default context, examples, and starter operating assets from day one.

## Grounded Repo Context

The repository already contains the right extension seams for this direction:

- business-model-aware example resolution already exists in onboarding example injection
- starter skeleton generation already hydrates discipline packs after approval
- plugin capability gating already exists for tenant-safe expansion patterns
- current template-family logic already recognizes families such as B2B, B2C, SaaS, services, consulting, and ecommerce

This milestone should extend those seams into a first-class library system rather than introduce a separate template engine or a parallel initialization stack.

## Locked Decisions

1. **Milestone posture:** This should be the next additive product milestone after v3.8.0, not a side experiment.
2. **Primary outcome:** MarkOS initialization should produce ready-to-consume literacy assets that feel tailored to the client immediately.
3. **Packaging model:** The library system should be layered:
   - a business-model base pack
   - an industry overlay pack
   - discipline-specific starter templates and examples
4. **Activation model:** Packs should be selected primarily from onboarding seed data, with optional operator override when needed.
5. **Business examples in scope:** B2B, B2C, SaaS, Ecommerce, and service-led companies are priority families.
6. **Industry examples in scope:** Travel, IT, Marketing Services, and Professional Services are strong initial verticals.
7. **Product boundary:** The goal is better initialization quality and faster time-to-value, not a rewrite of the full MarkOS architecture.
8. **Governance boundary:** All generated libraries must preserve the current tenant-safe, approval-aware, and evidence-friendly posture.

## What the Libraries Should Feed

The milestone should improve the assets MarkOS can generate or scaffold during project creation and approval, including:

- onboarding defaults and seed enrichment
- literacy reference examples used by agents during draft generation
- skeleton starter packs written into the local workspace
- discipline-specific prompts, checklists, and execution templates
- optional plugin-scoped bundles for repeatable business types

## Explicit Rejections

- Do not try to support every business model and every industry in one milestone.
- Do not replace the current MIR, MSP, approval, or onboarding systems.
- Do not make library selection opaque or fully autonomous with no operator visibility.
- Do not create a second template architecture that duplicates the existing resolver and skeleton flow.

## Immediate Planning Implications

1. **Library taxonomy and registry:** Define the canonical pack model for base family plus vertical overlay plus discipline assets.
2. **Coverage matrix:** Decide the first supported business-model and industry combinations with explicit acceptance coverage.
3. **Initialization flow wiring:** Extend onboarding and approval flows so the correct packs are loaded automatically into workspace setup.
4. **Authoring standards:** Standardize how literacy examples, skeletons, prompts, and starter docs are named, stored, and versioned.
5. **Validation and QA:** Add coverage checks so missing packs degrade gracefully and supported combinations are verifiably complete.

## Likely Phase Breakdown

1. **Phase 106:** Template library taxonomy, registry contract, and selection rules
2. **Phase 107:** Business-model starter pack expansion and library authoring baseline
3. **Phase 108:** Industry overlay packs and composable initialization rules
4. **Phase 109:** Onboarding, approval, and local workspace hydration integration
5. **Phase 110:** Coverage diagnostics, regression hardening, and usability closeout

## Key Open Questions For Planning

1. Which 4 to 6 industries should ship in the first milestone cut?
2. Which assets are mandatory at initialization versus nice-to-have?
3. Should operators be allowed to stack multiple overlays, or should one dominant profile win?
4. How much manual editing should be expected before a starter pack is considered useful?

## Recommended Next Planning Move

Turn this milestone direction into a formal v3.9.0 requirements and roadmap draft focused on:

- prioritized family and industry matrix
- initialization and plugin contracts
- template-authoring standards
- validation rules and regression gates

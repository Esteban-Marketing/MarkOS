# v3.3.1 Obsidian Mind Pivot Context

Status: Discussed, not yet inserted into the roadmap
Date: 2026-04-11
Discussion mode: Milestone-level replacement discussion

## Intent

Define the milestone direction for replacing the existing MIR + MSP folder model, workflows, and supporting infra with an Obsidian Mind-based knowledge system that becomes the canonical operating substrate for MarkOS.

## Locked Decisions

1. **Milestone posture:** `v3.3.1` is a replacement milestone, not a parallel-mode experiment.
2. **Product direction:** MarkOS becomes a **vault-first replacement** built around a local knowledge vault rather than the current MIR/MSP-first operating model.
3. **Scope depth:** This is a **full repo pivot**, not a narrow storage swap or compatibility-only layer.
4. **Knowledge model source:** MarkOS should take **strong inspiration from Obsidian Mind**, but not adopt the upstream structure verbatim.
5. **Terminology direction:** The milestone should **retire MIR and MSP as first-class structural concepts** in the new canonical layout.
6. **Migration contract:** Existing `.markos-local/MIR` and `.markos-local/MSP` installs should receive an **importer-only** migration path into the new vault. No long-lived sync bridge is desired.
7. **Onboarding direction:** The current guided onboarding UI may survive only if it **writes directly into the vault** instead of publishing into `.markos-local/` MIR/MSP trees.
8. **Dependency contract:** **Obsidian is required** for the new primary model. **QMD remains optional**.
9. **Workflow surface:** MarkOS should be **Obsidian Mind native first**, with marketing-specific overlays added on top rather than preserving the current MarkOS command surface as the primary abstraction.

## Upstream Obsidian Mind Elements Marked Non-Negotiable

- Session lifecycle hooks
- Vault-first folder philosophy
- Command family and agent patterns
- Performance review system
- Obsidian Bases dashboard model

## Explicit Rejections

- Do not keep MIR/MSP as long-lived canonical folders.
- Do not build a long-lived bidirectional bridge between legacy `.markos-local/` content and the new vault.
- Do not treat this as a knowledge-layer-only milestone.
- Do not preserve the current hosted-first posture as the primary product identity.

## Immediate Planning Implications

1. **Installer and updater rewrite:** `npx markos` and `npx markos update` will need a vault-first contract, including Obsidian-aware setup, dependency checks, and migration flows.
2. **Onboarding redesign:** `submit -> approve -> publish-to-MIR` must be replaced or rerouted so the output lands in the new vault taxonomy.
3. **Documentation rewrite:** README, protocol lore, agent boot docs, and architecture docs must stop describing MIR/MSP as the canonical model.
4. **Agent/workflow remap:** Existing MarkOS commands, skills, and workflows need to be re-evaluated against Obsidian Mind command families, hooks, and subagent patterns.
5. **Marketing taxonomy redesign:** A new vault-native marketing information model is required because MIR/MSP names are being retired structurally.
6. **Legacy path demotion:** `.markos-local/`, `write-mir.cjs`, MIR/MSP templates, and related readiness gates become migration/compatibility surfaces rather than the long-term core.

## Key Open Questions For Planning

1. What exact marketing taxonomy replaces MIR/MSP inside the vault?
2. How much of the current app/API/CRM runtime survives as product surface versus becoming a support shell around the vault?
3. What is the exact bootstrap flow for a fresh install on Windows/macOS/Linux when Obsidian is now required?
4. Which upstream Obsidian Mind commands should be adopted directly, and which require MarkOS-specific marketing variants?
5. How should review/performance constructs from Obsidian Mind be adapted to marketing execution evidence and campaign outcomes?

## Recommended Next Planning Move

Create a roadmap-ready milestone breakdown that separates:

- vault taxonomy and migration
- installer/update/bootstrap rewrite
- onboarding rewrite into vault writes
- agent/hook/command remapping
- documentation and packaging repositioning

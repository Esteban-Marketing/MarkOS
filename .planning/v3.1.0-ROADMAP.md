# Roadmap: v3.1.0 — Operator Surface Unification

## Milestone Overview

**Goal:** Unify marketing, sales, and customer communications execution in one operational surface with auditable workflows and measurable activation outcomes.

**Core Value Proposition:**
- **For Operators:** Execute end-to-end marketing tasks from a single UI with built-in approval workflows, evidence trails, and no-code approval gates
- **For Platform:** Complete API contract coverage, versioning guarantees, and role-based hardening that enables safe scaling to enterprise customers
- **For Teams:** Measured activation KPIs (time-to-first-task, evidence capture, self-service rate) with before/after baseline (v3.0 → v3.1.0)

**Status:** Roadmap locked 2026-04-02. Ready for Phase 45 planning and execution.

**Project Participation:**
- Operators (non-technical marketing/sales practitioners) — primary end-users
- Platform engineers (MarkOS core team) — implementation owners
- Project stakeholders (product leadership) — roadmap gatekeepers and acceptance reviewers

## Phase Breakdown (6 Phases: 45–50)

### Phase 45: Operations Flow Inventory & Canonical Contract Map

**Goal:** Audit all production MarkOS flows, create canonical flow registry, and establish API contract mapping foundation.

**Depends on:** Nothing (first phase of v3.1.0)

**Requirements:** 
- API-01: Every active MarkOS flow is mapped to versioned contracts
- *Partial* API-02: Contract schema established

**Deliverables:**
- `.planning/FLOW-INVENTORY.md` — Complete list of all active flows (onboarding, execution, reporting, admin) with actor roles, inputs, outputs
- `.planning/FLOW-CONTRACTS.md` — Mapping of each flow to contract(s) (flow_name → [contract_v1, contract_v2, ...])
- `contracts/schema.json` — Canonical contract schema for request/response/error structure
- `.planning/FLOW-VERIFICATION.md` — Checklist confirming 100% flow coverage, no orphans, zero duplicates
- `test/api-contracts/phase-45-flow-inventory.test.js` — Test suite validating flow coverage

**Success Criteria:**
1. Operator can browse canonical flow inventory UI mockup showing ≥10 active flows with descriptions (requirement traceability: API-01)
2. All production flows are mapped to ≥1 contract; FLOW-VERIFICATION.md shows 100% coverage with zero orphaned flows (requirement traceability: API-01)
3. Contract schema `contracts/schema.json` validates all existing flow contracts; CI validation passes (requirement traceability: partial API-02)
4. FLOW-VERIFICATION.md is reviewed and approved (zero open checklist items, documented as reviewed)
5. Baseline T0 KPI metrics are captured and stored in telemetry database for Phase 50 comparison (time-to-first-task, evidence rate, self-service rate, flow coverage)

**Dependencies & Blockers:**
- Must complete before Phase 46 UI can reference flows
- Must complete before Phase 47 OpenAPI generation (contracts need to exist first)

**Plans:** 4–5 plans

### Phase 46: Operator Task Graph UI (MVP)

**Goal:** Ship a functional, integrated task graph UI with step-by-step execution, approval checkpoints, and evidence panel.

**Depends on:** Phase 45 (flow inventory and contract mapping)

**Requirements:**
- OPS-01: Live task graph showing current execution state and queued actions
- OPS-02: Step-by-step action runner with explicit state transitions (start → in-progress → complete)
- OPS-03: Approval checkpoint gate blocking execution until human decision recorded
- OPS-04: Retry capability with optional input mutation and attempt logging
- OPS-05: Evidence panel displaying inputs, outputs, logs, timestamps, actor metadata

**Deliverables:**
- `app/(markos)/operations/tasks/task-graph.tsx` — Live task graph component rendering from approved MIR/MSP state
- `app/(markos)/operations/tasks/step-runner.tsx` — Step-by-step state machine and execution orchestrator
- `app/(markos)/operations/tasks/approval-gate.tsx` — Approval checkpoint modal with reason collection
- `app/(markos)/operations/tasks/evidence-panel.tsx` — Immutable evidence viewer (inputs, outputs, audit trail)
- `app/(markos)/operations/tasks/page.tsx` — Operations hub integrating all components
- `test/ui-operations/task-graph.test.js` — Component interaction and state transition tests
- `test/ui-operations/evidence-capture.test.js` — Evidence panel and immutability verification
- Storybook stories for 5 task UI states (queued, approved, executing, completed, failed)
- MarkOSDB integration: Task state mutations persist to Event Store for audit trail

**Success Criteria:**
1. Task graph UI renders without error for all 5 task states; Storybook stories are live and accessible (requirement traceability: OPS-01)
2. Step state transitions (start → in-progress → complete) are atomic and logged with timestamps in immutable Event Store (requirement traceability: OPS-02)
3. Approval checkpoint blocks task execution until human approves or rejects; rejection reason is captured and persisted (requirement traceability: OPS-03)
4. Failed task steps can be retried with input preview/edit; retry attempts are logged as separate execution records (requirement traceability: OPS-04)
5. Evidence panel displays for ≥95% of executed steps; evidence includes inputs, outputs, logs, timestamps, actor ID; evidence is immutable post-execution (requirement traceability: OPS-05)

**Dependencies & Blockers:**
- Requires Phase 45 flow inventory to populate task selectors
- Blocks Phase 50 onboarding (operator UI must exist before onboarding can reference it)
- T1 KPI measurement begins after Phase 46 ships; baseline post-UI time-to-first-task captured

**Plans:** 6–8 plans

### Phase 47: OpenAPI Generation Pipeline & Versioning Policy

**Goal:** Auto-generate authoritative OpenAPI spec from contract files; lock versioning policy and document per-flow contracts.

**Depends on:** Phase 45 (contracts exist) and Phase 46 (UI context informs API design decisions)

**Requirements:**
- API-02 (complete): Versioned OpenAPI contracts for all flows
- API-03: Auto-generated OpenAPI spec included in codebase
- CONTRACT-01: Versioning policy documented (SemVer + deprecation windows)

**Deliverables:**
- `bin/generate-openapi.cjs` — CLI tool to regenerate `api/openapi.yaml` from `contracts/` directory
- `api/openapi.yaml` — Authoritative OpenAPI 3.0 spec generated from contracts (human-readable, CI-validated)
- `api/contracts/{flow_name}.v1.yaml` — Per-flow versioned contracts (one file per flow per version)
- `.planning/codebase/API-VERSIONING-POLICY.md` — SemVer + deprecation window rules (e.g., 2-minor-version overlap), migration examples, breaking-change definition
- `.github/workflows/generate-openapi.yml` — CI step to validate spec generation; spec must match implementation
- `test/api-contracts/phase-47-openapi.test.js` — Tests for spec validity, endpoint coverage, version schema consistency

**Success Criteria:**
1. OpenAPI spec (`api/openapi.yaml`) is valid against OpenAPI 3.0 schema; CI validation passes (requirement traceability: API-02, API-03)
2. 100% of contract endpoints documented in OpenAPI spec; endpoint count matches contract count; no undocumented endpoints discovered (requirement traceability: API-02, API-03)
3. API versioning policy is documented in `.planning/codebase/API-VERSIONING-POLICY.md` and reviewed by project owner; policy covers SemVer, deprecation windows, compatibility guarantees (requirement traceability: CONTRACT-01)
4. All per-flow contracts follow canonical schema from Phase 45; `contracts/schema.json` validation passes for all versioned files (requirement traceability: API-02)
5. CI enforces spec ↔ implementation parity: `npm run generate-openapi` must pass before each commit; breaking changes are flagged (requirement traceability: API-02, API-03)

**Dependencies & Blockers:**
- Requires Phase 45 contracts to exist
- Enables Phase 48 contract testing (contracts must be stable before tests are written)
- Blocks v3.1.0 API public documentation (requires stable versioning policy)

**Plans:** 4–5 plans

### Phase 48: Contract Testing Framework & CI Compatibility Gates

**Goal:** Ship contract test suite enforcing backward compatibility; prevent unapproved breaking changes in CI.

**Depends on:** Phase 47 (contracts and versioning policy are stable)

**Requirements:**
- API-04: Contract test suite enforces request/response shape and lifecycle assertions
- CONTRACT-02: Backward compatibility enforced (v1 clients work on v2 endpoints)

**Deliverables:**
- `test/api-contracts/framework.js` — Reusable test utilities for contract validation (schema checking, version compat, error scenarios)
- `test/api-contracts/{flow_name}.v1.test.js` — ≥1 test per contract covering happy path, error paths, version compat (3–5 tests per contract × N contracts)
- `test/fixtures/contracts/` — Sample request/response payloads per contract for mock testing
- `.github/workflows/contract-tests.yml` — CI workflow running contract test suite; blocks merge on failure; reports breaking changes clearly
- `test/api-contracts/phase-48-breaking-changes.test.js` — Specific test to validate v1 client backward compatibility against v2 API
- Documentation: `.planning/codebase/CONTRACT-TESTING-GUIDE.md` with test authoring patterns and mock fixture structure

**Success Criteria:**
1. Contract test suite is a required CI gate: no merge without passing contract-tests workflow; CI blocks on failure with clear breaking-change message (requirement traceability: API-04, CONTRACT-02)
2. ≥1 test per contract covering happy path and ≥2 error scenarios (test count ≥ contract count × 3); all tests pass consistently (requirement traceability: API-04)
3. Version compatibility test (v1 client requests) successfully calls v2 endpoints without breaking; v1 response schema requirements are respected on v2 endpoints (requirement traceability: CONTRACT-02)
4. CI reports breaking changes clearly (i.e., endpoint removed, required field deleted, status code changed); report includes change type and migration path (requirement traceability: API-04, CONTRACT-02)
5. Contract fixtures cover ≥90% of endpoint parameter variations (optional params, error codes, edge cases); fixture coverage metric is tracked in CI (requirement traceability: API-04)

**Dependencies & Blockers:**
- Requires Phase 47 contracts and versioning policy
- Enables Phase 49 hardening (API must be stable and tested before role guards are applied)
- Required before v3.1.0 release (contract testing is mandatory acceptance gate)

**Plans:** 5–6 plans

### Phase 49: Hardening Layer (RBAC, Diagnostics, Preflight Checks, Rollback Safety)

**Goal:** Harden operations surface with role-based access control, health diagnostics, migration safety, and rollback mechanisms.

**Depends on:** Phase 46 (UI exists) and Phase 48 (APIs are tested and stable)

**Requirements:**
- OPS-06: Inline guardrails before risky actions (migrations, rollbacks, bulk ops)
- RBAC-01: Role-based access control enforced at UI and API layers; unauthorized actions blocked
- RBAC-02: Operator can see assigned role and permissions in-product
- HARD-01: Preflight checks prevent unsafe state transitions during migrations
- HARD-02: Rollback endpoint exists; restores pre-operation state; idempotent
- HARD-03: Health diagnostics surface operator/admin view (DB, vector store, provider status, 30s refresh)

**Deliverables:**
- `app/(markos)/operations/rbac-enforcer.ts` — Middleware guards for role-based access at API layer
- `app/(markos)/operations/role-badge.tsx` — Role badge + permission checklist UI component
- `app/(markos)/operations/preflight-check.tsx` — Pre-migration validation UI (current state → target state → rollback plan)
- `app/(markos)/operations/health-dashboard.tsx` — System health monitor (DB, vector store, provider status, 30s refresh)
- `api/rollback/[tx_id].ts` — Rollback endpoint with pre/post state validation and idempotency key
- `app/(markos)/operations/guardrails/risky-action-modal.tsx` — Inline warning UI before risky actions
- `test/rbac-operations/` — RBAC enforcement tests (all protected routes return 403 for unauthorized roles)
- `test/operations-hardening/` — Preflight validation, rollback idempotency, health probe failure modes
- Database migration: Add `transaction_id`, `pre_state_hash`, `post_state_hash` columns to audit tables

**Success Criteria:**
1. 100% of protected operations routes are guarded by RBAC middleware; unauthorized role attempts return 403; CI test suite verifies all protected routes with unauthorized roles (requirement traceability: RBAC-01)
2. Operator can view their assigned role and full permission list in settings page; role badge displays role name and permission count; permissions list is sortable and searchable (requirement traceability: RBAC-02)
3. Preflight check executes before migration; identifies unsafe state transitions; blocks migration if preflight fails; operator sees clear reason for block and remediation steps (requirement traceability: HARD-01)
4. Rollback endpoint (POST `/api/rollback/{tx_id}`) successfully restores pre-operation state; is idempotent (multiple calls produce same result); validates pre/post state hashes on completion (requirement traceability: HARD-02)
5. Health dashboard displays component status (green/yellow/red); updates every 30 seconds; alerts immediately on component degradation (DB timeout, vector store unavailable); "TRY-AGAIN" CTA available (requirement traceability: HARD-03)

**Inline Guardrails** (derived from OPS-06):
- Before migration execution: Modal shows "Current State → Target State" with abort/confirm
- Before bulk operations (>100 items): Banner warns "This will affect X items" with explicit confirm button
- Before rollback: Modal shows "Will restore to [timestamp]" with manual confirmation

**Dependencies & Blockers:**
- Requires Phase 46 UI to exist (guardrails displayed inline)
- Requires Phase 48 stable APIs (hardening doesn't add new endpoints, only guards existing ones)
- Blocks Phase 50 onboarding (onboarding must guide operators through role assignment)
- Must be complete before enterprise pilot launch

**Plans:** 7–8 plans

- `app/(markos)/onboarding/operator/role-selector.tsx` — Role selection UI with permission previews

- `.planning/ACTIVATION-KPI-REPORT.md` — Final T1 vs T0 comparison (target measurements: time ≤5 min, evidence ≥95%, self-service ≥85%, coverage 100%)

**Success Criteria:**
1. Operator onboarding wizard completes error-free; operator is assigned role and first task is queued after wizard completion (requirement traceability: ONBOARD-02)
2. New operator can execute first task without manual engineering support; exit survey shows zero blocking issues; support ticket count = 0 for first-time operators (requirement traceability: ONBOARD-01)
3. E2E test passes consistently in CI; fresh operator session → wizard → first task execution → evidence captured → task marked complete (requirement traceability: ONBOARD-01, ONBOARD-02)
4. T1 metrics show ≥70% improvement vs T0 baseline (i.e., time-to-first-task reduced from ~15–30 min to ≤5 min; evidence from 60% to ≥95%; self-service from 30% to ≥85%); telemetry records confirm measurements (requirement traceability: entire milestone)
5. All flows from FLOW-INVENTORY are accessible from operator UI task selector; flow coverage is 100%; no flows require CLI/code access (requirement traceability: API-01)

**Dependencies & Blockers:**
- Requires Phase 46 UI to exist (onboarding references task graph)
- Requires Phase 49 RBAC and health stable (onboarding must handle role assignment safely)
- Requires Phase 47 versioning policy stable (onboarding must not trigger unapproved API changes)
- Phase 50 completion = v3.1.0 milestone completion

```

    ↓ [UI exists, APIs stable]      ↓ [Contracts exist, versioning policy set]
    ↓                              Phase 48 (Contract Testing Framework)
```

**Dependency Rationale:**

| Dependency | Why |

| Phase 46 ← → Phase 47 | Can execute in parallel; UI informs API design, OpenAPI documents UI endpoints |
| Phase 47 → 48 | Contracts must be stable before writing tests (avoid test rewrites) |
| Phase 48 → 49 | APIs must be tested and validated before hardening layer adds guards |
| Phase 49 → 50 | RBAC role assignment and health checks required by onboarding flow |
| Phase 50 (final) | Measures T0 vs T1 baseline; confirms all phases deliver; gates v3.1.0 completion |

## Risk Summary & Controls

### Key Risks

| Risk | Likelihood | Impact | Control |
|------|------------|--------|---------|
| **Scope Explosion:** Trying to ship all UX, API parity, and hardening in one milestone | HIGH | CRITICAL | Freeze flow inventory (Phase 45) before UI/API work starts; reject scope creep requests mid-phase |
| **Inconsistent Contract Ownership:** Legacy routes have conflicting contract definitions | MEDIUM | HIGH | Phase 45 audit enforces single contract per flow; contract schema enforced in CI |
| **Hidden Migration Edge Cases:** Preflight checks miss unsafe transitions; rollback fails silently | MEDIUM | CRITICAL | Phase 49 preflight test covers known migration scenarios; rollback idempotency validated in CI |
| **Operator UX Overload:** Evidence panel, guardrails, role badge crowd UI and confuse users | MEDIUM | MEDIUM | Usability review in Phase 46 with operator cohort; simplify if needed before Phase 50 |
| **API Breaking Changes Slip Past Tests:** Contract tests don't catch subtle backward-compat issues | LOW | HIGH | Phase 48 contract framework includes version compat test; v1 client explicitly tested on v2 |
| **Health Dashboard Noise:** False positives trigger false alarms; operators ignore real alerts | LOW | MEDIUM | Phase 49 health thresholds tuned based on v2.6 ops baseline; alert tuning checkpoint before Phase 50 |
| **Onboarding Wizard Skip-Through:** New operators rush through step-by-step walkthrough without understanding | MEDIUM | LOW | Phase 50 wizard includes knowledge checks; completion confirmed before task execution allowed |

### Controls Applied

1. **Freeze Phase 45 Flow Registry:** No new flows added after Phase 45 complete (only backlog items or v3.2 deferred)
2. **Contract-as-Source-of-Truth:** Generate OpenAPI, tests, and docs from contract artifacts (single source reduces drift)
3. **Staged CI Gates:** Phase 45 gates flow coverage, Phase 47 gates spec validity, Phase 48 gates breaking changes, Phase 49 gates RBAC coverage
4. **Operator-First UX Review:** After Phase 46 MVP ships, conduct 2-3 operator walkthroughs to validate clarity and error recovery
5. **Baseline Measurement Protocol:** T0 KPI baseline captured at Phase 45 kickoff; T1 measured at Phase 50 completion (before/after comparison locked)
6. **Rollback Test Harness:** Phase 49 includes mock migration scenarios; rollback tested on each scenario (deterministic verification)
7. **Manual Approval Gate:** Phase 50 completion requires human sign-off on T1 metrics; threshold not auto-bypassed

# v4.0.0 Testing Environment Plan

**Date:** 2026-04-23  
**Purpose:** Define the canonical test program for Phases 204-228 so execution agents ship business-logic-backed evidence, not only implementation.

## Executive decision

MarkOS should treat testing as a three-layer system:

1. `Vitest` for deterministic business logic, contracts, reducers, schemas, policies, and integration harnesses.
2. `Playwright` for route-level and workflow-level browser proof.
3. `Chromatic` for stable UI state and visual regression coverage through Storybook.

The current repository already has broad automated coverage plus a Storybook/Chromatic lane, but it does **not** yet have a first-class `Vitest` or `Playwright` foundation in the repo root. This plan makes those stacks mandatory for new execution work from Phase 204 forward while allowing legacy `node --test` suites to remain in place during migration.

## Current-state audit

Grounded in the current repo:

- Primary automated test runner today is `node --test`.
- Storybook exists under `.storybook/` and Chromatic is wired through `package.json` and `.github/workflows/ui-quality.yml`.
- `axe-playwright` is already used by Storybook's test runner.
- Many current UI tests are source-contract assertions over route files and story files rather than full browser workflows.
- No `vitest` dependency or `vitest.config.*` exists yet.
- No `playwright` dependency or `playwright.config.*` exists yet.
- No single planning artifact currently tells execution agents which business truths must be proven by unit/integration/browser/visual tests per future phase, including the new commercial-engine lane from documents 18-26.

## Canonical doctrine

### 1. Tool roles are fixed

#### Vitest owns:

- pure business rules
- reducers and stores
- schemas, contracts, validators, and serializers
- route-handler logic and policy gates
- task/approval/run state machines
- pricing, evidence, billing, and growth decision logic
- integration harnesses that can run deterministically in CI

#### Playwright owns:

- critical user journeys
- route-level authorization and redirect behavior
- approval and operator workflows
- multi-step forms and wizards
- browser-visible error handling
- end-to-end proof for "can a human/operator actually do the thing"

#### Chromatic owns:

- visual state regression
- empty/loading/error/success/denied variants
- high-risk operator cards, tables, drawers, banners, and dialogs
- pricing, approval, evidence, billing, and growth surfaces where layout drift changes operator understanding

### 2. Business-logic tie is mandatory

Every plan must test the real business truth it introduces. Example:

- Pricing recommendation work is not done because the page renders; it is done when unsupported pricing changes are blocked and recommendation evidence remains explainable.
- Approval UI work is not done because buttons exist; it is done when unsupported claims, missing pricing context, and forbidden mutations are blocked in both the data contract and the browser flow.
- Connector recovery work is not done because statuses display; it is done when broken connectors pause unsafe downstream work and create recoverable operator tasks.

### 3. Legacy tests remain, but new phase work shifts to the target stack

- Existing `node --test` suites remain valid regression coverage.
- New phase 204-228 logic should default to `Vitest` unless the work is a legacy script path that cannot be moved without unnecessary churn.
- New browser workflow proof should default to `Playwright`.
- New stable UI states should default to Storybook + Chromatic.
- A plan may keep older node-test coverage, but it should not use that as a reason to skip the new stack where the new stack is the right fit.

## Required environment shape

## Repo-level test layers

The target environment should converge on this structure during execution:

- `test/vitest/**` or phase/domain-local `*.test.ts` files for `Vitest`
- `test/playwright/**` for browser journeys
- colocated `*.stories.tsx` for route and component states
- reusable fixture factories under `test/helpers/**`
- shared seeded records for tenants, roles, approvals, pricing, evidence, runs, and connector states

Exact file placement can follow local ownership patterns, but the three testing layers above must remain clear.

## Mandatory shared fixtures

Execution agents should not invent ad hoc test data per plan. The environment must provide reusable fixtures for:

- tenant identity: single-tenant, cross-tenant, Tenant 0
- roles: operator, tenant-admin, analyst, finance, support, growth, read-only
- auth posture: authenticated, expired, forbidden, partially provisioned
- billing posture: active, held, over-budget, pricing-pending
- run posture: queued, running, waiting, failed, paused, canceled, DLQ
- approval posture: pending, blocked, approved, rejected, edited
- evidence posture: supported, stale, contradictory, inferred, missing
- connector posture: healthy, stale, missing-scope, revoked, rate-limited
- SaaS posture: `business_type = saas`, non-saas, inactive suite, partially activated suite
- growth posture: planned but non-runnable agent tiers

## Environment tiers

### Fast lane

Runs on every PR and on every local execution attempt:

- `Vitest` domain/contract suites for touched logic
- targeted `Playwright` smoke for touched user journeys
- Chromatic build and changed-story review

### Phase lane

Runs before a phase is called complete:

- all phase-tied `Vitest` suites
- all phase-tied `Playwright` journeys
- all phase-tied Chromatic states
- migration/idempotency coverage
- authorization negative-path coverage
- phase-specific evidence, pricing, approval, or billing gate tests

### GA lane

Runs before a GA claim or public release:

- extended `Playwright` happy path + failure path coverage
- load or soak proof where the quality baseline requires it
- deterministic eval suites where agent behavior is involved
- accessibility and visual signoff

## Mandatory scripts and commands to implement during execution

The implementation should converge on a script surface like this:

- `test:unit` -> `vitest run --project unit`
- `test:integration` -> `vitest run --project integration`
- `test:coverage` -> `vitest run --coverage`
- `test:e2e` -> `playwright test`
- `test:e2e:smoke` -> critical-path subset
- `test:visual` -> `chromatic`
- `test:ui` -> storybook build + accessibility + Playwright component/browser smoke where applicable

Exact names can vary, but the repo needs explicit entry points for unit/integration/browser/visual layers.

## Coverage and gating policy

This plan operationalizes the Phase 200 quality baseline instead of replacing it.

### Coverage floor

- New `lib/markos/**` logic should hit at least 80% statement coverage.
- Critical surfaces stay at 100% behavior coverage expectation even when measured through a mix of `Vitest`, `Playwright`, and existing regression tests:
  - tenant isolation
  - auth/authz
  - billing enforcement
  - approval gates
  - migration idempotency
  - external mutation blocking
  - pricing placeholder policy
  - evidence claim blocking

### Browser gate floor

Any plan that introduces or changes a user-visible flow must define at least one `Playwright` path for:

- happy path
- permission-denied or blocked path
- meaningful empty/error/recovery path

### Visual gate floor

Any plan that introduces or changes a stable UI surface must define Chromatic-reviewed stories for:

- loading
- empty
- success
- error
- unauthorized or forbidden when applicable
- operator-critical warning or blocked state when applicable

## Testing contract required in every execution plan

Before code execution, each plan should derive a short testing contract from this template:

### Business truths

- What must be true in the domain after this plan ships?
- What must never be allowed?

### Vitest duties

- Which invariants, reducers, policies, migrations, schemas, and contract transforms must be proven deterministically?

### Playwright duties

- Which operator or customer flows must be proven in a real browser?
- Which blocked/denied/recovery states must be exercised?

### Chromatic duties

- Which stories must exist?
- Which visual states are merge-blocking?

### Evidence required for phase close

- command names run
- suite names added or updated
- stories added or updated
- risk left deferred, if any

## Test design rules for execution agents

### Do

- test business decisions, not only rendered strings
- prove tenant scoping and role behavior on every new sensitive surface
- keep fixtures small but realistic
- prefer deterministic test data over network randomness
- make pricing, evidence, approval, and billing failure modes explicit
- attach tests to the domain vocabulary already used in the vault and requirements

### Do not

- count source-file greps as sufficient proof for new v2 operator flows
- use Chromatic as a substitute for workflow verification
- use Playwright as a substitute for deterministic domain logic tests
- skip negative-path and forbidden-path coverage on sensitive mutations
- hide missing test infrastructure behind "future QA" notes for active work

## Cross-phase expectations

### Pricing-sensitive surfaces

Must prove:

- placeholder enforcement
- approved recommendation consumption
- no silent hard-coded public prices
- blocked mutation when pricing evidence is insufficient

### Evidence-sensitive surfaces

Must prove:

- unsupported claims are blocked or labeled
- stale evidence is downgraded
- inference is distinguishable from sourced fact

### Approval-sensitive surfaces

Must prove:

- approval state gates real mutations
- rejection reasons persist
- edits/overrides create audit lineage

### Multi-tenant surfaces

Must prove:

- no cross-tenant reads
- no cross-tenant writes
- operator/admin escalation still respects policy boundaries

### SaaS/growth future surfaces

Must prove:

- activation gating
- non-runnable future agents stay non-runnable
- pricing and incentive actions remain approval-aware

## CI alignment

The current `ui-quality.yml` workflow is a good start, but the future environment should separate:

- `Vitest` unit/integration jobs
- `Playwright` browser jobs
- Storybook/Chromatic jobs
- phase-specific high-risk suites

The CI contract should move from "UI quality exists" to "business logic, workflow safety, and visual stability all exist."

## Relationship to legacy node tests

Legacy `node --test` coverage should be preserved and reused where it protects mature domains. This plan does **not** require a risky immediate rewrite of the entire suite. It does require that future execution work stop relying only on legacy patterns when `Vitest`, `Playwright`, or Chromatic are the correct proof tools.

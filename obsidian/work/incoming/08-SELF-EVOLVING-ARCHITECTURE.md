# MarkOS Self-Evolving Architecture
## Living System Design — v5.0 Direction

---

## Overview: The Learning Machine

MarkOS today is a deterministic marketing OS — great canonical literacy, rigid agent pipelines, static brand packs. What you're designing is a **living system** that:

1. Continuously harvests new marketing intelligence from the web
2. Learns what actually works from cross-tenant performance data
3. Tunes each tenant's literacy overlay using their own results
4. Feeds all of this back into the central literacy corpus
5. Uses MarkOS itself as Tenant 0 — the most-instrumented test case

The architecture has five layers, each depending on the one below it.

---

## Layer 1: The Performance Dimension Registry

**Problem**: Right now MarkOS generates content but never learns from what it produced. Every output disappears into a CRM or a client's scheduled post. There is no feedback loop.

**Solution**: Tag every generated artifact with a full performance envelope at creation time, then collect outcome signals back into that envelope.

### The Dimension Schema

Every artifact generated by MarkOS (an email, ad, landing page, Instagram carousel, etc.) gets tagged with:

```yaml
generation_id: uuid
tenant_id: uuid
timestamp: ISO8601

# What was made
channel: instagram | email | linkedin | meta_ads | landing_page | sms | push | ...
format: carousel | reel | cold_email | nurture_sequence | short_form_video | ...
artifact_type: copy | visual_brief | full_campaign | creator_brief | ...

# Who it was for
industry: travel | b2b_saas | dtc | local_services | ecommerce | ...
audience_archetype_id: ref to archetype in MIR
pain_point_tag: high_acquisition_cost | low_conversions | ... (from taxonomy)
funnel_stage: awareness | interest | consideration | decision | retention | advocacy

# How it was made
agents_used: [MARKOS-AGT-STR-01, MARKOS-AGT-NEU-01, ...]
neuro_triggers: [B02, B04]
brand_voice_score: float  # from voice classifier at generation time
literacy_nodes_referenced: [node_id, ...]  # which literacy notes informed this
pipeline_version: semver
prompt_template_id: uuid

# Quality at generation time
claim_count: int
claim_all_sourced: bool
anti_pattern_violations: []
audit_pass: bool

# Outcome signals (filled in later via connector or webhook)
outcome:
  impressions: int | null
  clicks: int | null
  ctr: float | null
  conversions: int | null
  cvr: float | null
  revenue_attributed: float | null
  engagement_rate: float | null
  open_rate: float | null  # email
  reply_rate: float | null  # cold email
  roas: float | null
  human_rating: 1-5 | null  # operator thumbs up/down
  ab_winner: bool | null
  performance_percentile: float | null  # vs similar artifacts in same channel+format+industry
```

This schema goes into a new Supabase table: `markos_artifact_performance_log`. Every agent run that produces a publishable artifact writes a row. Outcome signals are patched in via:

- Platform webhooks (Meta, Google, LinkedIn, Klaviyo, Resend, Twilio)
- Connector polling jobs (GA4, PostHog)
- Manual operator rating (in-app thumbs up/down)
- A/B test resolution events

### What this unlocks

With 1,000 artifact rows you can answer: "For B2B SaaS cold emails targeting high_acquisition_cost pain, which neuro trigger combination wins?" With 10,000 rows you can answer that per industry, per funnel stage, per audience archetype. With 100,000 rows the system starts self-improving automatically.

## Layer 2: The Literacy Intelligence Engine

**Problem**: The current literacy corpus is hand-authored and static. It reflects best practices as of its last human edit. Marketing best practices change fast — TikTok Shop overtook Instagram commerce in 6 months, Google AI Overviews reshaped SEO overnight, MCP changed agent distribution in weeks.

**Solution**: A background engine that continuously refreshes literacy from three sources — web intelligence, performance data, and community signals — and surfaces update candidates for human review before promotion.

### The Three Refresh Sources

**Source A: Web Intelligence Crawler**

A scheduled agent (weekly, with topic-triggered hotfixes) that:

1. Queries public sources for each of the 27 literacy disciplines: industry blogs, platform engineering blogs (Meta, Google, TikTok, LinkedIn), research papers, practitioner newsletters, conference talks
2. Extracts signal using the existing MarkOS literacy schema: Definition / Why it matters / Mechanics / Pitfalls / Related
3. Diffs against current literacy note content
4. Flags significant divergences ("TikTok changed ad attribution model", "Google removed RSA character limits")
5. Drafts a proposed update note in `thinking/literacy-refresh/YYYY-MM-DD-<discipline>.md`
6. Tags it for human review before any promotion

Relevant search queries are auto-generated per literacy node from the node's tags, title, and existing related section. The `llms.txt` file and schema.org entities for each discipline seed the query.

**Source B: Cross-Tenant Performance Intelligence**

The performance dimension registry (Layer 1) is aggregated weekly into a `markos_cross_tenant_signals` table:

```sql
-- anonymized, no tenant_id, no PII
channel | format | industry | pain_point_tag | funnel_stage |
neuro_triggers | brand_voice_score_band |
p50_ctr | p75_ctr | p90_ctr |
p50_cvr | p75_cvr | p90_cvr |
sample_size | last_refreshed
```

When sample size crosses a threshold (configurable per channel — email needs 200, Instagram 500, cold email 50), the system generates a "performance insight" note:

```
For [channel=instagram, format=carousel, industry=travel, funnel_stage=awareness]:
- Neuro trigger combo B09+B10 (pattern interrupt + tribal belonging) outperforms B02+B03 at 2.1x CTR
- Brand voice score >87 correlates with 34% higher save rate
- Pain tag content_engagement responds best; high_acquisition_cost content underperforms
- 6-slide carousels at 1080x1080 beat 10-slide at p75 engagement
```

This is not automatically written into literacy. It's staged as a proposed update to the relevant literacy note (e.g., `28 Message Crafting/Social Post Message Crafting.md`) with provenance attached. A human admin approves before promotion.

**Source C: Operator Feedback Loops**

Within the MarkOS UI, every content approval flow includes:
- Thumbs up/down on generated content
- Optional tag: "what was wrong" (tone, claim, format, hook, proof)
- Post-publish rating (optional, collected 48h after dispatch)

These map to specific literacy nodes via the `literacy_nodes_referenced` field in the artifact log, creating a precision signal: "operators rate content citing `Email Message Crafting.md` at 3.1/5 when the hook is pain-based — but 4.3/5 when the hook is authority-based." That is an actionable update signal.

### The Literacy Freshness Monitor

A new brain note: `brain/Literacy Freshness Monitor.md` tracks:

```yaml
literacy_node: "05 SEO & Organic Discovery/Generative Engine Optimization"
last_human_review: 2026-04-16
last_web_refresh_check: 2026-05-03
staleness_signal: "Perplexity launched Shopping tab; GEO section needs update"
performance_signal: "0 cross-tenant artifacts cite this — possible gap"
urgency: high | medium | low | stable
proposed_update_path: thinking/literacy-refresh/2026-05-03-geo.md
```

The admin dashboard shows this as a "Literacy Health" panel. Red = stale + high urgency. Green = recently reviewed + performing.

## Layer 3: Tenant-Specific Literacy Overlays

**Problem**: Every tenant gets the same central literacy. A travel agency, a B2B SaaS startup, and a local restaurant have radically different optimal practices for the same channel.

**Solution**: A two-tier literacy system. Central literacy is the shared foundation. Tenant overlays are derived extensions that live in the tenant's vault (`.markos-local/`) and in a new Supabase table.

### How Overlays Work

When a tenant runs MarkOS long enough to generate 20+ artifacts in a specific channel × format × industry combination, the system creates a **Tenant Literacy Overlay**:

```
.markos-local/literacy-overlays/
  instagram/carousel/travel/
    OVERLAY.md       # tenant-specific refinements
    PERFORMANCE.md   # what's worked for this tenant specifically
```

The overlay structure:
```yaml
---
base_literacy_node: "28 Message Crafting/Social Post Message Crafting"
tenant_id: <anonymized>
channel: instagram
format: carousel
industry: travel
observations: 12 artifacts, 8 with outcome data
last_refreshed: 2026-05-10
confidence: medium  # low < 20 artifacts, medium < 100, high >= 100
---

# What works for us specifically

## Hook patterns that convert
- "Have you been to X?" outperforms "5 reasons to visit X" at 1.8x CTR (n=6)
- Visual hook (hero destination image) + 3-word text overlay beats long text (n=4)

## Proof patterns that work
- User-generated photos outperform stock at 2.1x save rate
- "From our community" framing activates B10 tribal belonging effectively here

## What doesn't work for our audience
- Listicle format (5 tips for...) generates low saves despite decent reach
- Price-forward hooks trigger swipe-away on awareness posts

## Best performers (anonymized examples)
- [ref to artifact_id] — 4.2% CTR, 890 saves
- [ref to artifact_id] — 3.9% CTR, 1.2k saves
```

The overlay agent (`MARKOS-AGT-OVERLAY-01`, new) runs after each batch of performance data arrives and refreshes the overlay file. The context loader and content creator agents now check for an overlay before defaulting to central literacy — overlay takes precedence where it exists.

### Overlay Propagation to Central Literacy

When a tenant overlay reaches high confidence (100+ artifacts) and shows consistent patterns that diverge from central literacy, the admin review queue gets a **"Cross-tenant promotion candidate"** — a proposed addition to the central literacy note. If 3+ tenants in the same industry show the same pattern, it promotes automatically with admin notification.

This is the core learning loop: tenant data → tenant overlay → (if consistent across tenants) → central literacy update.

## Layer 4: The Literacy Regeneration Agent Network

This is where the system becomes self-maintaining. Four new agents handle the refresh cycle:

### MARKOS-AGT-LIT-01: Literacy Scout

**Schedule**: Weekly + on-demand hotfix trigger

**What it does**:
1. Reads each literacy note's `tags`, `title`, and `Related` section
2. Generates targeted search queries: "Instagram carousel best practices 2026", "TikTok algorithm change latest", etc.
3. Uses web search + defuddle to extract clean content from industry sources
4. Generates a structured diff: what changed vs current note
5. Writes `thinking/literacy-refresh/YYYY-MM-DD-<node>.md` with proposed changes + source citations
6. Does NOT write to literacy directly — everything queues for human review

**Hotfix triggers**: The scout runs immediately when:
- A major platform announces algorithm changes (keyword triggers in news feeds)
- A central literacy note receives 5+ negative operator ratings in a week
- A platform connector reports a new API version or deprecation

### MARKOS-AGT-LIT-02: Performance Synthesizer

**Schedule**: Weekly, after performance data has been aggregated

**What it does**:
1. Reads `markos_cross_tenant_signals` for each channel × format combination with sufficient sample size
2. Compares patterns to what the current literacy note recommends
3. Flags divergences: "Literacy says B04 (authority) works best for consideration stage emails, but cross-tenant data shows B07 (commitment ladder) outperforms at p75 CVR"
4. Drafts additions to the `## Mechanics` and `## Pitfalls` sections of the relevant literacy note
5. Queues for admin review with full data provenance attached

### MARKOS-AGT-LIT-03: Tenant Overlay Refresher

**Schedule**: After each performance data sync for a tenant

**What it does**:
1. Reads the tenant's `markos_artifact_performance_log` for the past 30 days
2. Groups by channel × format × industry combinations
3. Extracts patterns (hook patterns vs CTR, proof patterns vs save rate, trigger combos vs CVR)
4. Writes or updates the tenant overlay file in `.markos-local/`
5. Flags promotion candidates to admin queue if pattern is statistically consistent

### MARKOS-AGT-LIT-04: Literacy Promoter

**Trigger**: Admin approves a staged update

**What it does**:
1. Takes an approved `thinking/literacy-refresh/*.md` update
2. Applies it to the target literacy note using a safe edit pattern (preserves structure, adds or updates specific sections)
3. Creates a git commit with provenance: "literacy: update GEO note — source: cross-tenant perf + Perplexity Shopping launch"
4. Updates the `last_human_review` date in the Freshness Monitor
5. Triggers a QMD re-index so the updated note is immediately queryable
6. Archives the source `thinking/` note with outcome tag

## Layer 5: Admin Intelligence Layer

The SaaS admin (you) needs a single place to see:

### 1. Platform-Wide Health

```
Tenants active:          47
Artifacts generated:     12,304
Outcome data connected:  68%
Cross-tenant signals ready: 23 channel×format combinations

Literacy health:
  ✅ Updated < 30 days:     18 / 27 disciplines
  ⚠️  Needs review:          6 / 27 disciplines
  🔴  Stale > 90 days:       3 / 27 disciplines

Pending literacy updates:  8 staged, awaiting approval
```

### 2. Performance Intelligence Matrix

A queryable table: given any combination of channel + format + industry + pain_point + funnel_stage, show:

- p50/p75/p90 performance benchmarks
- Top performing neuro trigger combinations
- Brand voice score correlation
- Sample size (for confidence)
- Trend direction (improving / declining / stable)

This is the admin's answer to "what should our travel agency tenant do differently on Instagram?"

### 3. Tenant Comparison View (anonymized)

Show performance curves for all tenants in the same industry side-by-side (no names, no PII). A tenant that's underperforming on email open rates can see the p75 benchmark for their industry without seeing competitor content.

The admin can also drill in: "why is Tenant #14 underperforming on LinkedIn?" The system surfaces: their brand voice score averages 71 (vs p75 of 84 in their industry), and their hook patterns cluster in the weakest-performing category. Recommended action: run a voice calibration session + hook A/B test.

### 4. Literacy Update Review Queue

All staged literacy updates appear here with:
- Source (web crawl / performance data / tenant overlay promotion candidate)
- Affected literacy note
- Diff preview
- Supporting evidence (links, sample sizes, confidence)
- Approve / Reject / Request revision

One-click approval triggers MARKOS-AGT-LIT-04.

## Tenant 0: MarkOS Markets Itself

MarkOS is the first tenant. It is also the most visible, most instrumentable, and most strategically important test case. The setup:

### Tenant 0 Configuration

- **Industry**: `b2b_saas` (primary) + `ai_dev_tools` (secondary)
- **ICP**: Three archetypes from [[Target ICP]]: Solo Founder Sam, Growth Gina, Commerce Clara
- **Channels**: LinkedIn (organic + paid) · X/Twitter · Product Hunt · Claude Marketplace · GitHub · YouTube · Hacker News
- **Pain tags**: All 8 — MarkOS speaks to every marketing pain by definition

### What makes Tenant 0 special

1. **Full observability**: Every piece of marketing MarkOS generates for itself is instrumented end-to-end. PostHog captures signups, MQL conversion, trial activation, paid conversion. This is the cleanest dataset in the platform.

2. **Instant feedback loop**: When a new literacy update is promoted, MarkOS's own content is the first test. If the new "developer-native hook pattern" works, it shows up in signups within days.

3. **Competitive intelligence**: The Literacy Scout runs extra queries for MarkOS Tenant 0 — tracking Jasper, Copy.ai, Adobe GenStudio, Typeface, and Notion's positioning. Competitive signal feeds directly into the marketing strategy.

4. **Dogfood gate**: No marketing is published for MarkOS without passing MarkOS's own brand classifier (voice score ≥ 87, zero anti-pattern violations, all claims sourced). The system markets itself at its own standard.

5. **The product IS the case study**: Every successful MarkOS-generated campaign becomes a public case study in the marketing literacy corpus. "How MarkOS generated a Product Hunt #1 using its own neuro-trigger pipeline" is a real literacy example, not a hypothetical.

### Tenant 0 Agent Setup

Because MarkOS markets itself and the primary channel is developer-native content, the agent configuration differs from typical tenants:

```
Primary channels: 
  Technical content → Documentation-as-Marketing, GitHub README, developer blog
  Community → HN, Reddit r/MachineLearning, Claude Marketplace reviews
  Thought leadership → LinkedIn founder-led content
  Product launches → Product Hunt, Claude Marketplace, Hacker News Show HN

Special agents for Tenant 0:
  MARKOS-AGT-COMP-01: Competitor monitor (runs daily for Tenant 0, weekly for others)
  MARKOS-AGT-DEVREL-01: Developer relations — GitHub, docs-as-marketing, API examples
  MARKOS-AGT-PH-01: Product Hunt launch coordinator (specialized for sequential launch strategy)
```

```
Central Literacy (shared, human-governed)
         │
         ▼
   Tenant Ingestion
   (at onboarding, tenant gets a read-only copy of relevant literacy
    + an empty overlay for their industry+channels)
         │
         ▼
   Content Generation
   (agents read central + overlay, overlay takes precedence)
         │
         ▼
   Performance Collection
   (outcome signals arrive via connectors, webhooks, operator ratings)
         │
         ▼
   Overlay Refresh (MARKOS-AGT-LIT-03)
   (tenant-specific patterns extracted from performance data)
         │
         ▼
   Cross-Tenant Aggregation (anonymized)
   (signals aggregated weekly into markos_cross_tenant_signals)
         │
         ├─── Performance Synthesizer (MARKOS-AGT-LIT-02)
         │    proposes literacy updates based on aggregate patterns
         │
         ├─── Web Scout (MARKOS-AGT-LIT-01)
         │    proposes literacy updates based on industry changes
         │
         ▼
   Admin Review Queue
   (human reviews, approves, rejects)
         │
         ▼
   Literacy Promoter (MARKOS-AGT-LIT-04)
   applies approved updates → Central Literacy updated
         │
         ▼
   Loop repeats — system gets smarter with each cycle
```

## Key Design Decisions

**Decision 1: Human in the loop for literacy promotion**
No literacy update goes live without admin approval. The system can be fully autonomous later once trust is established, but starting with human review prevents a bad performance signal or a biased web crawl from corrupting the central literacy that all tenants depend on.

**Decision 2: Overlay > central literacy, always**
When a tenant overlay exists for a channel×format combo and has medium+ confidence, it overrides the central note for that tenant. This respects that a travel agency's Instagram carousel best practices differ from a B2B SaaS's, even if the underlying mechanics are similar.

**Decision 3: All signals anonymized before aggregation**
No tenant sees another tenant's content or metadata. The aggregation layer produces only statistical distributions. This is both a privacy decision and a competitive one — tenants don't want their performing content patterns shared.

**Decision 4: Performance percentile is relative, not absolute**
A 1% CTR on LinkedIn feed means something very different than 1% on Instagram Reels. Performance percentiles are always computed within the same channel + format + industry bucket. This prevents misleading benchmarks.

**Decision 5: Tenant 0 gets the most aggressive learning cadence**
MARKOS-AGT-COMP-01 runs daily for Tenant 0. Literacy Scout runs on hotfix trigger for topics relevant to MarkOS's own positioning. The system invests more in learning from itself than from any other tenant, because those learnings are immediately applicable to every future marketing output.

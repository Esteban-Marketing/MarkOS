# MarkOS Human Interface Layer
## Task Board · Approval Inbox · Notification Engine · Client Portal · Chat · The Operator Day

---

## The Interface Philosophy

The MarkOS interface has one job: **make human decisions fast and confident.**

Agents run in the background continuously. They generate briefs, drafts, analyses, recommendations,
alerts, and proposed actions — hundreds of artifacts per week for an active tenant. The interface
is how all of that production surfaces to humans in a way that is actionable, not overwhelming.

The wrong interface design makes MarkOS feel like a firehose. Every notification is noise.
Every approval is a chore. Every report is a scroll through something you don't trust.

The right interface design makes MarkOS feel like the best-run team you've ever worked with.
Every notification is relevant. Every approval takes 15 seconds. Every report tells you exactly
what changed and what to do about it.

Every UI decision is evaluated against this standard:
**Does this help the human decide faster, or does it ask them to do more work?**

## Part 1: The Operator Day

Before specifying components, define what using MarkOS actually feels like.

### 8:00am — Morning brief

When the operator opens MarkOS, the first screen is not a dashboard of metrics. It is a
**Morning Brief** — a generated summary of what happened overnight and what needs attention today.

```
Good morning, Ana.

Last night:
→ Your LinkedIn post on attribution modeling hit 12.4K impressions (3.8× your average). 
  The comments are positive — 4 need replies.
→ Your Google Ads CPC increased 18% between 9pm and midnight. The Anomaly Detector flagged it. 
  Investigation queued — see task #4821.
→ 3 new leads came in from the "B2B Attribution Guide" landing page. 
  All three are scored and in the nurture sequence.

Today:
→ 6 approvals waiting (2 social posts, 1 email campaign, 3 comment replies)
→ Tuesday's LinkedIn post needs to be finalized by 2pm
→ Acme Corp demo at 3pm — prep brief is ready

Nothing urgent. Est. 22 minutes to clear your queue.
```

The morning brief is generated by a lightweight synthesis agent that reads:
- All agent runs completed in the last 12 hours
- All pending approvals
- All calendar events for the day
- All open tasks due today or overdue

It does not replace the dashboard. It replaces the compulsive need to check every dashboard.

### The rest of the day

The operator interacts with MarkOS in three modes:

**Active mode** (15–30 min/day): clearing the approval inbox, reviewing the morning brief,
making decisions on escalated items.

**Reactive mode** (as needed): responding to notifications — a crisis alert, a high-intent
DM, a client question in the portal.

**Strategic mode** (weekly): reviewing performance, adjusting OKRs, approving content strategy,
reviewing the self-evolving system's literacy updates.

The UI is designed for active mode to be efficient, reactive mode to be fast, and
strategic mode to be thorough.

## Part 2: The Task Board

### Model

Every agent output, every escalation, every human action needed — all become tasks.
MarkOS's task board is the operational center of the product.

```typescript
interface Task {
  task_id: string
  tenant_id: string

  // Identity
  title: string
  description: string | null
  task_type: TaskType
  source: TaskSource

  // Context
  linked_artifact_id: string | null      // content, report, campaign, etc.
  linked_run_id: string | null           // the agent run that created this
  linked_campaign_id: string | null
  linked_contact_id: string | null       // if related to a CRM contact
  client_id: string | null               // if visible to a client in their portal

  // Assignment & Scheduling
  assigned_to: string | null             // user_id
  assigned_to_agent: string | null       // agent token (for agent-handled tasks)
  due_at: string | null                  // ISO8601
  priority: 'urgent' | 'high' | 'normal' | 'low'

  // Status
  status: TaskStatus
  approval_gate_id: string | null        // if this task IS an approval decision
  blocked_by: string[]                   // task_ids this depends on

  // Client-facing
  visible_to_client: boolean
  client_facing_title: string | null     // simplified title for client view

  // Metadata
  tags: string[]
  created_at: string
  updated_at: string
  completed_at: string | null
  completed_by: string | null
}

type TaskType =
  | 'approval'         // an agent output needing human sign-off
  | 'review'           // something to read and assess (no publish action)
  | 'action'           // a human must do something manually
  | 'decision'         // a strategic decision that will unlock next steps
  | 'escalation'       // an agent failure or anomaly requiring investigation
  | 'client_request'   // something a client sent through the portal
  | 'follow_up'        // time-boxed reminder with context

type TaskSource =
  | 'agent_run'        // created by an agent
  | 'schedule'         // created by the scheduler (e.g. weekly review reminder)
  | 'operator'         // created manually by an operator
  | 'client'           // created by a client via portal
  | 'system'           // created by the platform (e.g. budget alert)

type TaskStatus =
  | 'inbox'            // newly created, not yet triaged
  | 'todo'             // triaged, not started
  | 'in_progress'      // actively being worked
  | 'awaiting_input'   // blocked on external input (client, third party)
  | 'done'             // completed
  | 'cancelled'        // dismissed with reason
```

### Board views

**Inbox view:** All `status: 'inbox'` tasks, newest first. Each card shows: task type badge,
title, source (which agent / which client), linked artifact preview thumbnail, due date, priority.
One-click to open full detail. Bulk triage actions: approve, reject, assign, snooze.

**Board view (kanban):** Columns: Inbox → To Do → In Progress → Awaiting Input → Done.
Filterable by: assigned user, campaign, client, task type, priority. Drag to move between columns.

**Calendar view:** Tasks plotted on calendar by due date. Content calendar overlay: shows
scheduled publishes alongside tasks. Dependency lines between tasks shown on hover.

**Campaign view:** All tasks filtered to a specific campaign, shown as a timeline.
The campaign DAG (from `09-ORCHESTRATION-CONTRACT.md`) is visible here — you can see which
agent runs are queued, which are running, which are awaiting approval.

**Client view:** Tasks that are `visible_to_client: true` and assigned to a specific client.
This is the backend of the client portal.

### The approval inbox (sub-view)

The approval inbox is a focused view of all `task_type: 'approval'` tasks in `status: 'inbox'`.
This is the most-used view in the product.

Each approval card shows:

```
[ Social post approval ]                        LINKEDIN · Due in 2h · 📋 Brand score: 94/100

Post preview (rendered, platform-accurate):
┌──────────────────────────────────────────┐
│ Most marketers think attribution is a    │
│ data problem. It's actually a decision   │
│ problem. Here's the framework...         │
└──────────────────────────────────────────┘

Agent: CONT-06 Social Content Creator
Campaign: June Thought Leadership
Triggered by: Weekly content calendar (scheduled)
Voice dimensions: Direct ✓  On-brand ✓  Tone ✓  Complexity ✓
Compliance: ✓ No issues flagged

[ ✓ Approve ]  [ ✏ Edit & Approve ]  [ ✕ Reject ]  [ ↻ Reassign ]
```

The "Edit & Approve" action opens an inline editor. On save, the edit is recorded,
the voice score is re-run, and the task is marked approved. The diff between the agent draft
and the final approved version is logged and feeds the literacy system.

**Batch approval** is available when multiple posts in the same campaign or from the same
template are ready. The operator sees them side-by-side and can approve/reject each with one click.

## Part 3: The Notification Engine

### Notification taxonomy

Four levels, each with different delivery rules:

| Level | Label | What it means | In-app | Push | Email | SMS |
|-------|-------|---------------|--------|------|-------|-----|
| L0 | Emergency | Requires immediate human action — crisis, security, system failure | ✅ Banner | ✅ | ✅ | ✅ if configured |
| L1 | Urgent | Decision needed within 1–2 hours | ✅ Badge | ✅ | ✅ | — |
| L2 | Normal | Needs attention today | ✅ Badge | — | Digest | — |
| L3 | Info | FYI, no action required | ✅ Feed | — | Weekly digest | — |

### Notification types and their default levels

| Notification | Default level | Configurable? |
|-------------|--------------|---------------|
| Social crisis detected | L0 | Level down to L1 only |
| Compliance block on content | L1 | No — always L1+ |
| Budget 90% used | L1 | Yes |
| Approval pending > 4h (time-sensitive) | L1 | Yes |
| High-intent DM received (lead score > 80) | L1 | Yes |
| Agent chain failed | L1 | Yes |
| New lead from form fill | L2 | Yes |
| Approval newly created (normal) | L2 | Yes |
| Weekly OKR health report ready | L2 | Yes |
| Content published successfully | L3 | Yes |
| Agent run completed (routine) | L3 | Yes — can suppress |
| New follower (high-profile) | L3 | Yes |

### Digest emails

Two digest schedules:
- **Daily digest** (8am, operator TZ): pending approvals, lead activity, key metric movements,
  outstanding tasks due today
- **Weekly digest** (Monday 8am): OKR health, top-performing content, pipeline created,
  budget pacing, upcoming week's calendar

Digests are generated by a lightweight synthesis agent. They are not templated newsletters —
they are dynamically generated summaries calibrated to what actually happened.

### Notification preferences

Per user, configurable:
- Delivery channel per level (which channels to use for each level)
- Working hours (L2 and below suppressed outside working hours, L0 and L1 always deliver)
- "Focus mode" — suppress all L2/L3 for a defined period, L0/L1 still deliver
- Per-notification type overrides (e.g. always get push for new leads, never email for content published)

## Part 4: The Client Portal

### What the portal is

The client portal is a white-labeled view of MarkOS scoped to a single client's data.
It is not a full MarkOS account. It is a curated, simplified window into:
- What's been done on their behalf
- What's coming up
- What needs their input or approval
- How things are performing

The portal is available at a white-labeled subdomain: `client.{agencyname}.com` or
`{clientname}.{agencyname}.com`.

### Portal sections

**Dashboard (client home)**

```
Welcome back, Carlos.

This week at Acme Corp:
→ 3 pieces of content published (blog, LinkedIn, email)
→ Your LinkedIn following grew 4.2% (+312 followers)
→ 2 items need your approval

Performance snapshot:
  Organic traffic    ↑ 18%  vs last week
  Leads from content  8     vs 5 last week
  Email open rate    42%    vs 38% industry avg
```

**Content calendar view**

Simplified calendar showing:
- Published content (with links, performance data)
- Scheduled content (previews, with client approval flag)
- Upcoming planned content (title, format, publish date)

Client can: leave comments on scheduled content, approve/reject content flagged for client approval.

**Campaign status**

Active campaigns visible to this client:
- Campaign name, goals, current phase
- Spend vs budget (for clients with budget visibility configured)
- Key metrics since launch
- Next milestone

**Reports**

Agency-generated reports shared to this client.
Each report: viewable in-portal, downloadable as PDF, shareable link with expiry.

**Messages**

Client-to-agency messaging. Threaded. Linked to tasks on the agency side.
When a client sends a message, it creates a `task_type: 'client_request'` task
in the agency's task board with a P2 priority.

**Approvals**

Content or strategy items flagged for client approval:
- Rendered preview (same rendering as agency approval inbox)
- Approve / request changes button
- Changes request: inline comment on the artifact, creates task on agency side

### Portal access management

Clients are users with a `portal` role. They cannot access:
- Other clients' data
- Agent run details
- Internal task board (except items explicitly shared with them)
- Connector credentials or API settings
- Billing and cost data

Client users can be created by the agency operator and receive an invite email.
Multi-contact support: multiple client-side users can have portal access for the same client account.

## Part 5: Contextual Chat

### What chat is for in MarkOS

Chat is not a feature for talking to Claude. It is a feature for **communication in context** —
between operators, between operators and clients, and between operators and the MarkOS system
when they need a quick answer or clarification.

### Chat types

**Artifact thread:** Every artifact in MarkOS (a content brief, a strategy doc, a campaign plan,
a report) has a comment thread. Operators can leave comments, tag team members, and have
threaded discussions directly on the artifact. This replaces the "email the draft around" workflow.

```
On: June Content Calendar — LinkedIn posts
─────────────────────────────────────────
Ana:    The hooks on posts 3 and 5 feel similar. Can we differentiate?
        @Sofia take a look at #3 specifically

Sofia:  On it. I'll update and re-submit for approval.

CONT-06: [Updated] LinkedIn post #3 revised — hook changed from "X marketers think..." 
          to "The 2026 LinkedIn algorithm punishes..." — updated draft ready for approval.
          Voice score: 91 (↑3 from original)
```

Notice: the agent participates in the thread after being tasked. Agents can be
@-mentioned in artifact threads to trigger a run with that artifact as context.

**Client thread:** Messages between agency operators and portal clients. Visible in both
the client portal and the agency task board. Threaded to the relevant artifact or task.

**Direct messages:** Operator-to-operator within an agency account. Standard DM behavior.

**System messages:** MarkOS system notifications that appear as messages (for context-rich
alerts that need more than a push notification). The anomaly detector's detailed alert,
for example, appears as a message in a system thread that can be replied to.

### @-mentioning agents

In any artifact thread, an operator can @-mention an agent by token ID or name:

```
@content-editor Can you review this for voice alignment and flag anything 
that sounds too formal? We're targeting founders, not enterprise buyers.
```

This creates a P2 agent run with:
- The artifact as primary context
- The message text as the task instruction
- The thread as the output destination

The agent's response appears in the thread within minutes.

WORK

### Agency mode vs single-tenant mode

When an account manages multiple clients (agency mode), a client switcher appears at the top
of the sidebar. The entire workspace context switches to the selected client.

In agency mode:
- All data views are scoped to the active client
- A cross-client view is available in Analytics for portfolio-level reporting
- The task board has a "All Clients" view that aggregates tasks across clients
- The morning brief can be scoped to one client or the full portfolio

## Part 7: Mobile Experience

The mobile app (iOS and Android) is not a full replica of the desktop product.
It is optimized for the **reactive mode** operator use case — the things that need
a response when you're not at a computer.

Mobile priority views:
1. **Approval inbox** — approve/reject/edit social posts and responses on the go
2. **Notifications** — all notification levels, with deep links to relevant artifacts
3. **Social inbox** — DM and comment responses that need human review
4. **Morning brief** — read-only, daily summary
5. **Client messages** — reply to client questions from portal
6. **Quick task view** — see and complete simple tasks (mark done, add note, reassign)

What is deliberately NOT on mobile:
- Campaign creation
- Strategy documents
- Full analytics dashboards
- Agent queue management

These require deliberate, seated attention. Mobile is for keeping things moving when
you're away from the desk.
